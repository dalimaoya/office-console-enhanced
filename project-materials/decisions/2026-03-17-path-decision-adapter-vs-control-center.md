## 1. 技术维度（Jax）

### 六维评分对比

| 维度 | adapter | control-center |
|------|:---:|:---:|
| 技术栈与架构成熟度 | 中 | 好 |
| 功能覆盖度 | 差（约20%） | 好（约90%） |
| 代码质量与可维护性 | 中 | 好 |
| 与目标契合度 | 差 | 好 |
| 扩展成本 | 差（6-8周） | 好（1-2周） |
| 迁移成本（adapter→CC） | 低 | N/A |

### 关键技术理由
1. **功能结构性差距**：adapter 仅覆盖目标功能约 20%；control-center 覆盖约 90%（8大分区+27+个架构模块）
2. **架构根本瓶颈**：adapter 通过子进程调用获取数据，性能与可靠性存在根本瓶颈；control-center 直接读取文件系统 + WebSocket，架构更优
3. **安全框架差距**：adapter 无鉴权、无只读模式；control-center 有完善的 READONLY_MODE + LOCAL_TOKEN_AUTH + 审批 gate
4. **成本差距**：在 adapter 上补齐到 control-center 水平约需 6-8 周；基于 control-center 定制约需 1-2 周
5. **唯一需迁移的资产**：adapter 的 YAML 模板配置系统（约 3 个核心文件），可独立移植

完整技术报告：`docs/2026-03-16-architecture-comparison-report-jax.md`

## 2. 产品维度（Ekko）

### 结论
完全认同技术结论，产品价值更加清晰。

### 核心发现
- control-center 的员工/协作/任务/总览四大分区与办公场景高度契合
- adapter 中唯一需要保留的差异化产品决策：**飞书双向集成理念**（control-center 完全没有）

### 最重要的定制工作
**P0（阻塞核心价值）**：
1. 飞书消息通道集成（agent 状态推送到飞书）
2. 飞书指令入口（在飞书群/单聊触发 agent 任务）
3. 办公场景 agent 角色映射（Teemo、Jax、Ekko 等角色定制显示）

**P1（提升体验）**：
1. 办公用户信息层级简化（为非技术用户提供摘要视图）
2. 告警→飞书通知完整闭环

### 两周产品优先级路线图
第一周：飞书单向推送 + agent 角色映射 + 核心三分区（总览/员工/协作）办公简化层
第二周：飞书指令入口 + 告警飞书闭环 + 用量与任务分区数据接线验证

## 3. 办公场景维度（Jayce）

### 结论
完全认同，飞书双向集成是从"知道有这个工具"到"真正每天用它"的关键跨越。

### 核心发现
- 绝大多数办公用户**不会主动打开 Web UI**，日常路径是：飞书接收通知 → 点击跳转查看详情 → 必要时做操作
- control-center 当前界面偏开发者向，普通办公用户直接使用接受度预计**中低**

### 用户接受度提升关键
1. **飞书卡片替代独立页面**：核心诉求 80% 在飞书内解决
2. **信息精简至三层**：状态 / 可操作动作 / 历史
3. **Web 控制台定位为"高级管理入口"**，而非日常入口

### 飞书集成优先级
- P0：通知推送（任务完成/失败/阻塞）
- P0：指令入口（飞书群/单聊直接触发）
- P1：状态同步（群卡片实时展示 agent 状态）

## 4. 决策路径与风险

### 选项 A（借鉴融合）- 不推荐
- 继续当前 adapter 项目，把 control-center 的优秀元素逐步融合
- **问题**：需要 6-8 周才能补齐基础功能，且架构瓶颈（子进程调用）在底层无法解决
- **结论**：性价比极低，不推荐

### 选项 B（二次开发）- 强烈推荐
- Fork/基于 openclaw-control-center 做二次开发，保留核心扩展点
- **优势**：90% 功能已有、架构更优、1-2 周即可完成核心定制
- **主要工作**：飞书双向集成 + 办公用户适配 + 角色定制
- **迁移成本**：仅需移植 adapter 中的 YAML 模板配置系统（约 3 个文件）
- **结论**：三视角一致推荐，性价比最高

## 7. 决策摘要

**结论**：基于 openclaw-control-center 做二次开发（选项 B）
**依据**：技术/产品/办公场景三视角一致，功能覆盖度、架构质量、开发成本均指向同一方向
**核心增值工作**：飞书双向集成（P0）+ 办公用户信息层级简化
**迁移成本**：低，仅 3 个核心文件需移植
**预期收益**：节省 4-6 周开发时间，起点更高，扩展性更强
