# 缓存降级策略决策

## 决策背景
在办公控制台增强项目技术验证阶段，适配层缓存策略出现了以下争议：

### 原缓存策略假设
1. **缓存命中**：返回 200 OK，包含 `cached: true`
2. **缓存未命中但Gateway可用**：调用Gateway后返回 200 OK 新数据
3. **缓存有效但Gateway不可用**：强制返回 503 Service Unavailable，同时包含 `fallbackData: true, cachedData: {数据}`
   - 这是为了防止前端“静默使用过时缓存”而不自知

### 技术验证发现的问题
在真实联调中，`503 + cachedData` 方案面临以下挑战：
1. 契约复杂性增加，前端需要额外处理 503 但含有有效数据的特殊场景
2. 在 office 环境下，`503` 状态码可能被监控系统误报为系统级故障
3. 实际办公用户对“状态码”与“实际可读数据”的优先级是“先拿到有用信息，再看警示标签”

## 决策
基于产品、架构、后端验证结果，做出以下正式决策：

### 1. 缓存降级行为变更
**原方案**：`503 Service Unavailable` + `cachedData`
**新方案**：`200 OK` + `cached: true, stale: true`

**原理变更**：
- Gateway 不可用且有有效缓存时，服务仍返回 200 OK
- 但 `cached: true, stale: true` 强标识表示“降级模式”
- 前端必须根据 `stale` 字段展示明显降级提示（如警告色、说明文案）

### 2. 无缓存+Gateway不可用的兜底
**保持不变**：`503 Service Unavailable`
- 不包含 `cachedData`（因为没有缓存）
- 前端应展示"后端服务当前不可用"等强阻塞性提示

### 3. 性能风险处理原则
聚合未命中性能（缓存失效时的首次请求慢问题）属于：
1. **已知风险**：承认存在，不掩饰
2. **非阻塞性质**：不会导致系统崩溃，只会影响单次请求体验
3. **明确写入结论**：在验证结论中写明“已识别，待优化”，而不是“已解决”

### 4. 状态区分标准
必须明确区分以下三种情况：
1. **已解决**：问题有完整修复方案并已验证通过
2. **已接受风险**：问题存在，在当前版本可接受，有明确跟踪计划  
3. **仍待后续优化**：属于后续版本规划，不在当前 MVP 排期

## 决策依据
1. **产品友好性**：办公室场景中，用户关心“能不能继续工作”而非“HTTP语义的完整性”
2. **工程可行性**：`503 + cachedData` 模式增加了不必要的特殊路径处理复杂度
3. **风险可管理性**：`stale: true` 强提示已足够让用户知晓数据新鲜度问题
4. **未来兼容性**：此方案仍允许未来升级为更复杂的降级策略

## 实施要求
1. **立即生效**：此决策替代之前关于“必须503才能返回缓存数据”的任何假设
2. **契约更新**：更新 `/office/dashboard` API 契约，将降级响应从 503 改为 200 + `stale: true`
3. **前端配合**：前端必须对 `stale: true` 做明确视觉提示
4. **测试覆盖**：新增测试用例验证降级路径
5. **文档更新**：将此决策写入项目协议与错误处理手册

## 影响分析
### 积极影响
1. 简化接口契约，减少特殊状态码处理
2. 提供更符合办公习惯的用户体验
3. 降低监控系统误报风险
4. 为后续“多级降级”策略保留扩展能力

### 风险与应对
1. **用户可能忽视陈旧数据提示** → 通过强视觉提示（警告色、图标、文案）规避
2. **监控可能无法捕捉降级事件** → 通过服务端主动日志+告警系统补充
3. **性能瓶颈隐藏** → 强制在启动验收中写明已知风险，并在首版上线后优先优化
