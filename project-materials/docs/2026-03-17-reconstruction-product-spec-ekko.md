## 〇、产品定位纠正声明

> 本文档是对之前产品规格的定位纠正版本。核心变化：目标用户不再是"普通办公用户"，而是**有 AI 应用能力、愿意折腾、用 AI 提升效率的专业人群**。所有功能设计、优先级、飞书策略均基于此定位重新校准。

## 一、目标用户画像（更新版）

### 1.1 核心用户：效率驱动型 AI 从业者

**典型特征**：

| 维度 | 描述 |
|------|------|
| 技术能力 | 能看懂日志、理解 agent 调度、会折腾配置文件 |
| AI 使用深度 | 不只是"用 ChatGPT 写邮件"，会部署和调度多 agent 系统 |
| 目标导向 | 关注结果和文件状态，不需要追踪每一步执行过程 |
| 工作场景 | 多项目并行、批量任务、有明确的输出交付物 |
| 工具偏好 | 习惯 Web 后台管理视图，接受一定学习成本 |
| 对通知的态度 | 不想被打扰，只要关键事件推到飞书即可，其余在控制台自查 |

**不是我们的用户**：
- 需要手把手引导的小白
- 期望全程在飞书操作的轻量用户
- 只使用单个 AI Agent、无多项目并行需求的偶发用户

### 1.2 Jayce 角色定位纠正

> Jayce（AI 办公顾问）过去被设定为"代表普通办公白领"的视角，这是错的。

**更新后的 Jayce 定位**：

- **不是**普通白领，**是**有学习意愿、愿意拥抱 AI 工具的知识工作者
- 站在"想用 AI 提升个人效率"的专业人群视角提供建议
- 关注哪些场景 AI 能真正节省时间，而不是功能完整性
- 会验收"这个控制台能让我更快看到想要的信息"，而不是"这个功能普通人看得懂吗"

### 1.3 核心使用场景

| 场景 | 频率 | 典型行为 |
|------|------|---------|
| 站点监控 | 高频（日常） | 快速查看 agent 状态、项目健康度，有问题直接跳详情 |
| 知识库管理 | 高频 | 在控制台浏览/查找文档、任务交接物、决策记录 |
| 批量文档处理 | 中频 | 发起批量任务，查看执行结果文件，不需要监控每步 |
| 多项目并行开发 | 高频 | 多项目任务看板、agent 跨项目工作状态全览 |
| 模型配置与调优 | 低频 | 修改 agent 配置、切换模型、调整 memory 文件 |

## 二、产品核心主张

**主战场**：Web 后台（桌面端），最小视图宽度 1200px

**三个"不"原则**：
1. **不逐步跟踪**：用户关注最终结果和文件，不需要追踪每步执行过程
2. **不飞书重交互**：飞书只推关键通知 + 跳转链接，操作全在 Web 控制台
3. **不引入数据库**：文件系统就是数据库，保持零额外基础设施依赖

**一个核心体验**：
> 打开控制台 → 5 秒内看到想要的结果状态 → 有问题点进去处理 → 关闭

## 三、8 大功能分区定义与优先级

### P0（核心分区，必须在前 3 个迭代完成）

#### 3.1 Overview（总览）— P0，Iter-1 完成

**产品定义**：系统健康仪表盘，用户打开控制台首先看到的全局状态视图。

**对目标用户的核心价值**：
- 多项目并行时，**1 次浏览** 知道所有项目 + agent 的当前健康状态
- 发现异常直接点击跳转，不需要逐个翻看

**关键功能**：
- Gateway 连接状态（绿/黄/红）
- Agent 总数 / 活跃数 / 异常数（概览卡片）
- 最近 24h 项目活动时间线（by 文件落地时间）
- 快速跳转入口（8 个分区）
- 系统版本、磁盘用量

**不做什么**：
- 不展示每条 agent 消息的内容（那是 Collaboration 的事）
- 不做实时动态 feed（避免用户沉迷跟踪过程）

---

#### 3.2 Agents（Agent 状态）— P0，Iter-1 完成

**产品定义**：所有 agent 的配置和运行状态管理视图。

**对目标用户的核心价值**：
- 站点监控场景下，快速判断哪个 agent 出了问题
- 模型配置场景下，查看 fallback 链、修改模型绑定

**关键功能**：
- Agent 列表（名称、角色描述来自 SOUL.md/IDENTITY.md、状态、模型）
- 每个 agent 的模型配置和 fallback 链展示
- workspace 文件树预览（核心文件：SOUL.md、USER.md、TOOLS.md、HEARTBEAT.md）
- 最近 session 列表（时间、来源渠道、状态）

**不做什么**：
- 不展示 session 里的完整对话内容（那是 Collaboration 分区）
- 不提供 agent 创建向导（配置通过文件/Settings 管理）

---

#### 3.3 Tasks（任务）— P0，Iter-4 完成

**产品定义**：基于项目 `tasks/` 目录文件的任务看板视图。

**对目标用户的核心价值**：
- 多项目并行时，一个看板视图看所有任务的状态
- 文件就是任务 source of truth，控制台只做可视化，不引入额外数据源

**关键功能**：
- 任务看板（按状态分组：待处理 / 进行中 / 已完成 / 阻塞）
- 任务详情（Markdown 渲染 + 元信息提取：标题/状态/责任人/日期）
- 按项目过滤
- 文件变更自动刷新（<2s，via SSE）
- 任务文件编辑（写操作需经过 Gate）

**不做什么**：
- 不做复杂的任务依赖图（本期不做，保持 markdown 文件优先）
- 不提供任务评论功能（保持飞书轻量通知策略）

---

### P1（重要分区，Iter-4~5 完成）

#### 3.4 Docs（文档工作台）— P1，Iter-4 完成

**产品定义**：项目文档、决策、协议、交接物的统一浏览和管理工作台。

**对目标用户的核心价值**：
- 知识库管理场景下，在控制台直接浏览所有落地文件，不需要 SSH 进服务器
- 批量文档处理后，在 Docs 里直接看输出结果文件

**关键功能**：
- 多项目目录树导航（docs/、status/、reviews/、protocols/、decisions/、artifacts/）
- Markdown 渲染预览
- 全文搜索（grep 级别，无需数据库）
- 文件变更实时感知（SSE 联动）
- 文档编辑（写操作 Gate 保护）

**不做什么**：
- 不做富文本编辑器（Markdown 源码 + 预览就够了）
- 不做版本历史（Git 有这个，控制台不需要重复）

---

#### 3.5 Collaboration（协作/会话接力）— P1，Iter-5 完成

**产品定义**：活跃 session 和跨 agent 协作路径的可视化视图。

**对目标用户的核心价值**：
- 不需要追踪每步过程，但需要知道"当前有哪些协作链在跑"
- 发现协作链卡住时，快速定位是哪个 agent

**关键功能**：
- 活跃 session 列表（来源渠道、目标 agent、状态、时长）
- 父子 session 关系树（subagent spawn 链）
- session 消息摘要（最近 N 条，不是完整对话）

**不做什么**：
- 不展示完整对话内容（用户关心结果，不是过程）
- 不提供手动干预 session 的操作（本期只读）

---

#### 3.6 Memory（记忆管理）— P1，Iter-5 完成

**产品定义**：各 agent workspace 核心文件的查看和管理界面。

**对目标用户的核心价值**：
- 调优 agent 时，在控制台直接查看和编辑 SOUL.md、USER.md、TOOLS.md，不需要 SSH
- 多项目并行时，快速确认各 agent 的记忆配置是否对齐

**关键功能**：
- 各 agent memory 文件列表（SOUL.md / USER.md / TOOLS.md / MEMORY.md 等）
- 文件内容查看（Markdown 渲染）
- 文件编辑（写操作 Gate，dry-run 默认）
- 文件大小和用量统计

**不做什么**：
- 不提供 memory "总结压缩" UI 功能（由 agent 自行管理）

---

#### 3.7 Usage（用量/成本）— P1，Iter-5 完成

**产品定义**：Token 消耗和成本的统计视图。

**对目标用户的核心价值**：
- 多 agent 并行时，快速发现哪个 agent 或项目在异常烧 token
- 成本优化场景下，有数据支撑模型切换决策

**关键功能**：
- Token 消耗（按 agent、按模型、按时间段）
- 成本估算（基于模型定价表）
- 上下文窗口压力分析
- 趋势图（日/周/月）
- 阈值告警配置（超出预算 → 飞书通知）

**不做什么**：
- 不提供账单支付功能
- 不做实时 token 逐字计费（统计粒度到会话级别足够）

---

### P2（完善分区，Iter-6 完成）

#### 3.8 Settings（设置）— P2，Iter-6 完成

**产品定义**：OpenClaw 核心配置查看与模板管理。

**对目标用户的核心价值**：
- 配置管理场景下，不需要手动编辑 JSON，在控制台图形化查看
- YAML 模板是差异化资产，保留并强化为可视化管理

**关键功能**：
- OpenClaw 核心配置查看（agents / models / bindings / channels）
- YAML 模板管理（查看 / 应用 / 校验）
- 安全设置（READONLY_MODE 状态、Token 配置）
- Gateway 连接状态与诊断
- 写操作全部 dry-run 默认

**不做什么**：
- 不提供批量 agent 注册向导（初期）
- 不开放 openclaw.json 直接编辑（只读查看 + 模板应用方式修改）

## 五、飞书通知策略（轻量，非重交互）

### 5.1 设计原则

> 飞书 = 类 TG 风格通知机器人，不是操作界面

- ✅ 推送关键事件 + 跳转链接
- ✅ 通知简洁，15 字以内能 get 核心
- ❌ 不在飞书里展示完整报表
- ❌ 不在飞书里做审批、确认、复杂操作
- ❌ 不推送正常过程信息（只推异常和关键节点）

### 5.2 飞书通知触发场景

| 事件 | 通知时机 | 优先级 | 消息模板示例 |
|------|---------|--------|------------|
| Agent 状态异常 | 状态变为 error 时 | P0 | `⚠️ Agent [jax] 状态异常 → [查看详情]` |
| 任务进入阻塞 | 检测到 blocker 标记 | P0 | `🚨 阻塞：[任务名] → [控制台]` |
| 阶段完成 | Phase 完成标记出现 | P0 | `✅ Phase 10 Iter-1 完成 → [详情]` |
| Token 超预算 | 用量超阈值 | P1 | `📊 token 用量超阈值：[agent/jax] → [控制台]` |
| 安全事件 | 写操作审计异常 | P1 | `🔐 安全事件：[描述]` |
| 项目日报 | 定时（cron 每天 18:00） | P2 | `📋 项目日报 → [控制台]` |
| Gateway 断连 | Gateway 连接中断超 30s | P0 | `🔌 Gateway 断连，已尝试重连` |

### 5.3 不应推送飞书的内容

- agent 正在执行某个步骤（过程细节）
- 文件被读取、session 被创建（正常运转信息）
- 每条 agent 回复内容（太细，会 spam）
- 定时心跳正常状态报告（没事不打扰）

### 5.4 飞书通知设计约束

- 消息不超过 200 字符
- 必须带 Web 控制台跳转链接（详情在 Web，不在飞书展开）
- 控制台内有完整通知历史（飞书只是推送渠道，不是 source of truth）

## 六、各迭代阶段产品验收标准

### Iteration 1：基础架构升级（产品验收角度）

**验收条件**：

1. **响应速度**：Overview 页面打开后，数据加载完成时间 < 500ms（当前 1-3s）
2. **数据准确性**：agent 列表、状态与 `openclaw.json` 文件内容一致
3. **实时性**：Gateway 连接状态变化后，控制台 Overview 在 <3s 内更新
4. **降级可用**：Gateway 断连时，控制台仍能展示文件直读数据，不白屏

**产品 Pass 标准**：用户打开控制台，5 秒内能看到所有 agent 的当前状态。

---

### Iteration 2：安全框架（产品验收角度）

**验收条件**：

1. **写操作保护**：`READONLY_MODE=true` 时，所有编辑按钮 disabled 且 tooltip 说明原因
2. **干净的 UI 反馈**：403 错误展示友好提示，不是裸错误信息
3. **审计可见**：写操作审计日志可在 Docs 分区（或 Settings）查看
4. **不影响正常阅览**：安全框架引入后，所有 GET 操作速度不退化

**产品 Pass 标准**：用户在只读模式下操作，体验到的是"这里不能改"而不是"这里报错了"。

---

### Iteration 3：前端分区化（产品验收角度）

**验收条件**：

1. **导航清晰**：8 个分区导航标签，在 1200px 宽度下不折叠、不遮挡
2. **分区状态保持**：在分区间切换时，已展开的内容不重置
3. **Overview + Agents 数据正确**：这两个 P0 分区显示数据与文件一致
4. **加载态**：数据加载中时有 skeleton 或 loading 指示，不白屏

**产品 Pass 标准**：导航体验流畅，用户能在 3 次点击内到达任意功能区。

---

### Iteration 4：Tasks + Docs 分区（产品验收角度）

**验收条件**：

1. **任务看板**：项目 `tasks/` 目录下的 Markdown 文件被正确解析为任务卡片，状态分组正确
2. **文档树**：`docs/`、`decisions/`、`protocols/` 等目录可导航，Markdown 正确渲染
3. **实时刷新**：文件被修改后，<2s 内控制台对应区域自动更新
4. **搜索可用**：全文搜索能在 2s 内返回结果

**产品 Pass 标准**：在控制台打开 Docs，能找到 Jax 的架构方案文件并正确显示。

---

### Iteration 5：Collaboration + Usage + Memory 分区（产品验收角度）

**验收条件**：

1. **Collaboration**：能看到当前活跃的 session 列表及父子关系，状态数据延迟 <5s
2. **Usage**：按 agent 维度展示的 token 消耗数据，图表可渲染
3. **Memory**：各 agent 的 SOUL.md、USER.md 可查看；编辑操作有 dry-run 确认弹窗
4. **写操作 Gate**：Memory 编辑必须经过二次确认（"确认修改 xxx 的记忆文件？"）

**产品 Pass 标准**：调优 agent 时，不需要 SSH，在控制台完成查看和编辑。

---

### Iteration 6：Settings + 飞书通知 + 清理（产品验收角度）

**验收条件**：

1. **Settings**：OpenClaw 核心配置可查看（不可直接编辑 JSON），模板系统可用
2. **飞书通知**：agent 状态异常后 <5s 飞书群收到通知，消息格式符合规范
3. **通知历史**：控制台内可查看所有通知历史，按类型过滤
4. **CLI adapter 已移除**：所有 API 响应时间满足 <200ms，无旧的 CLI 子进程调用
5. **全分区端到端**：8 个分区数据均正确展示，无 broken 状态

**产品 Pass 标准**：产品达到可以向外部演示的状态，控制台能代表"有 AI 应用能力的效率型用户"日常工作的核心操作面。

## 七、产品边界说明（本次重构范围）

### 在范围内

- 8 大功能分区的 Web 控制台（桌面端）
- 飞书轻量通知推送
- 文件系统驱动的任务/文档/记忆管理
- 基础安全框架（只读模式 + Token + 写操作 Gate）
- 现有模板系统的保留和整合

### 不在范围内（明确 Out-of-Scope）

- 移动端适配（桌面端优先，最小宽度 1200px）
- 复杂审批工作流（任务编辑直接走文件，无审批链）
- 飞书内做复杂操作（飞书永远只是通知渠道）
- 独立数据库（文件系统就是数据库）
- 多用户权限体系（当前单用户，Token 作为基础隔离）
- 跨 OpenClaw 实例管理（单实例管控即可）

## 八、Jayce（AI 办公顾问）验收视角

Jayce 在各迭代的验收侧重点：

| 迭代 | Jayce 关注的验收场景 |
|------|---------------------|
| Iter-1 | 打开控制台能快速看到自己的 agent 状态，有问题时能快速跳到详情 |
| Iter-3 | 导航自然，不需要看文档就能找到 8 个分区 |
| Iter-4 | 在 Tasks 里快速找到"当前阻塞的任务"；在 Docs 里找到最新的架构方案 |
| Iter-5 | 知道自己今天 token 花了多少，哪个项目花得最多 |
| Iter-6 | 收到飞书通知后，点链接进控制台 3 步内解决问题，然后关掉 |

**Jayce 的核心验收问句**：
> "我今天需要查一件事，从打开控制台到找到答案，要几步？"
> 目标：核心场景不超过 3 步。

## 九、与 Jax 架构方案的对齐确认

| 架构决策 | 产品认可 | 说明 |
|---------|---------|------|
| 无独立数据库 | ✅ 支持 | 与"文件落地"产品价值一致 |
| 渐进式迭代 6 步 | ✅ 支持 | 每迭代有独立验收标准，符合产品验收节奏 |
| READONLY_MODE 默认开启 | ✅ 支持 | 目标用户可以接受"要改就显式解锁"的设计 |
