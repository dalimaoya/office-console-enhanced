## 二、各分区功能深度对比

| 功能项 | ODA | CC |
|-------|-----|-----|
| 系统状态 KPI 卡片 | ✅ 有（agent 总数、活跃数、任务数） | ✅ 有（更丰富，含预算风险、待处理事项） |

## 五、优先级建议（产品视角）

根据对用户价值和实现复杂度的综合判断：

### P0 — 影响日常使用的核心缺口

1. **任务状态可写** — 用户在控制台看到任务但无法更新状态，是一个严重的操作中断。建议 Iter-7 补齐 PATCH /api/tasks/:id/status。
2. **全站搜索** — 用户在列表变长后无法快速定位，是高频痛点。建议 Iter-8 引入基础搜索 API。
3. **接线状态 + 安全风险摘要** — 用户看到数据缺失时无法自助诊断，直接影响信任感。建议在设置分区补齐两张卡片。

### P1 — 提升产品成熟度的重要缺口

4. **上下文压力指标** — 对用量分区有实质价值，帮助用户预判成本和性能风险。
5. **记忆状态可用性** — 记忆分区目前只是文件列表，加上健康状态指示大幅提升实用性。
6. **Digest 摘要 + 回放索引** — 帮助用户回顾历史、排查问题。
7. **.env 模板文件** — 改善新用户首次接入体验，避免硬编码配置问题。

### P2 — 差异化和生产就绪

8. **审批操作系统** — 高风险操作需要审批机制，是生产场景必备。
9. **导入/导出** — 状态备份和恢复能力。
10. **国际化（中英双语）** — 如果团队有英文用户，此项重要性上升为 P1。
11. **PM2 + 安装向导** — 生产部署就绪的必要条件。

### 保持/强化我们的差异化优势

- ✅ **SSE 实时推送** — 这是我们相比 CC 最大的技术优势，必须维护。
- ✅ **飞书 Webhook 告警** — 深度融合飞书生态，是面向中国办公场景的独特竞争力。
- ✅ **YAML 模板系统** — 降低 agent 配置门槛，是办公场景的独特价值。

## 六、总结

### 我们有而 CC 没有的（ODA 独特优势）

1. SSE 实时推送到浏览器（Gateway → SseHub → EventSource 全链路）
2. 飞书 Webhook 告警通知
3. YAML 模板系统（预置办公模板，一键应用）
4. IP 白名单写操作控制
5. 零构建运行（tsx 直接执行 TypeScript）
6. 开发热重载（tsx watch）
7. Controller/Service/Data 三层清晰分离架构

### CC 有而我们没有的（主要差距，共 53 个功能点）

- 任务管理写操作（创建/更新/过滤）
- 全站搜索
- 审批操作系统
- 预算治理引擎
- 费用趋势与 Token 消耗归因
- 上下文压力指标
- 任务回放索引
- Cron 定时任务总览
- 待办操作队列
- 异常流管理
- Digest 摘要
- 项目管理 CRUD
- 导入/导出/备份
- 接线状态诊断
- 安全风险摘要
- 版本更新状态
- 国际化（中英双语）
- PM2 生产部署
- 安装向导
- Done Checklist 评分体系

### 量化总结

| 指标 | 数值 |
|------|------|
| CC 总功能点数 | ~87 个 |
| ODA 已覆盖功能点数 | ~34 个 |
| **总体覆盖率** | **~39%** |
| CC API 端点数 | 32 个 |
| ODA API 端点数 | 14 个 |
| 端点覆盖率 | ~44% |
| ODA 独有功能点 | 7 个（实时通信、飞书、模板等） |

> **产品定性结论**：ODA 是一个"观察型控制台"（只看不写），CC 是一个"运营型控制台"（看 + 做）。要从观察型进化到运营型，最关键的补齐动作是：任务写操作 → 搜索 → 审批 → 费用分析。
