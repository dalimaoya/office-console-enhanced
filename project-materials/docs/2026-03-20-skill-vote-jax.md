# 「项目过程沉淀 skill」方案 — 架构评审与投票

**评审人：** Jax（架构守门人）
**日期：** 2026-03-20
**结论：** ✅ 有条件赞成

---

## 一、架构可行性评估

### 可行
- skill 本质是「约定 + 读写工具」，不引入运行时依赖，耦合度天然低
- OpenClaw skill 机制本身支持跨角色加载，复用路径成立

### 风险点
1. **并发写冲突**：多角色同时写同一日志文件会产生竞态。skill 必须约定「一角色一文件」或「追加写 + 时间戳前缀」，不能用覆盖写
2. **路径耦合**：如果 skill 硬编码项目路径，反而比当前方案更脆弱。必须用相对路径或环境变量注入项目根
3. **一致性幻觉**：skill 只能约定格式，无法强制执行。角色不遵守约定时，skill 本身无纠错能力——这不是 bug，是边界，需接受

## 二、适合封装 vs 不适合封装

| 适合封装进 skill | 不适合封装 |
|---|---|
| 会话摘要写入（模板 + 路径约定） | 角色 SOUL.md / IDENTITY.md（属于角色私有状态） |
| 每日日志追加（格式 + 命名规则） | 项目决策文档（内容非结构化，不宜模板化） |
| 上下文快照的读取索引 | Teemo 的调度逻辑（属于 L1 层职责） |
| 跨角色共享文件的目录结构约定 | 心跳/巡检状态（属于 L0 层，不应下沉到 skill） |

## 三、最小可行设计（MVP）

### 存什么
- `sessions/{role}/{YYYY-MM-DD-HHmm}.md` — 会话摘要
- `daily/{YYYY-MM-DD}.md` — 每日聊天日志（追加模式）

### 怎么写
- skill 提供 SKILL.md 指令：角色在会话结束时，按模板写入摘要
- 日志用 `append` 模式，每条带 `[HH:MM] [role]` 前缀
- 不引入锁机制，靠「一角色一文件」避免冲突

### 怎么读
- skill 提供索引指令：按日期/角色检索摘要
- 读取时用 `read` 工具 + glob 匹配，不需要额外索引文件
- 首次实现不做全文搜索，后续可扩展

### 不做什么（MVP 边界）
- 不做自动压缩/归档
- 不做跨项目共享（一个 skill 实例绑定一个项目）
- 不做实时同步或通知

## 四、投票结论

### ✅ 有条件赞成

**赞成理由：**
- 将沉淀逻辑从各角色 AGENTS.md 中抽离，减少重复配置，方向正确
- skill 机制天然支持复用，不需要额外基础设施
- MVP 足够轻量，失败成本低

**条件（必须满足）：**
1. **路径不硬编码**：skill 必须通过项目根变量解析路径，不写死绝对路径
2. **写入模式为追加**：禁止覆盖写，避免并发丢数据
3. **不侵入角色 SOUL/IDENTITY**：skill 只管沉淀文件，不碰角色私有状态
4. **先在一个项目试点**：当前 office-console-enhanced 项目先跑通，再推广

**反对全盘封装的理由：**
- 项目决策文档、架构文档等非结构化内容不应模板化，强行封装会降低表达力
- 心跳/巡检状态属于 L0 层职责，不应下沉到 skill 层
