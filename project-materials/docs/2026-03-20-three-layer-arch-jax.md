## 一、三层定义与职责边界

### L0 — 全局巡检层

**定位：** 系统的「瞭望塔」，不参与业务执行，只负责监督、调度触发和异常通知。

| 维度 | 说明 |
|---|---|
| 职责 | 心跳巡检、角色存活检测、异常升级、定时调度触发、全局通知 |
| 不做 | 不做业务决策、不直接调度角色、不持有项目上下文 |
| 触发方式 | cron / heartbeat |
| 输出 | 通知消息、blocker 升级、调度信号（发给 L1） |
| 实例 | 当前无独立实体，可由系统 cron + 简单巡检脚本承担，后续可独立为角色 |

### L1 — 项目总控层

**定位：** 项目的「单入口指挥官」，所有项目指令经此层分发。

| 维度 | 说明 |
|---|---|
| 职责 | 接收用户指令、任务拆解、角色调度、进度追踪、交接协调、成果物验收 |
| 不做 | 不做具体执行（代码/设计/测试）、不做全局监控 |
| 触发方式 | 用户消息、L0 调度信号 |
| 输出 | 子任务分派、进度报告、交接指令 |
| 实例 | Teemo（提莫） |

### L2 — 专业角色层

**定位：** 实际干活的「专家组」，每个角色内化执行能力，不依赖消息中转。

| 维度 | 说明 |
|---|---|
| 职责 | 接收 L1 分派的任务、独立执行、产出成果物、写入共享文件区、完成后自行通知 |
| 不做 | 不做跨角色协调（交给 L1）、不做全局监控（交给 L0） |
| 触发方式 | L1 subagent 调度 |
| 输出 | 成果物文件、完成通知、blocker 上报 |
| 实例 | Jax / Jayce / Ekko / Ezreal / Leona / Galio / Lux |

### 层间通信规则

```
L0 ──(调度信号/异常通知)──▶ L1
L1 ──(任务分派/subagent)──▶ L2
L2 ──(成果物/完成通知)────▶ L1
L2 ──(群通知)─────────────▶ 飞书群（仅通知，非交接）
L0 ──(异常告警)───────────▶ 用户/飞书群
```

**关键约束：**
- L2 之间不直接交接，必须经 L1 协调
- L0 不直接指挥 L2，只能通过 L1
- 成果物交接以文件为准，群消息仅作通知

## 二、与「当皇上」架构的对比

「当皇上」架构的核心思路：用户像皇帝一样发旨意，通过消息路由（太监传话）层层下达。

| 维度 | 「当皇上」（消息路由） | 三层架构 |
|---|---|---|
| 调度方式 | 消息转发链，角色间传话 | subagent 直接调度，无中间传话 |
| 执行模式 | 角色收到消息后自行解读执行 | 角色收到结构化任务，明确输入输出 |
| 失败处理 | 消息丢失 = 任务丢失，难追溯 | 任务有明确状态，blocker 可上报 |
| 上下文传递 | 靠消息内容携带，容易丢失/变形 | 靠共享文件区 + 任务描述，持久化 |
| 扩展性 | 加角色 = 加路由规则，复杂度线性增长 | 加角色 = L2 加实例，L1 逻辑不变 |
| 监督能力 | 无独立监督，靠人工检查 | L0 独立巡检，异常自动升级 |

**为什么不用消息路由：**
1. **消息是易失的**：群消息不是可靠的任务载体，丢了就丢了
2. **传话会变形**：每经过一个角色转述，上下文损耗一层
3. **无法监督**：消息路由模式下，没有独立的监督视角，全靠人盯
4. **调试困难**：出问题时，消息链条难以回溯，不如文件 + 日志清晰

## 三、L0 巡检层具体职责建议

### 心跳规则

| 规则 | 说明 |
|---|---|
| 频率 | 每 30 分钟一次（可配置） |
| 检测对象 | L1（Teemo）存活状态 |
| 超时阈值 | 连续 2 次心跳无响应视为异常 |
| 异常动作 | 向飞书群发送告警，抄送用户 |

### 监督范围

1. **角色存活**：L1 是否响应心跳（L2 由 L1 负责监督，L0 不越级）
2. **任务超时**：L1 分派的任务是否在预期时间内完成（需 L1 上报任务状态）
3. **blocker 升级**：L2 上报的 blocker 如果 L1 未在 1 小时内处理，L0 升级给用户
4. **日常巡检**：检查共享文件区是否有孤立文件（无归属的成果物）

### 通知触发条件

| 触发条件 | 通知对象 | 通知方式 |
|---|---|---|
| L1 心跳超时 | 用户 | 飞书群 @用户 |
| 任务超时（>2h 未完成） | L1 + 用户 | 飞书群通知 |
| blocker 未处理（>1h） | 用户 | 飞书群 @用户 |
| 每日摘要（可选） | 飞书群 | 定时发送 |

### L0 实现建议（当前阶段）

当前不需要独立角色，用以下方式实现：
- OpenClaw cron 定时触发心跳检查脚本
- 脚本检查 L1 最后活跃时间（基于文件时间戳或心跳状态文件）
- 异常时通过 message 工具发送飞书通知
- 后续规模扩大后，可升级为独立 agent
