# 架构与故事线论证 —— 产品视角（Ekko）

**日期：** 2026-03-20  
**角色：** 艾克（Product）  
**论证类型：** 架构合理性 + 故事线选型

---

## 一、目标用户与核心场景

我们的目标用户是**有深度项目管理需求的技术团队/个人开发者**，核心场景是：

- 一个项目从需求→设计→开发→测试→交付的全流程协作
- 用户把任务交给 orchestrator，由专业角色分工执行，产出可交接的成果物
- **不是**泛用聊天助手，不是"什么都能聊"的万能 AI

## 二、架构对比：我们 vs「当皇上」

| 维度 | 当皇上（三省六部） | 我们（提莫体系） |
|------|-------------------|-----------------|
| 调度模型 | 司礼监→内阁→六部，层级式消息路由 | 提莫单入口→专业角色直派 |
| 交接方式 | 消息会话流转 | 共享文件区，成果物驱动 |
| 执行纵深 | 六部扁平，缺乏角色内专精 | 每个角色有明确专业边界和产出标准 |
| 定位 | 泛用聊天，多人使用 | 深度项目执行，单项目纵深 |
| 可追溯性 | 消息流，易丢失上下文 | 文件成果物，天然可追溯 |

**产品判断：我们的架构更合理。**

核心理由：
1. **成果物驱动 > 消息驱动**：项目管理的本质是产出可交付物，不是聊天记录。文件交接天然支持版本管理、审计追溯。
2. **专业纵深 > 扁平分工**：「当皇上」的六部更像是"分类路由"，每个部没有独立的专业判断力。我们的角色是真正的 specialist——产品有验收标准，架构有技术选型，QA 有测试策略。
3. **单入口 + 文件交接 = 确定性高**：消息会话交接容易丢失上下文，角色间互通会引入复杂的状态管理。我们的设计更简洁、更可控。

**不需要调整架构。** 当前设计在深度项目场景下是正确选择。

## 三、故事线论证：英雄联盟 vs 军队体系 vs 项目管理体系

### 评估维度

| 维度 | 英雄联盟 | 军队体系 | 项目管理体系 |
|------|---------|---------|-------------|
| **角色辨识度** | ⭐⭐⭐⭐⭐ 每个英雄有独特人格 | ⭐⭐⭐ 军衔有序但角色同质 | ⭐⭐ 职能标签，无人格 |
| **话题度/传播力** | ⭐⭐⭐⭐⭐ 游戏 IP 天然有粉丝基础 | ⭐⭐ 严肃正式，缺乏趣味 | ⭐ 纯工具感，无传播力 |
| **项目管理契合度** | ⭐⭐⭐ 团战协作隐喻成立 | ⭐⭐⭐⭐ 指挥链、任务执行天然匹配 | ⭐⭐⭐⭐⭐ 直接映射但无趣 |
| **认知粘性** | ⭐⭐⭐⭐⭐ 用户记得住"提莫""艾克" | ⭐⭐⭐ 记得住"司令""参谋" | ⭐ "PM""Dev Lead"毫无记忆点 |
| **扩展性** | ⭐⭐⭐⭐⭐ 160+ 英雄可选 | ⭐⭐⭐ 军种有限 | ⭐⭐ 职能有限 |
| **与「当皇上」差异化** | ⭐⭐⭐⭐⭐ 完全不同的文化圈层 | ⭐⭐ 同属"等级制"隐喻 | ⭐⭐⭐ 不同但无趣 |

### 深度分析

**英雄联盟的核心优势不是"好玩"，而是角色-能力的天然映射：**

- 提莫（Teemo）→ 全图视野、信息收集 → orchestrator 的信息中枢定位
- 艾克（Ekko）→ 时间操控、节奏掌握 → 产品经理的优先级和节奏管理
- 贾克斯（Jax）→ 万能武器大师 → 架构师的技术选型能力
- 伊泽瑞尔（Ezreal）→ 精准远程输出 → 前端的精确实现
- 雷欧娜（Leona）→ 护盾守卫 → 后端的稳定性保障
- 加里奥（Galio）→ 反魔法守护者 → QA 的质量守卫
- 拉克丝（Lux）→ 光之束缚 → UI 设计的视觉表达
- 杰斯（Jayce）→ 变形双态 → 办公工具的灵活切换

**这些映射不是硬凑的，是有认知逻辑的。** 军队体系做不到这种角色-能力的自然关联（"上尉负责前端"毫无认知锚点）。项目管理体系则直接取消了趣味性。

**军队体系的致命问题：**
- 军队隐喻的核心是"服从命令"，但项目管理需要的是"专业判断"
- 军衔等级暗示上下级关系，与我们 specialist 平等协作的设计矛盾
- 与「当皇上」的皇权体系在文化隐喻上太接近，差异化不足

**项目管理体系（PM/Tech Lead/Dev/QA）的问题：**
- 毫无记忆点，用户不会对"Dev Lead Agent"产生任何情感连接
- 传播力为零——没人会分享"我的 QA Agent 发现了个 bug"
- 但如果有人说"加里奥拦住了一个生产事故"，这就是一个有画面感的故事

## 四、产品推荐

### ⏱️ 结论：保留英雄联盟体系

**理由优先级排序：**

1. **差异化是产品生死线。** 「当皇上」选了古代皇权体系，这在中文 AI 社区已经占位。如果我们换军队体系，就是在同一隐喻维度竞争。英雄联盟把我们拉到完全不同的文化圈层——年轻、全球化、技术向。

2. **认知粘性决定用户留存。** 用户用了一周后还记不记得每个角色叫什么、干什么，直接影响他是否会继续用。英雄联盟的角色辨识度远超其他方案。

3. **角色-能力映射是天然的产品说明书。** 新用户看到"提莫负责全局视野"就大概知道 orchestrator 干什么。这省了大量 onboarding 成本。

4. **话题度和传播力是免费获客。** 英雄联盟的全球玩家基础 = 自来水传播。"用英雄联盟的方式做项目管理"本身就是一个有传播力的 positioning。

5. **扩展性充足。** 未来增加新角色（比如"金克丝"做 CI/CD 自动化）完全自然，160+ 英雄池提供了几乎无限的扩展空间。

### 需要注意的风险

- **IP 版权**：角色名和技能描述不能直接复用拳头的美术资产和官方文案，需确保在"致敬/二创"的合理范围内
- **非玩家用户**：部分用户没玩过英雄联盟，需要角色介绍做到"不玩也能懂"——这通过角色-能力的自然映射可以做到
- **避免过度 cosplay**：故事线是认知脚手架，不是 RPG。角色交互应保持专业感，不要变成"提莫在草丛里种蘑菇"式的中二对话

## 五、一句话总结

> 架构不需要动，故事线保留英雄联盟。我们和「当皇上」的差异不在架构层（都是 orchestrator + specialist），而在**产品定位层**——深度项目执行 vs 泛用聊天。英雄联盟故事线是差异化定位的放大器，不是装饰品。
