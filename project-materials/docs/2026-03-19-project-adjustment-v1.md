## 一、最终架构结论

### 1. 为什么不采用"司礼监—六部"作为核心执行模型

司礼监—六部的根本问题不是故事线，而是**执行模型的设计目标**：

- 司礼监是消息中枢，所有指令都经过它路由。随着项目推进，上下文线性增长，不可回避地面对"上下文爆炸"。这是消息驱动中心化架构的结构性缺陷，无法靠优化解决，只能靠架构换代。
- 六部之间的协作依赖 @mention 触发，属于即时消息驱动。这在轻度任务/娱乐场景中工作良好，但不适合持续多天、多阶段、有正式交接的深度项目执行。
- 司礼监面向的是"一次性任务 + 聊天答复"场景，不是"项目生命周期管理"场景。

**结论**：不采用，也不做局部借鉴改造。只借鉴其产品化呈现能力（一键部署、诊断、控制台）。

### 2. 为什么继续坚持文件驱动/对象驱动/正式交接

- **可追溯性**：文件是天然的版本载体，任何变更都有记录，不依赖会话连续性。
- **抗上下文爆炸**：角色之间的交接不走消息，走文件。每个角色只需要读取对应的 handoff/brief/review，不需要维护完整的对话历史。
- **可并发**：不同角色可以并发写入各自的文件区，不存在"必须等消息队列"的串行瓶颈。
- **外部可见**：文件可以同步到 GitHub、飞书文档、共享目录，团队成员（包括人类）可以直接检阅，不需要通过AI作为唯一入口。

**结论**：坚守，并在一期优先做"文件契约显式化"来强化这个设计。

### 3. 为什么故事线只放到皮肤层

- 故事线（英雄联盟角色代号）是用来帮助人类快速记忆和理解角色职能的叙事工具，不是架构约束。
- 把故事线嵌入到系统治理框架中（如"司礼监负责路由""六部负责执行"），会让架构决策受限于叙事逻辑，产生不必要的认知负担和改造阻力。
- 现有 LoL 角色代号和 SOUL.md/IDENTITY.md 配置已经形成了稳定的内部认知体系，继续保留成本低、价值高。换故事线的迁移成本远大于收益。

**结论**：故事线只作为内部代号，不参与治理框架设计。治理框架用项目管理语言。

### 4. 为什么整体治理框架应采用项目管理/指挥控制语言

- 政府体系（三省六部/司礼监）的设计目标是稳定性和合法性，不是执行效率。
- 军队体系的核心是指挥链（Command Chain）+ 单位职责（Unit Responsibility），非常接近项目管理的工作分解（WBS）+ 责任矩阵（RACI）。
- 项目管理体系的语言（目标/里程碑/交付物/验收/复盘）与我们的对象体系（brief/handoff/review/artifact）天然对应。

**结论**：对外产品语言收敛到 Mission Control / 项目控制台 / 项目指挥部，对内保留 LoL 代号。

## 二、产品主线重组

### 主线一：运行控制台（Control Center）

**定位**：实时可见 + 环境健康 + 状态总览

| 能力 | 描述 | 期次 |
|------|------|------|
| Gateway/Agent 状态总览 | Agent 在线状态、模型、当前任务、上下文压力 | **一期** |
| 运行健康监控 | 连接状态、响应延迟、SSE心跳、告警阈值 | **一期** |
| 项目状态总览 | 当前活跃项目、阶段、owner、阻塞项 | **一期** |
| Token用量与费用 | 今日/本周/按角色分层统计 | **一期** |
| 操作审计日志 | 关键操作记录，带时间戳 | **一期** |
| 环境诊断 | 一键检查配置完整性、连通性、权限 | **二期** |
| 异常摘要与根因推断 | 自动识别卡住/超时/角色失联 | **二期** |
| 跨项目调度视图 | 多项目并行状态、资源冲突检测 | **三期** |

### 主线二：项目组工厂（Team Factory）

**定位**：项目组的创建、配置、实例化

| 能力 | 描述 | 期次 |
|------|------|------|
| 角色原型池 | 已有8个specialist角色的标准化定义（职责/模型/工具集） | **一期** |
| 项目组模板 | 预定义的角色组合（如"开发团队模板"/"内容团队模板"） | **二期** |
| 皮肤包 | LoL代号↔对外产品名的映射配置，可替换 | **二期** |
| 项目实例 | 一个项目对应一套角色配置+文件目录+状态机 | **二期** |
| 一键实例化 | 从模板创建新项目组，自动生成目录结构+初始配置 | **三期** |
| 一键部署向导 | 新用户从零开始部署整套体系的引导流程 | **三期** |

**一期严格限制**：不做工厂，只做"当前一套团队的清晰化和标准化"。

## 三、底层必补能力（技术底座）

优先级说明：P0=必须在一期完成，P1=二期，P2=三期

### P0 — 必须先做

| 能力 | 为什么必须先做 |
|------|--------------|
| **文件契约/Schema显式化** | 当前 handoff/brief/review 格式靠约定，不靠规范。没有 schema，多角色并发写文件会产生不兼容的格式，导致消费方读不懂。这是文件驱动架构的基础设施。 |
| **对象注册表** | 每个项目对象（任务/决策/交付物）需要有唯一ID和归属，才能在控制台上可视化追踪。没有注册表，对象只是散落的文件，无法聚合成"项目状态"。 |
| **操作/事件日志（只追加）** | 今天的bug（??语法错误导致全局崩溃）说明：没有操作日志，排查代价极高。事件日志是可观测性的最低要求，也是"系统是否在按预期运行"的唯一客观依据。 |

### P1 — 二期补齐

| 能力 | 为什么是P1 |
|------|-----------|
| **项目状态机** | 项目阶段（启动/执行/审查/收口/归档）需要有显式的状态转移规则，防止"永远在执行"或"跳过验收"。P0先做完注册表再做状态机，否则状态机无处挂载。 |
| **并发写入保护** | 多角色并发写同一文件时会产生竞争。一期内团队规模小，靠串行调度规避。二期引入更多并发后必须有文件锁或版本冲突检测。 |
| **版本与告警阈值配置化** | 当前告警阈值（上下文压力≥80%/空闲>2h）硬编码在代码里。二期需要让这些配置外显，可以从控制台调整。 |
| **冷启动索引** | 新会话开始时，角色需要快速重建项目上下文。当前依赖手动读文件。二期需要一个"上次状态摘要"机制，让角色10秒内就知道"现在该干什么"。 |

### P2 — 三期完善

| 能力 | 说明 |
|------|------|
| **适配层边界显式化** | 当前 office-dashboard-adapter 的"适配"范围靠约定，三期需要明确什么是适配层职责，什么要穿透到 OpenClaw 核心。 |
| **DTO与办公友好接口稳定性** | 前端字段映射今天出了多个bug（totalToken/contextUsedEstimate等）。三期建立前后端接口契约文档，接口变更走 changelog，不靠注释。 |

## 《下一步最应该先做的 3 件事》

### 第1件：文件契约 Schema 文档（艾克 + 贾克斯，3天内）

**为什么排第一**：当前最大的风险是"文件格式靠约定，没有验证"。角色交接时写出的文件格式不一致，消费方读不到期望字段——这比代码bug更难发现，影响时间更长。

**产出**：一份 `schemas/` 目录下的 JSON Schema 或 Markdown 规范，覆盖 handoff/brief/review/artifact 四类核心对象。

---

### 第2件：产品主线能力边界文档（艾克，3天内）

**为什么排第二**：当前"运行控制台"已经有大量功能，但哪些是核心、哪些是加分项、哪些是未来不做——没有一份清单。不冻结边界，功能会持续蔓延，一期永远做不完。

**产出**：一份 `docs/product-scope-v1.md`，明确一期/二期/三期能力清单，并写明"一期不做什么"。

---

### 第3件：内外双名映射正式文件（提莫，1天内）

**为什么排第三**：这是最低成本、最快落地的一件事，但它影响后续所有对外材料的表达一致性。现在就做，后面不用反复解释。

**产出**：一份 `docs/naming-map.md`，包含内部代号↔对外名称↔职能说明，附上在飞书群/控制台/文档中的使用规则。

## 附录：项目目录约定（2026-03-20 补充）

### 代码实现根目录

所有后端/前端代码实现落在：

```
artifacts/office-dashboard-adapter/
├── src/
│   ├── services/        ← 后端服务（event-log-service.ts、registry-service.ts 等）
│   ├── routes/          ← API 路由
│   ├── controllers/     ← 控制器
│   └── public/          ← 前端 app.js / style.css / index.html
└── data/                ← 运行时数据（events.ndjson、timeline.log、cache-snapshots.json）
```

### 项目文档 / 规范 / 注册表根目录

所有设计文档、规范、注册表落在：

```
/root/.openclaw/workspace/projects/office-console-enhanced/
├── docs/        ← 设计文档
├── reviews/     ← 验收报告
├── registry/    ← 对象注册表（objects.md）
├── status/      ← 项目状态
└── tasks/       ← 任务文件
```

**结论**：验收时路径应指向 `artifacts/office-dashboard-adapter/src/...`，不是项目根 `src/...`。这是约定，后续所有任务文件和验收标准统一使用此路径。
