# 「项目过程沉淀 skill」方案投票 — Ekko（产品视角）

**日期：** 2026-03-20  
**投票人：** Ekko（Product）  
**投票结论：** ✅ 有条件赞成

---

## 一、对推广目标的价值评估

**价值：高（8/10）**

当前多角色协作体系的最大复制门槛不是代码或配置，而是「过程知识的断裂」——新用户搭建系统后，无法继承项目演进逻辑，每次都从零开始摸索。将过程沉淀封装为 skill，直接降低了复制成本，从"手动抄目录结构"变为"装一个 skill 即用"。这是从定制品到产品的关键一步。

## 二、用户最需要的沉淀能力（优先级排序）

1. **会话上下文摘要自动写入** — 这是最高频痛点，解决"昨天聊了什么"的断裂问题
2. **跨角色共享文件索引** — 多 agent 协作的前提，没有它角色间无法衔接
3. **每日日志持久化** — 审计与回溯用，优先级次于前两项
4. **项目里程碑快照** — 低频但关键，可作为 v2 功能

## 三、核心价值主张与命名建议

**核心价值主张：** "让多 agent 项目的每一步决策都有迹可循，每一次协作都有上下文可继承。"

**命名建议：**
- 推荐名：`project-memory`（简洁、通用、易理解）
- 备选：`session-ledger` / `context-trail`
- 避免：`过程沉淀` 太抽象，非中文用户无法理解

**对外描述（一句话）：**  
> Automatic context persistence for multi-agent projects — session summaries, shared file index, and daily logs, out of the box.

## 四、投票理由

**有条件赞成。** 条件如下：

1. **MVP 必须极简** — 第一版只做「会话摘要自动写入」+「共享文件索引」，不要贪全
2. **不依赖特定目录约定** — skill 应支持用户自定义存储路径，否则复用性打折
3. **需要有清晰的 README 和 onboarding** — 目标用户是"想复制这套系统的人"，不是我们自己；如果装完还需要看源码才能用，那就不是产品

**风险提示：**
- 当前沉淀逻辑和本项目深度耦合，抽象成通用 skill 可能需要重构，工作量不可忽视
- 如果 skill 做得太重（日志+摘要+索引+快照），会变成一个"框架"而非"skill"，用户学习成本反而增加

---

**结论：有条件赞成，建议先出 MVP 验证价值，再逐步扩展。**
