# OpenClaw Office Console（Windows 版）推荐办公 Agent 输入输出定义
> 基于 MVP 架构稿继续细化的首批办公 Agent 设计文档

---

## 1. 文档目的

本文档用于定义 **OpenClaw Office Console（Windows 版）** 第一版推荐办公 Agent 的产品侧规格，重点明确：

- 每个 Agent 解决什么问题
- 每个 Agent 面向谁
- 每个 Agent 需要什么前置条件
- 每个 Agent 的输入是什么
- 每个 Agent 的输出是什么
- 运行过程如何体现
- 失败时如何处理

本文件主要服务于：

- 产品功能定义
- 首页与 Agent 中心页面设计
- Console Agent 推荐逻辑
- 后续 Agent 执行接口设计
- MVP 范围取舍

---

## 2. Agent 设计原则

### 2.1 第一版只做少量高价值 Agent
第一版不追求 Agent 多，而追求：

- 作用清晰
- 输入简单
- 输出可见
- 跑通闭环
- 容易理解

### 2.2 不强调“多 Agent 技术概念”
从用户视角，Agent 不应被解释为复杂技术系统，而应被理解成：

- 飞书知识助手
- 文档初稿助手
- 浏览器操作助手
- 会议纪要助手

### 2.3 所有 Agent 必须具备 5 个要素
1. 明确用途
2. 明确前置条件
3. 明确输入
4. 明确输出
5. 明确失败处理

---

## 3. 第一版推荐 Agent 总览

第一版建议先聚焦以下 4 个 Agent：

1. Console Agent（默认中枢 Agent）
2. 飞书知识助手
3. 文档初稿助手
4. 浏览器操作助手

会议纪要助手可作为 P1 或 P0.5 预留。

---

# 4. Console Agent

## 4.1 定位
Console Agent 是控制台中枢，不是普通业务 Agent。

它负责：

- 汇总系统状态
- 识别当前问题
- 给出推荐动作
- 提示更新与风险

## 4.2 面向用户
所有用户，尤其是：

- 新用户
- 尚未完成配置的用户
- 遇到错误的用户
- 不知道下一步做什么的用户

## 4.3 前置条件
无需额外业务前置条件，但依赖系统状态数据可用。

### 依赖
- 环境检测结果
- 配置状态
- Agent 状态
- 任务状态
- 诊断摘要

## 4.4 输入定义
Console Agent 不接受自由业务输入，其输入来自控制台内部状态：

- 模型配置状态
- 飞书配置状态
- 浏览器状态
- 工作空间状态
- Agent 健康状态
- 任务摘要
- 错误摘要
- 更新事件

## 4.5 输出定义

### 首页输出结构
- 当前摘要
- 发现的问题
- 推荐下一步
- 最近更新

### 输出结果形态
- 首页建议卡片
- 全局提醒
- 跳转按钮

## 4.6 运行状态
### 状态类型
- `ready`
- `degraded`
- `blocked`

### 含义
- ready：可正常生成建议
- degraded：部分输入缺失，但仍可生成基础建议
- blocked：关键状态数据不可用

## 4.7 失败处理
若 Console Agent 无法完整生成建议，应降级为：

- 固定规则提醒
- 基础配置检查清单
- 默认帮助入口

# 5. 飞书知识助手

## 5.1 定位
飞书知识助手用于把输入内容整理成适合飞书沉淀和协作的结构化文档，并写入飞书知识空间。

## 5.2 适用场景
- 整理教程
- 整理部署说明
- 整理工作总结
- 将本地内容同步到飞书知识库
- 建立团队可复用资料

## 5.3 面向用户
- 用飞书办公的个人用户
- 小团队知识沉淀需求
- 想把 OpenClaw 结果沉淀到飞书的人

## 5.4 前置条件

### 必需
- 模型已配置并测试通过
- 飞书已完成授权
- 飞书写入测试已通过

### 建议
- 已设置默认飞书目标位置
- 已安装飞书模板（如有）

## 5.5 输入定义

### 最小输入
- `title`：标题
- `content_source`：内容来源
- `output_style`：输出风格（教程 / 说明 / 知识卡片）

### 典型输入形式
#### 方式 1：纯文本输入
```json
{
  "title": "智慧园区平台部署说明",
  "content_source": "这里是一段原始说明文本",
  "output_style": "教程"
}
```

#### 方式 2：文件输入
```json
{
  "title": "智慧园区平台部署说明",
  "content_source": "file://C:/docs/deploy.md",
  "output_style": "教程"
}
```

#### 方式 3：任务结果转存
```json
{
  "title": "部署教程",
  "content_source": "task://task_001_result",
  "output_style": "教程"
}
```

## 5.6 输出定义

### 主输出
- 飞书文档页面
- 飞书目标链接
- 写入结果摘要

### 辅助输出
- 本地 Markdown 副本（可选）
- 写入日志
- 结构化大纲

### 输出字段建议
- `doc_title`
- `doc_url`
- `target_space`
- `write_status`
- `summary`

## 5.7 运行步骤建议
1. 读取输入内容
2. 提炼结构
3. 按飞书适配格式整理内容
4. 写入飞书目标位置
5. 返回写入结果

## 5.8 状态定义
- `available`
- `needs_config`
- `running`
- `failed`
- `completed`

## 5.9 失败处理
常见失败原因：
- 飞书授权失效
- 写入目标不可用
- 模型生成失败
- 内容结构化失败

失败后建议：
- 提示错误摘要
- 给出“重新授权 / 测试写入 / 重试任务”动作
- 尽量保留已生成的中间结果

# 6. 文档初稿助手

## 6.1 定位
文档初稿助手用于根据主题或输入资料，快速生成一份结构化文档初稿。

它不是“专业文档工厂”，而是：

> 第一版里最轻量、最容易理解的文档生成 Agent。

## 6.2 适用场景
- 方案初稿
- 说明文档
- 部署教程
- 简单需求说明
- 培训材料提纲与正文初稿

## 6.3 面向用户
- 需要写文档但不想从零开始的人
- 想先拿到一个 60~80 分结构化底稿的人
- 还不需要完整长文档流水线的人

## 6.4 前置条件
### 必需
- 模型已配置并测试通过

### 可选
- 已配置默认模板
- 已配置输出目录

## 6.5 输入定义

### 最小输入
- `topic`：文档主题

### 扩展输入
- `doc_type`：文档类型
- `background`：背景说明
- `requirements`：关键要求
- `output_format`：输出格式

### 最小输入示例
```json
{
  "topic": "智慧园区平台部署教程"
}
```

### 扩展输入示例
```json
{
  "topic": "智慧园区平台部署教程",
  "doc_type": "tutorial",
  "background": "面向内部实施人员",
  "requirements": [
    "包含环境准备",
    "包含部署步骤",
    "包含常见问题"
  ],
  "output_format": "markdown"
}
```

## 6.6 输出定义

### 主输出
- Markdown 初稿
- 标题结构
- 文档摘要

### 辅助输出
- 输出文件路径
- 生成日志
- 可选 DOCX（P1 以后）

### 输出字段建议
- `title`
- `outline`
- `content_path`
- `summary`
- `status`

## 6.7 运行步骤建议
1. 解析输入主题
2. 生成结构提纲
3. 生成正文初稿
4. 保存本地文件
5. 返回输出结果

## 6.8 状态定义
- `available`
- `running`
- `failed`
- `completed`

## 6.9 失败处理
常见失败原因：
- 模型未配置
- 模型连接失败
- 输出目录不可写
- 生成中断

失败后建议：
- 重试
- 重新测试模型
- 更换输出目录
- 查看中间结果

# 7. 浏览器操作助手

## 7.1 定位
浏览器操作助手用于帮助用户执行标准化网页操作流程。

第一版不应把它定义成“万能浏览器自动化”，而应是：

> 一个可见、可监控、可诊断的网页操作助手。

## 7.2 适用场景
- 打开指定网页并完成固定步骤
- 登录后的固定流程执行
- 简单网页内容抓取与整理
- 重复性的网页办公动作

## 7.3 面向用户
- 需要做重复网页操作的人
- 想把部分网页流程交给 Agent 执行的人

## 7.4 前置条件

### 必需
- 模型已配置
- 浏览器连接能力正常

### 可选
- 用户已完成必要登录
- 已配置默认浏览器模式

## 7.5 输入定义

### 最小输入
- `task_goal`：要做什么

### 扩展输入
- `target_url`
- `constraints`
- `expected_result`

### 最小输入示例
```json
{
  "task_goal": "打开公司知识库，搜索智慧园区部署文档，并整理结果"
}
```

### 扩展输入示例
```json
{
  "task_goal": "登录管理后台并查看最近告警",
  "target_url": "https://example.com",
  "constraints": [
    "不执行不可逆操作",
    "如需登录则停在登录页提示用户"
  ],
  "expected_result": "返回告警摘要"
}
```

## 7.6 输出定义

### 主输出
- 任务执行摘要
- 浏览器操作结果
- 抓取结果 / 页面摘要

### 辅助输出
- 关键步骤列表
- 截图（可选）
- 错误摘要

### 输出字段建议
- `result_summary`
- `visited_pages`
- `captured_items`
- `status`

## 7.7 运行步骤建议
1. 打开目标页面
2. 判断页面状态
3. 执行固定步骤
4. 提取结果
5. 返回摘要

## 7.8 状态定义
- `needs_browser`
- `available`
- `running`
- `failed`
- `completed`

## 7.9 失败处理
常见失败原因：
- 浏览器能力不可用
- 页面无法访问
- 登录态缺失
- 页面结构变化
- 用户限制动作导致中断

失败后建议：
- 重新检测浏览器
- 提示用户补充登录
- 允许重试
- 输出已执行到哪一步

# 8. 会议纪要助手（P1 预留）

## 8.1 定位
会议纪要助手用于把会议记录、聊天内容或输入笔记整理成结构化纪要。

## 8.2 适用场景
- 整理会议纪要
- 提取责任项
- 提取待确认问题
- 提取后续行动项

## 8.3 前置条件
- 模型已连接

## 8.4 输入定义
- `meeting_input`
- `participants`（可选）
- `output_style`（简版 / 正式版）

## 8.5 输出定义
- 会议背景
- 核心讨论点
- 决议事项
- 待办事项
- 责任人（如可识别）

## 9. Agent 结果对象统一建议

为便于前端展示，所有业务 Agent 建议返回统一结果对象：

```json
{
  "agent_id": "doc_draft",
  "task_id": "task_001",
  "status": "completed",
  "title": "智慧园区平台部署教程",
  "summary": "已生成一份 Markdown 初稿",
  "primary_output": {
    "type": "file",
    "value": "C:/OpenClaw/workspace/outputs/tutorial.md"
  },
  "secondary_outputs": [],
  "created_at": "2026-03-09T10:30:00"
}
```

## 10. Agent 状态统一建议

所有 Agent 建议统一使用以下状态模型：

### 配置类状态
- `needs_config`
- `disabled`

### 运行类状态
- `available`
- `running`
- `failed`
- `completed`

### 特殊类状态
- `abnormal`
- `blocked`

## 11. 首页推荐逻辑与 Agent 的关系

### 11.1 新用户推荐顺序
1. Console Agent
2. 文档初稿助手
3. 飞书知识助手
4. 浏览器操作助手

原因：
- 文档初稿助手最容易建立“第一次成功体验”
- 飞书知识助手在完成配置后价值明显
- 浏览器助手更适合已有一定理解的用户

### 11.2 老用户推荐顺序
根据：
- 最近失败任务
- 已配置渠道
- 已安装 Agent
- 最近结果类型

进行动态推荐。

## 12. 第一版 MVP 中每个 Agent 的价值判断

### 12.1 Console Agent
必须有。  
因为它是全局中枢。

### 12.2 文档初稿助手
建议第一版必须有。  
因为它是最容易建立“可见成果”的 Agent。

### 12.3 飞书知识助手
建议第一版必须有。  
因为它是办公场景里最明确的落地能力之一。

### 12.4 浏览器操作助手
建议第一版可做，但边界要收紧。  
不要承诺“什么网页都能自动化”，而是先做受控、可诊断的基础版本。

### 12.5 会议纪要助手
可作为 P1 预留。  
有价值，但优先级略低于前三者。

---

## 13. 总结

第一版推荐办公 Agent 的核心目标不是“展示技术有多强”，而是建立三个用户认知：

1. OpenClaw 不只是一个聊天框
2. 它可以通过 Agent 直接做办公动作
3. 每个 Agent 都有明确用途、输入、输出和结果

因此第一版 Agent 设计必须坚持以下原则：

- 少而清晰
- 输出可见
- 状态透明
- 失败可处理
- 前置条件明确
