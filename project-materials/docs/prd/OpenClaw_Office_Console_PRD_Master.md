# 2. 产品概述

## 2.1 产品名称

- 中文名：**OpenClaw 办公增强控制台**
- 英文名：**OpenClaw Office Console**
- 仓库建议名：`openclaw-office-console`

## 2.2 产品一句话定义

> 一个面向办公用户的 Windows 本地可视化控制台，用于帮助用户完成 OpenClaw 的配置接入、办公 Agent 管理、任务执行可视化、错误反馈与办公能力落地。

## 2.3 产品形态

第一版产品形态定义为：

**Windows 桌面安装包（exe） + 本地控制台 UI + 本地服务层 + OpenClaw 兼容适配层**

用户感知：

- 下载并安装一个桌面应用
- 双击启动，像普通办公软件一样使用
- 在应用中完成配置、安装 Agent、发起任务、查看结果和错误

内部实现：

- 桌面壳中承载本地 Web 控制台
- 控制台通过本地服务层调用 OpenClaw 兼容接口

# 3. 产品背景

## 3.1 背景问题

当前 OpenClaw 虽然具备较强能力，但大量用户在实际使用前期普遍存在以下问题：

- 不知道如何配置模型
- 不知道如何接入飞书
- 不知道模型、Agent、Skill、浏览器之间是什么关系
- 不知道接入之后可以做什么
- 不知道任务现在执行到哪一步
- 出错后不知道如何判断问题类型和处理方式
- 对“安装好后怎么用”缺少明确路径

这导致大量用户即便已经安装 OpenClaw，也很难进入“真正可用”的状态。

## 3.2 核心矛盾

用户当前真正缺少的，不只是功能，而是一个：

> **面向办公用户的控制层与解释层**

OpenClaw 原生更偏底层能力平台，而目标用户大量是办公型、轻技术型、非开发者型用户。这些用户需要的不是更强的底层系统，而是：

- 看得懂的状态
- 配得上的入口
- 跑得通的流程
- 错了能处理的反馈

## 3.3 为什么现在做这个产品

从需求判断看，当前阶段做这个产品成立的原因在于：

1. OpenClaw 的能力已具备足够可用基础
2. 但用户上手与办公落地层明显缺失
3. 大厂或官方更可能做通用启动与接入层
4. 办公场景落地、飞书承载、Agent 过程透明、错误反馈等层面仍存在空间

因此本产品不应与“通用一键部署器”正面竞争，而应定位为：

> **OpenClaw 的办公增强控制台**

# 4. 产品目标

## 4.1 总目标

帮助普通办公用户在 Windows 本地环境下，更容易地完成 OpenClaw 的：

- 基础配置
- Agent 启用
- 办公任务执行
- 状态查看
- 错误处理
- 初步价值体验

## 4.2 第一版目标

第一版只聚焦 5 件事：

1. **让用户知道当前系统状态**
2. **让用户完成模型与飞书等关键配置**
3. **让用户理解有哪些办公 Agent 可用**
4. **让用户看到任务过程与结果**
5. **让用户看懂错误并知道下一步动作**

## 4.3 非目标

第一版不以以下目标为核心：

- 不追求全平台覆盖
- 不追求通用一键部署全场景
- 不追求多 Agent 炫技式可视化
- 不追求完整长文档生产平台
- 不追求复杂自动修复系统
- 不追求覆盖所有渠道接入

# 5. 用户与场景

## 5.1 目标用户画像

### 用户类型 A：轻技术办公用户
特征：
- 使用 Windows 办公电脑
- 会使用桌面软件，不擅长命令行
- 希望把 OpenClaw 用于日常办公、整理资料、写初稿、同步飞书
- 对错误诊断和配置缺乏经验

### 用户类型 B：有一定技术能力的办公型用户
特征：
- 能理解模型、配置和基础目录概念
- 已经装过 OpenClaw，但缺少更顺手的控制台
- 希望快速完成模型/飞书接入、查看任务状态、安装办公 Agent

### 用户类型 C：希望探索办公自动化的人
特征：
- 关心飞书知识库、教程助手、文档助手、浏览器自动化
- 需要一个容易理解的入口去理解“OpenClaw 到底能做什么”

## 5.2 核心使用场景

### 场景 1：首次安装和关联已有 OpenClaw
用户希望快速完成：
- 环境检查
- 模型配置
- 飞书配置
- 安装推荐办公 Agent

### 场景 2：试用第一个办公 Agent
用户第一次希望尝试：
- 文档初稿生成
- 飞书知识写入
- 浏览器执行基础任务

### 场景 3：查看任务进度
用户发起任务后，希望知道：
- 当前做到哪一步
- 哪一步失败
- 能否重试

### 场景 4：遇到错误后的处理
用户遇到问题时，希望看到：
- 问题类型
- 影响范围
- 推荐动作
- 可跳转处理入口

### 场景 5：将 OpenClaw 变成办公工具
用户希望通过控制台逐步建立对 OpenClaw 的“办公化理解”，而不只是聊天。

# 6. 产品定位与边界

## 6.1 产品定位

本产品定位不是：

- 安装器
- 聊天助手
- 技术调试台
- 通用自动化平台

而是：

> **办公增强控制台**

它是一个面向办公用户的可视化控制台，作用是把 OpenClaw 的底层能力翻译成用户能理解、能配置、能运行、能排障的办公化入口。

## 6.2 产品边界

### 做什么
- 配置可视化
- 状态可视化
- Agent 管理
- 任务可视化
- 错误分类与建议
- 办公能力入口

### 不做什么
- 不重做 OpenClaw 全部底层逻辑
- 不替代原生后台全部能力
- 不承担所有高级调试需求
- 不把第一版做成完整办公平台
- 不做复杂云端中台

---

# 7. 核心产品原则

## 原则 1：Windows 优先
第一版优先验证 Windows 办公用户是否真的能用起来。

## 原则 2：对外是桌面应用，对内是本地控制台
兼顾产品感与工程效率。

## 原则 3：先做控制台，不先做最重能力
先解决“上手、理解、过程透明”，再解决“专业深度”。

## 原则 4：先做少量高价值 Agent
避免能力太多导致认知负担。

## 原则 5：状态与错误反馈是基础设施
没有这层，整个产品不会被认为可靠。

## 原则 6：Console Agent 作为全局中枢
负责解释、建议、提醒、更新通知。

## 原则 7：兼容原生后台，但不直接用原后台做主界面
角色分工清晰。

## 原则 8：Skill 是能力内核，控制台是产品壳
要长期坚持这个分层。

# 9. 关键角色：Console Agent

## 9.1 定位

Console Agent 是控制台默认中枢 Agent，负责：

- 汇总当前系统摘要
- 发现关键问题
- 推荐下一步
- 输出更新通知

## 9.2 作用

如果没有 Console Agent：
- 全局建议会散落在多个页面
- 用户很难形成“下一步该做什么”的明确感知

因此它承担的是：

> **全局解释层**

## 9.3 输入来源

- 环境检测结果
- 配置状态
- Agent 状态
- 任务摘要
- 诊断错误摘要
- 更新事件

## 9.4 输出结构

固定输出 4 个部分：

1. 当前摘要
2. 发现的问题
3. 推荐下一步
4. 最近更新

## 9.5 不是聊天机器人

Console Agent 不是大而全聊天机器人，不承担：
- 业务执行
- 任意自由问答
- 替代所有其他 Agent

# 10. 第一版推荐办公 Agent

第一版建议聚焦以下 4 个：

## 10.1 Console Agent
控制台中枢解释层，默认启用。

## 10.2 文档初稿助手
根据主题快速生成结构化文档初稿。

### 适用场景
- 方案初稿
- 说明文档
- 部署教程
- 简单需求说明

### 为什么优先
它最容易帮助用户快速看到“可见成果”。

## 10.3 飞书知识助手
将内容整理并写入飞书知识空间。

### 适用场景
- 教程沉淀
- 文档写入飞书
- 团队知识复用

### 为什么优先
飞书是办公场景里最直接的承载层之一。

## 10.4 浏览器操作助手
执行基础网页操作并返回结果摘要。

### 第一版边界
- 不承诺万能自动化
- 先做受控、可监控、可诊断版本

# 11. 需求拆解

## 11.1 启动与首启相关需求

### 需求 1：支持新安装与关联已有实例
用户首次打开应用时，系统应能：
- 检测是否存在已有 OpenClaw 实例
- 允许用户选择“新安装”或“关联已有实例”

### 需求 2：提供欢迎页与首启引导
应在欢迎页中明确说明：
- 这是一个办公增强控制台
- 用户接下来可以完成哪些关键操作

---

## 11.2 环境检测相关需求

### 需求 3：支持基础环境检测
应检测：
- 操作系统版本
- OpenClaw 环境
- 工作目录可用性
- 端口占用
- 浏览器能力
- 基础网络连通性

### 需求 4：检测项必须有统一状态
状态分为：
- 通过
- 警告
- 失败
- 未检测

### 需求 5：每个检测项必须提供建议动作
例如：
- 去配置
- 重新检测
- 更换目录
- 查看详情

---

## 11.3 配置相关需求

### 需求 6：支持模型配置
用户应可完成：
- Provider 选择
- API Key 配置
- Base URL 配置
- 模型名称配置
- 连接测试

### 需求 7：模型状态必须在首页显眼展示
模型是否可用应成为最关键的状态项之一。

### 需求 8：支持飞书配置
用户应可完成：
- App ID / App Secret 配置
- 授权测试
- 写入测试

### 需求 9：飞书配置结果必须可验证
不是只保存配置，而要：
- 测试授权
- 测试写入
- 给出明确结果

### 需求 10：支持工作目录与基础设置管理
用户应可配置：
- 工作目录
- 输出目录
- 日志目录
- 默认模板

---

## 11.4 首页总览相关需求

### 需求 11：首页必须展示系统状态摘要
展示：
- 模型状态
- 飞书状态
- 浏览器状态
- Workspace 状态
- Agent 健康状态
- 任务摘要

### 需求 12：首页必须展示 Console Agent 建议区
建议区需输出：
- 当前摘要
- 发现的问题
- 推荐下一步
- 最近更新

### 需求 13：首页必须展示推荐 Agent 区
至少展示首批核心 Agent：
- 文档初稿助手
- 飞书知识助手
- 浏览器操作助手

### 需求 14：首页必须展示当前任务
如果有任务运行，需展示：
- 当前任务名
- 所属 Agent
- 当前阶段
- 已完成步骤
- 查看详情入口

### 需求 15：首页需展示最近结果和更新动态
帮助用户理解：
- 最近生成了什么
- 系统最近有什么新变化

---

## 11.5 Agent 中心相关需求

### 需求 16：支持查看 Agent 列表
每个 Agent 需展示：
- 名称
- 描述
- 输入输出说明
- 当前状态
- 最近结果
- 快捷操作

### 需求 17：支持 Agent 启用 / 禁用
允许用户控制 Agent 是否启用。

### 需求 18：支持从 Agent 中心直接发起任务
对于可运行的 Agent，应支持“立即运行”或“新建任务”。

---

## 11.6 任务中心相关需求

### 需求 19：支持任务列表查看
任务列表需展示：
- 任务名称
- 来源 Agent
- 当前状态
- 当前步骤
- 创建时间
- 最近动作摘要

### 需求 20：支持任务详情页
任务详情页需展示：
- 输入摘要
- 执行步骤
- 当前阶段
- 日志摘要
- 错误信息
- 结果入口

### 需求 21：支持任务重试
对于失败任务，应支持重试。

### 需求 22：支持“需人工处理”状态
适合办公场景中存在人工补充环节的任务。

---

## 11.7 日志与诊断需求

### 需求 23：支持错误分类
第一版建议分类：
- 系统错误
- OpenClaw 错误
- 运行时错误
- 模型错误
- 飞书错误
- 浏览器错误
- 任务错误

### 需求 24：每个错误必须有影响说明与建议动作
错误不能只展示技术信息，必须有：
- 错误标题
- 问题摘要
- 影响模块
- 推荐动作

### 需求 25：支持重新检测与关联处理入口
错误出现后，应能：
- 重新检测
- 跳转到对应配置页
- 重试任务

---

## 11.8 模板与能力包需求

### 需求 26：支持查看已安装 Skill / 能力包
应展示：
- 能力名称
- 类型
- 是否已安装
- 是否启用
- 版本

### 需求 27：支持安装推荐办公 Skill
第一版建议可安装：
- 飞书知识助手
- 文档初稿助手
- 浏览器操作助手

# 14. 关键业务流程

## 14.1 首次使用主流程

```text
安装应用
→ 启动应用
→ 检测是否已有 OpenClaw
→ 选择新安装 / 关联实例
→ 环境检测
→ 模型配置
→ 飞书配置（可跳过）
→ 安装推荐办公 Agent
→ 进入首页总览
```

## 14.2 第一次成功体验流程

```text
进入首页
→ 查看 Console Agent 推荐
→ 试用文档初稿助手
→ 任务开始执行
→ 在任务中心查看过程
→ 输出 Markdown 初稿
→ 可选：转存到飞书
```

## 14.3 出错处理流程

```text
任务失败
→ 首页 / 任务中心提示失败
→ 查看任务详情或日志与诊断页
→ 阅读错误摘要与建议动作
→ 重新检测 / 重试任务 / 去配置页修复
```

# 18. 成功标准

第一版成功不看“功能多不多”，而看以下 5 点是否成立：

1. 新用户能完成模型配置
2. 用户能理解至少 2~3 个办公 Agent 的用途
3. 用户能成功跑通一次任务
4. 用户能看见任务过程，不再黑盒
5. 用户出错后能知道下一步该做什么

如果这 5 点做不到，即使后面接更多能力，产品也不会真正成立。

# 19. 风险与取舍

## 19.1 风险 1：产品做重
如果第一版同时做：
- 多平台
- 长文档工厂
- 多渠道接入
- 复杂多 Agent 编排

大概率会失焦。

### 取舍原则
第一版必须克制，只保留控制台最核心闭环。

## 19.2 风险 2：过度技术化
如果前台大量使用：
- 多 Agent
- Planner
- Executor
- Toolchain
等概念，会劝退办公用户。

### 取舍原则
前台应使用业务化语言和动作化语言。

## 19.3 风险 3：状态不透明
如果任务过程仍然黑盒，产品会被认为“不稳定、不可控”。

### 取舍原则
状态和错误反馈必须优先实现。

## 19.4 风险 4：与 OpenClaw 原系统割裂
如果另起一套底层体系，维护成本和兼容成本都会变高。

### 取舍原则
坚持适配层思路，尽量复用原系统。

# 21. 总结

OpenClaw Office Console 第一版的核心，不是“做一个更强的 OpenClaw”，而是：

> **做一个普通办公用户终于能理解、能配置、能运行、能排障、能获得初步价值的 OpenClaw 控制台。**

因此，第一版必须坚持以下判断：

- 优先 Windows，不做多端发散
- 对外是桌面产品，对内是本地控制台
- 先做控制台，不先做最重能力
- 先做少量高价值 Agent
- 状态和错误反馈是基础设施
- Console Agent 是全局中枢
- Skill 是能力内核，控制台是产品壳

只要第一版能把“配置、状态、Agent、任务、错误”这五件事做清楚，后面的飞书知识沉淀、文档工厂、项目材料增强都可以自然接进来。
