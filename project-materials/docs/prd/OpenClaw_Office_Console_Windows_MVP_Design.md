# OpenClaw Office Console（Windows 版）MVP 功能清单 + 技术架构草案
> 基于前序讨论整理的完整设计稿（聚焦 Windows 第一版）

---

## 1. 文档目的

本文档用于系统性整理 **OpenClaw Office Console** 的产品方向、边界、功能、技术架构与落地建议，目标是把此前零散讨论收敛成一份可用于后续 PRD、原型设计、项目立项、GitHub 仓库规划和开发拆解的基础文档。

本文档同时回顾并吸收前面讨论中有价值的判断，也明确指出哪些方向应暂缓、哪些方向应收敛，避免第一版产品做重、做散、做偏。

---

## 2. 项目名称与定位

### 2.1 项目名称

- 中文名：**OpenClaw 办公增强控制台**
- 英文名：**OpenClaw Office Console**
- GitHub 仓库名建议：`openclaw-office-console`

### 2.2 项目定位

这是一个基于 OpenClaw 的 **办公场景可视化控制台**，不是纯安装器，也不是单个聊天助手，而是一个面向办公用户的：

- 配置引导入口
- Agent 管理入口
- 任务执行与状态反馈入口
- 办公能力包入口
- 故障解释与修复引导入口

### 2.3 它不是什么

第一版它**不是**：

- 通用 OpenClaw 一键部署器
- 云平台替代品
- 大而全的多 Agent 编排平台
- 直接面向所有平台的全端产品
- 一上来就主打几百页长文档工厂的专业平台

---

## 3. 为什么要做这个产品

### 3.1 来自真实用户的核心问题

根据前序讨论，当前许多 OpenClaw 用户并不是卡在“功能不够强”，而是卡在以下问题：

- 不知道怎么配置模型
- 不知道怎么接飞书
- 不知道接入后能做什么
- 不理解 Agent、Skill、渠道接入之间的关系
- 不知道当前任务执行到了哪一步
- 不知道为什么报错
- 不知道该如何修复
- 不知道安装好之后怎么“养”它、怎么实际办公使用

### 3.2 痛点本质

用户缺的不是单一功能，而是一个 **能看懂、能配置、能运行、能反馈、能引导下一步** 的控制层。

换句话说，OpenClaw 原生能力很强，但很多人缺少一个：

> 把复杂能力翻译成办公用户能理解的操作系统界面。

### 3.3 为什么不是先做“文档工厂”

我们讨论后形成的客观判断是：

长文档工厂、自动排版、专业文档流水线都很有价值，但它们更适合成为 **高级能力**，而不是第一版产品的主入口。

原因：

- 新用户尚未跨过基础配置门槛
- 大多数用户还不理解接入后的价值路径
- 一上来展示太重的能力，会造成“看起来很强，但不知道怎么开始”
- 第一版更需要“低门槛上手”和“过程透明”

因此第一版主轴应是：

> **可视化办公控制台**，而不是一开始就主打“最强能力”。

## 4. 核心产品判断与结论

### 4.1 优先平台结论

**第一版优先做 Windows。**

不建议一开始同时做：

- Windows
- macOS
- Linux

原因：

1. 目标用户以办公用户为主，Windows 覆盖面最大
2. Windows 更适合验证非技术用户是否真的能顺利用起来
3. 多平台会显著增加安装、权限、路径、依赖、浏览器控制等复杂度
4. 第一版更需要闭环而不是平台覆盖

### 4.2 最终成品形态结论

**对用户交付为 Windows 桌面安装包（exe）。**  
**内部实现采用“桌面壳 + 本地网页控制台”的结构。**

也就是：

- 用户下载并安装一个桌面应用
- 启动后看到的是桌面窗口
- 桌面窗口内运行的是本地前端控制台
- 前端通过本地服务层与 OpenClaw 后端、配置文件、Skill 安装逻辑交互

### 4.3 为什么不是纯本地网页

如果只做 localhost 网页：

- 产品感太弱
- 用户容易迷糊“后台有没有跑起来”
- 本地网页与 OpenClaw 原有页面容易混淆
- 不够像面向普通办公用户的产品

### 4.4 为什么不是纯原生桌面 UI

如果一开始全原生做：

- 工程量明显增大
- 后续扩展与复用不够灵活
- 不利于快速迭代控制台页面

因此最佳路线是：

> **Windows 桌面壳 + Web 控制台**

## 5. 项目应该解决什么问题

### 5.1 第一版必须解决的核心问题

1. 用户打开后能知道系统当前状态
2. 用户能知道缺哪些配置
3. 用户能快速完成基础配置
4. 用户能看到有哪些办公 Agent 可用
5. 用户能知道当前任务执行到了哪一步
6. 用户能看懂错误和建议动作
7. 用户能安装和启用办公能力包
8. 用户能关联已有 OpenClaw 实例

### 5.2 第一版不必解决的问题

1. 所有平台统一安装
2. 所有 IM 渠道统一接入
3. 超复杂的多 Agent 编排
4. 复杂可观测性平台
5. 完整长文档工厂系统
6. 纯云端托管方案
7. 面向极客和开发者的高级调试全覆盖

## 6. 总体产品形态

### 6.1 产品结构

OpenClaw Office Console 第一版建议由以下部分组成：

```text
OpenClaw Office Console
├─ Windows 桌面应用壳
├─ 本地 Web 控制台
├─ Console Agent（默认中枢 Agent）
├─ 基础配置模块
├─ Agent/Skill 管理模块
├─ 任务与状态模块
├─ 日志与诊断模块
└─ OpenClaw 兼容接口层
```

### 6.2 它和 OpenClaw 原本后台的关系

这个产品不应替代原后台，而应作为上层控制台存在。

#### 兼容原则

- **接口兼容**：尽量复用原有接口
- **数据兼容**：尽量复用原有配置、workspace、skills、日志
- **页面兼容**：保留入口打开原始后台，但不把原后台直接作为主界面

#### 职责分工

**Office Console 负责：**

- 小白配置引导
- 状态总览
- Agent 能力展示
- 任务与错误可视化
- 办公能力包安装

**原生后台负责：**

- 高级管理
- 调试与高级配置
- 原生底层能力管理

## 7. Console Agent 设计

### 7.1 为什么要有 Console Agent

前面讨论得出的关键结论之一是：

> 系统中需要一个“统一解释层”。

否则建议、提醒、配置缺失、更新通知会散落在多个页面，用户会乱。

### 7.2 Console Agent 的定位

Console Agent 是默认中枢 Agent，负责：

- 汇总系统健康状态
- 输出当前建议
- 检查配置缺口
- 推送更新通知
- 推荐下一步操作
- 汇总风险提醒

它**不是**一个大而全聊天机器人，也不承担复杂执行任务。

### 7.3 Console Agent 负责什么

#### 负责

- 全局配置提醒
- 推荐下一步
- 最近更新说明
- 当前系统摘要
- 全局级风险与警告

#### 不负责

- 替代所有业务 Agent
- 承担所有自由问答
- 承担复杂工具执行
- 变成一个大聊天框

### 7.4 Console Agent 的输出格式建议

建议固定为 4 个结构块：

1. 当前摘要
2. 发现的问题
3. 推荐下一步
4. 最近更新

例如：

- 当前摘要：模型已连接，浏览器正常，飞书未连接
- 发现的问题：飞书知识助手当前不可用
- 推荐下一步：先完成飞书连接并测试写入
- 最近更新：文档初稿助手新增“需求规格说明书”模板

## 9. 首页结构设计

首页是第一版最重要的页面，不能做成聊天框首页，应该做成 **状态和动作首页**。

### 9.1 首页结构

```text
A. 顶部状态摘要栏
B. Console Agent 建议区
C. Agent 快速入口区
D. 当前任务与最近任务区
E. 最近结果与更新动态区
```

### 9.2 顶部状态摘要栏

展示最关键的系统状态：

- 模型状态
- 飞书状态
- 浏览器状态
- Workspace 状态
- Agent 健康状态
- 最近任务状态

目标：用户不进子页面也知道当前系统是否能用。

### 9.3 Console Agent 建议区

展示：

- 当前系统摘要
- 当前缺失项
- 推荐下一步
- 重要更新与提醒

按钮建议：

- 去完成配置
- 体验推荐 Agent
- 查看全部建议

### 9.4 Agent 快速入口区

第一版建议放 3~4 个最核心 Agent 卡片：

- 飞书知识助手
- 文档初稿助手
- 浏览器操作助手
- 会议纪要助手（可选）

每个卡片字段：

- 名称
- 一句话作用
- 当前状态
- 前置条件
- 快捷操作

### 9.5 当前任务与最近任务区

这是“过程透明感”的关键区域。

需要展示：

- 正在运行的任务
- 当前步骤
- 已完成动作
- 待执行动作
- 最近成功/失败任务
- 查看详情 / 重试

### 9.6 最近结果与更新动态区

用于展示：

- 最近输出的文件或结果
- 最近创建的飞书文档
- 新增模板
- Agent 更新说明
- 修复与通知

## 10. Agent 中心设计

### 10.1 产品目标

让用户理解：

- 系统有哪些能力
- 每个 Agent 能干什么
- 当前是否可用
- 需要哪些前置条件
- 如何启动

### 10.2 Agent 中心筛选维度

状态筛选：

- 全部
- 可用
- 待配置
- 运行中
- 异常

类型筛选：

- 文档类
- 知识类
- 浏览器类
- 会议类

### 10.3 Agent 卡片结构

每张卡片建议包含：

- Agent 名称
- 功能描述
- 输入说明
- 输出说明
- 前置条件
- 当前状态
- 最近运行结果
- 快捷操作

### 10.4 第一版 Agent 建议清单

#### 1. Console Agent
- 作用：系统建议、配置提醒、更新通知、健康摘要
- 默认启用

#### 2. 飞书知识助手
- 作用：整理内容并写入飞书知识空间
- 适用人群：飞书办公用户

#### 3. 文档初稿助手
- 作用：快速生成结构化文档初稿
- 适用人群：需要写方案、说明、教程、材料的人

#### 4. 浏览器操作助手
- 作用：自动访问网页并完成重复步骤
- 适用人群：需要自动化网页操作的人

#### 5. 会议纪要助手（可作为 P1）
- 作用：将会议输入整理为结构化纪要
- 适用人群：经常开会、整理纪要的人

## 11. 任务中心设计

### 11.1 为什么任务中心重要

我们前面已明确：

> 运行状态和报错反馈不是可选项，是产品成立的基础设施。

没有任务中心，Agent 运行就是黑盒，用户无法建立信任。

### 11.2 任务中心应该显示什么

#### 顶部摘要
- 当前运行任务数
- 今日成功任务数
- 今日失败任务数
- 需人工处理任务数

#### 任务列表字段
- 任务名称
- 来源 Agent
- 创建时间
- 当前状态
- 当前步骤
- 最近动作
- 查看结果入口

### 11.3 任务状态统一标准

建议统一为 6 类：

- 待执行
- 运行中
- 已暂停
- 已完成
- 执行失败
- 需人工处理

其中“需人工处理”非常适合办公场景。

### 11.4 任务详情页字段

- 基本信息
- 输入参数
- 已执行步骤
- 当前阶段
- 最近日志摘要
- 错误信息
- 修复建议
- 重试入口
- 查看结果

### 11.5 状态反馈的粒度要求

第一版无需做复杂观测平台，但必须做到：

- 当前任务是什么
- 当前任务属于哪个 Agent
- 现在做到哪一步
- 最近一次错误是什么
- 建议动作是什么
- 能否重试

## 12. 配置中心设计

### 12.1 配置中心需要解决的问题

不是单纯“填参数”，而是让用户知道：

- 现在系统缺什么
- 哪些配置已通过测试
- 哪些配置有问题
- 如何重试与修改

### 12.2 第一版配置中心分区

#### 1. 模型配置
字段建议：

- Provider
- API Key
- Base URL
- Model Name
- 连接测试按钮
- 连接状态

#### 2. 飞书配置
字段建议：

- App ID
- App Secret
- Space / Folder（可选）
- 测试授权按钮
- 测试写入按钮
- 写入状态

#### 3. 浏览器能力配置
内容建议：

- 浏览器连接状态
- 最近检测时间
- 重新检测按钮
- 异常说明

#### 4. 工作空间与模板
内容建议：

- 默认输出目录
- 默认模板
- 默认日志位置
- 默认飞书写入位置

### 12.3 配置体验原则

- 默认优先配置模型
- 模型状态放在最显眼位置
- 飞书作为第一优先办公渠道
- 所有配置都要有“测试”动作
- 测试成功要尽量给出明确可见结果

例如飞书配置成功后，不仅提示成功，还应支持：

- 已验证授权
- 已写入测试文档

## 14. 日志与诊断设计

### 14.1 为什么必须做

用户最核心的问题之一不是“有没有功能”，而是：

- 为什么失败
- 失败后怎么办
- 是不是我配错了
- 要不要重试

### 14.2 第一版应做什么

#### 1. 错误归类
归类成用户看得懂的类型：

- 模型连接失败
- 飞书授权失败
- 浏览器连接失败
- 文件读写失败
- Skill 安装失败
- 任务执行中断

#### 2. 修复建议
每类错误需要给出：

- 原因说明
- 推荐动作
- 可复制操作或命令
- 重新检测入口

#### 3. 再检测 / 重试
不能只报错不行动，必须支持：

- 重新检测
- 重新测试
- 重试任务

### 14.3 第一版不要做太重的内容

暂时不要一开始就做：

- token 级 tracing
- 复杂拓扑图
- 工程师向监控平台
- 超详细底层请求堆栈展示

第一版重点是：

> **用户能看懂的状态和建议。**

## 15. Windows 第一版用户路径设计

### 15.1 成品形式

最终用户拿到的是：

- `OpenClaw Office Console Setup.exe`

### 15.2 首次启动流程

#### 第一步：检测本机是否已有 OpenClaw
提供两个入口：

- 新安装
- 关联已有实例

说明：

- 这是一个办公增强控制台

- 可写目录

- 浏览器可用性
- 本地运行条件

优先展示，必须先做。

#### 第五步：飞书配置
作为第一优先办公渠道接入。

#### 第六步：推荐 Agent / Skill 安装
建议勾选：

- Console Agent（默认）
- 飞书知识助手
- 文档初稿助手
- 浏览器操作助手

#### 第七步：完成并进入首页

## 16. 与 OpenClaw 原后台兼容的设计策略

### 16.1 接口兼容

建议优先复用 OpenClaw 原本后端的：

- 配置接口
- Agent/Skill 接口
- 任务状态接口
- 日志接口
- 健康检查接口

如不足，则在本地服务层做适配。

### 16.2 数据兼容

应尽量兼容：

- 原有 workspace
- 原有配置文件
- 原有 skills 目录
- 原有日志目录
- 原有运行实例

### 16.3 页面兼容

Office Console 不嵌入原后台做主界面，但可提供：

- 打开原始后台
- 进入高级后台
- 查看原始日志

这样角色分层清晰。

## 17. 技术架构草案

### 17.1 推荐整体技术路线

#### 桌面壳
- **Tauri**

#### 前端
- **React**
- **TypeScript**
- **Tailwind**
- 可选：**shadcn/ui**

#### 本地服务层
负责：

- 环境检测
- 配置读写
- Skill 管理
- 本地进程管理
- 接口代理
- 日志聚合
- 健康检查

#### OpenClaw 后端兼容层
负责与原有 OpenClaw 后端交互。

### 17.2 为什么优先 Tauri 而不是 Electron

基于前序讨论，更推荐 Tauri 的原因：

- 体积更小
- 桌面应用资源占用更克制
- 更适合作为轻量级本地控制台壳
- 易于结合 Web 控制台和本地系统调用

### 17.3 建议架构图

```text
Windows Desktop App (.exe)
├─ Tauri Shell
│  └─ React Web UI
├─ Local Service Layer
│  ├─ Environment Checker
│  ├─ Config Manager
│  ├─ Skill Manager
│  ├─ Process Manager
│  ├─ Diagnostics Service
│  └─ OpenClaw Adapter
└─ OpenClaw Backend / Existing Instance
```

## 19. 之前讨论中哪些方向是有价值的

这一节用于回顾前面对话中真正值得保留的结论。

### 19.1 有价值且应保留

#### 1. 飞书优先
这是办公场景里最贴近落地的渠道之一，适合做：

- 知识库承载
- 教程助手落地
- 文档写入
- 组织化沉淀

#### 2. 模型配置必须显眼
对大多数用户来说，能不能连接模型是最直接的系统价值入口。

#### 3. 运行状态与错误反馈必须做
这是产品成立的基础设施，不是锦上添花。

#### 4. 文档工厂方向很有价值
但应作为后续高级能力，而不是第一版主入口。

#### 5. Skill 应作为能力内核
控制台负责可视化入口，Skill 负责真正的执行逻辑，这是合理结构。

#### 6. 与原有 OpenClaw 后台兼容
尽量避免另起一套底层系统。

### 19.2 有价值但应降级的方向

#### 1. 多 Agent 可视化管理
价值在于“任务过程可见”，不是“炫技式展示多 Agent 协同”。

应降级为：

- Agent 卡片管理
- 任务状态透明
- 步骤可见
- 错误可修复

而不是先做复杂编排图。

#### 2. 一键安装器
有价值，但如果脱离办公控制台本身单独做“通用安装器”，差异化不足。

应转为：

- Office Console 的安装与首启体验
- 而不是一个泛安装器产品

### 19.3 当前不建议作为主轴的方向

#### 1. 一开始就做通用多平台
会明显做重。

#### 2. 一开始就做专业长文档工厂主界面
会劝退新用户。

#### 3. 一开始把产品卖点定义成“多 Agent 平台”
用户更关心“当前能不能用”和“现在做到哪一步”。

#### 4. 过早做复杂平台层
你的壁垒更可能来自办公场景包，而不是通用平台层。

## 20. 专业文档生成与排版流水线的关系

虽然第一版不应把文档工厂放在主入口，但这条线仍然值得保留，并可作为第二阶段能力。

### 20.1 文档工厂应处于什么位置

建议作为：

- 高级 Agent
- 高级办公能力包
- 模板中心中的高级模式
- 后续付费或增强能力方向

### 20.2 其技术方向仍成立

此前讨论形成的判断仍成立：

- 内容生成层：LLM 分章节生成 Markdown / QMD
- 排版层：Pandoc / Quarto + reference-docx
- 输出层：DOCX / PDF
- RAG 不是前置必需，可后加

### 20.3 为什么现在不放前台

因为：

- 用户尚未完成基础配置认知
- 这属于专业重能力
- 与“控制台”的第一版主任务不一致

## 24. 关键产品原则（最终收敛版）

这一节是对整个对话最重要判断的最终收敛。

### 原则 1：第一版优先做 Windows
因为目标是验证普通办公用户是否真的能用起来。

### 原则 2：对外是桌面应用，对内是本地网页控制台
兼顾产品感和工程实现效率。

### 原则 3：先做控制台，不先做最重能力
先解决“上手”和“信任”，再解决“专业深度”。

### 原则 4：先做少量高价值 Agent
不要一开始堆太多能力。

### 原则 5：运行状态与错误反馈必须是核心能力
这是第一版能否成立的关键。

### 原则 6：Console Agent 必须存在
它是系统建议、配置提醒和更新分发的统一中枢。

### 原则 7：与原生后台兼容，但不依赖原后台页面做主入口
让控制台更面向办公用户。

### 原则 8：Skill 是能力内核，控制台是产品壳
这个分层必须坚持。

## 27. 最后的客观判断

这个项目如果继续推进，最容易成功的前提不是“功能很多”，而是以下三点同时成立：

1. **新用户能看懂**
2. **运行过程能看见**
3. **出错以后有反馈、有建议、有重试**

如果做不到这三点，哪怕后面接了飞书、接了很多 Agent、接了文档工厂，也仍然会像一个“功能堆”。

如果先把这三点做扎实，后续不论你接飞书知识库、文档工厂、项目材料助手，都会很自然地长出来。
