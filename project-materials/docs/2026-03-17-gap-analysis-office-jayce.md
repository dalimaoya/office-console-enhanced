## 一、专业用户真正需要什么

### 1.1 用户画像锚定

目标用户不是小白，他们的特征是：

- 自己搭了多个 agent，知道 OpenClaw 是怎么运行的
- 日常用 AI 做任务编排、内容生产、信息处理等
- 在意效率，不愿意翻目录查文件来了解进度
- 愿意花时间配置，但不希望每次操作都要进命令行

**他们打开控制台的心理模型只有三个动作：看、推、控。**

> "我的 agent 们现在在做什么？任务进展怎样？有没有超出预算？"

---

### 1.2 高频高价值操作（Daily Driver）

这些是专业用户每天打开控制台都要做的事，必须顺畅：

| 优先级 | 操作 | 当前满足度 | 说明 |
|--------|------|-----------|------|
| ★★★ | 快速了解哪些 agent 正在工作、谁卡住了 | ⚠️ 部分满足 | 有列表，但状态精度低（无法区分"正在执行"vs"待处理"） |
| ★★★ | 查看任务推进进度（哪个在跑、哪个阻塞） | ❌ 无法满足 | 任务只读，无法写入状态、无法推进 |
| ★★★ | 快速定位某个 agent/任务/会话 | ❌ 无法满足 | 无搜索，数量多时只能滚动 |
| ★★★ | 实时感知变化（不需要手动刷新） | ✅ 满足 | SSE 推送是我们的核心优势，优于 CC |
| ★★★ | 接收飞书告警通知（任务阻塞/异常） | ✅ 满足 | 飞书 Webhook 集成，办公场景独有优势 |
| ★★ | 推进任务状态（标记完成/阻塞） | ❌ 无法满足 | 无写操作 API，核心缺口 |
| ★★ | 查看当日 token 消耗与花费趋势 | ❌ 无法满足 | 用量分区只有基础统计，无归因和趋势 |
| ★★ | 了解哪个 agent 快撑爆上下文窗口 | ❌ 无法满足 | 无上下文压力指标 |
| ★★ | 快速应用 agent 配置模板 | ✅ 满足 | YAML 模板系统，CC 无此能力 |
| ★★ | 查看 agent 的最近产出 | ❌ 无法满足 | agent 详情无最近产出展示 |

### 1.3 低频但必要的操作（Must-Have, Not Daily）

| 操作 | 当前满足度 | 说明 |
|------|-----------|------|
| 审批/拒绝高风险操作 | ❌ 没有 | 多 agent 生产环境需要审批机制 |
| 查看安全配置状态 | ✅ 满足 | 设置分区基本覆盖 |
| 导出当前状态快照（备份） | ❌ 没有 | 生产环境需要 |
| 查看接线状态诊断 | ❌ 没有 | 数据缺失时无法自助排查 |
| Cron 定时任务健康检查 | ❌ 没有 | 定时任务跑偏了得知道 |
| 查看/更新配置文件 | ❌ 部分（只读） | 记忆/文档均无写能力 |
| 版本更新状态 | ❌ 没有 | 知道要不要升级 |

### 1.4 专业用户根本不需要的

专业用户有 AI 能力、愿意折腾，以下内容对他们价值极低甚至是噪音：

| 功能 | 理由 |
|------|------|
| 新手安装向导 | 他们自己会装，详细 INSTALL_PROMPT 是多余的提示 |
| 非技术风险翻译卡片 | 他们能直接读懂配置，不需要"小白语言"解释风险 |
| 系统 readiness 评分 Done Checklist | 配置对了自然就知道，不需要分数化评估 |
| 过于详细的 onboarding 文档 | 读 README 够了，不需要控制台里再内嵌文档 |
| 全套中英双语 UI | 目前团队是中文用户，国际化是锦上添花不是刚需 |

## 二、当前控制台的办公适配度

### 2.1 当前能满足的场景

| 场景 | 描述 | 满足质量 |
|------|------|---------|
| **实时监控** | 无刷新实时感知 agent/任务状态变化 | ⭐⭐⭐⭐⭐ 优秀，优于 CC |
| **飞书告警** | agent 异常、任务阻塞自动推送飞书 | ⭐⭐⭐⭐⭐ 优秀，CC 无此能力 |
| **agent 上线状态总览** | 看哪些 agent 活着、哪些有告警 | ⭐⭐⭐ 基础可用 |
| **模板快速部署** | 用预置 YAML 模板一键配置 agent | ⭐⭐⭐⭐ 良好，CC 无此能力 |
| **文档/记忆浏览** | 查看各 agent 的工作目录、文档 | ⭐⭐⭐ 基础可用（只读） |
| **协作接力可视化** | 看父子会话接力关系 | ⭐⭐ 有但展示质量不高 |
| **安全配置确认** | 只读模式、Token 状态确认 | ⭐⭐⭐ 基础可用 |

### 2.2 无法满足的场景

这些是专业用户在办公中实际会遇到的需求：

| 场景 | 具体问题 |
|------|---------|
| **任务推进** | 看到任务卡住了，想标记状态或触发下一步，控制台里做不到 |
| **快速找到一个会话/任务** | 有 10+ 个 agent 在跑，找某个任务要一直滚列表 |
| **成本控制决策** | 不知道今天花了多少 token，不知道哪个 agent 是消耗大户 |
| **上下文风险预判** | 不知道哪个会话快满了，容易出现 context overflow 后才发现 |
| **审批高风险动作** | 多人协作场景下，无法对高风险操作做二次确认 |
| **排查数据异常** | 某个分区数据不对，不知道是接线问题还是 bug |
| **回溯历史执行** | 某个任务一周前跑了什么，没有回放能力 |
| **Cron 任务健康感知** | 定时任务静默失败，控制台里完全看不到 |

### 2.3 操作顺畅度评估

| 维度 | 评分 | 理由 |
|------|------|------|
| **实时感知** | 9/10 | SSE 推送体验极佳，远超 CC 的手动刷新 |
| **信息密度** | 5/10 | 有数据但语义浅，"看不懂在干嘛" |
| **操作闭环** | 2/10 | 看到问题无法直接处理，需要跳出控制台 |
| **搜索与定位** | 1/10 | 完全没有搜索，用户量多后无法使用 |
| **成本感知** | 2/10 | 用量分区信息太少，无法支持决策 |
| **整体顺畅度** | **4/10** | 展示层可用，操作层几乎缺失 |

> **定性结论**：当前控制台是一个"高质量仪表盘"但不是"工作工具"。专业用户能感知状态，但无法在控制台里完成任何实质性工作推进。

## 三、参考项目（CC）在办公场景上的优势

CC 有而我们没有、且对办公场景直接有价值的能力：

### 3.1 高影响力差距

| CC 功能 | 办公价值 | 缺失影响 |
|---------|---------|---------|
| **任务 CRUD（创建/更新/过滤）** | 核心工作流，不能在控制台推进任务就只能用控制台"看" | 操作链断裂，用户必须切出去做 |
| **全站搜索** | 10+ agent 场景下快速定位是基本需求 | 只能滚列表，用户体验极差 |
| **上下文压力指标** | 预判哪个会话快满，主动干预避免失败 | 等出错才知道，成本高 |
| **Token 消耗归因** | 知道哪个任务/agent 最贵，支持预算分配 | 无法做任何成本优化 |
| **待办操作队列** | 自动聚合需要处理的异常，变被动为主动 | 问题分散各处，容易漏掉 |

### 3.2 中等影响力差距

| CC 功能 | 办公价值 | 缺失影响 |
|---------|---------|---------|
| **审批操作系统** | 高风险动作需要二次确认，多人协作场景必备 | 安全治理空白 |
| **Cron 定时任务总览** | 定时任务是自动化的核心，需要健康监控 | 静默失败无感知 |
| **会话详情页** | 深入某个具体会话的执行内容 | 只能看摘要，无法溯源 |
| **接线状态诊断** | 数据缺失时自助排查 | 用户不知道是配置问题还是 bug |
| **异常流集中管理** | 所有异常统一入口，按严重级别排序 | 异常散落各处，需要逐个排查 |
| **任务回放索引** | 回溯历史问题 | 出了问题无法追溯 |

### 3.3 低影响力差距（对当前目标用户）

| CC 功能 | 说明 |
|---------|------|
| 安全风险翻译卡片 | 专业用户不需要"非技术语言"解释 |
| Done Checklist 评分 | 过度工程化，专业用户一眼就知道接了什么 |
| 国际化（中英双语） | 当前团队中文用户为主 |
| 安装向导 | 专业用户不需要 |
| PM2 部署脚本 | 有价值但不在使用频率内 |

## 四、差距机会点

### 如果要让专业用户"真正用起来"，最缺的 5 个功能

---

#### 🔴 机会点 1：任务推进操作（Make it Actionable）

**问题描述**：用户打开控制台，看到任务卡着、agent 阻塞，却什么也做不了，只能去命令行或飞书手动触发。这是"断链"体验。

**需要什么**：
- `PATCH /api/v1/tasks/:id/status` — 标记任务进行中/完成/阻塞
- `POST /api/v1/tasks` — 从控制台创建新任务
- 任务卡片可交互（点击状态标签弹出操作）

**办公价值**：把控制台从"仪表盘"变成"工作面板"，用户不用离开控制台就能推进工作。

---

#### 🔴 机会点 2：全站搜索（Find Anything Fast）

**问题描述**：用超过 10 个 agent 时，找一个任务/会话靠滚列表，完全不可用。专业用户需要"直达"而不是"浏览"。

**需要什么**：
- `GET /api/v1/search?q=&type=tasks|sessions|agents` — 统一搜索入口
- 顶栏搜索框，`Cmd/Ctrl+K` 快捷键
- 支持 agent 名、任务标题、会话内容关键字

**办公价值**：时间就是生产力。找一个东西从 30 秒变 3 秒，专业用户天天都用。

---

#### 🔴 机会点 3：成本感知与控制（Know What You're Spending）

**问题描述**：多 agent 跑起来之后，用量分区只告诉你"用了多少"，不告诉你"是谁用的""超了没有""还能用多少"。专业用户在做效率提升，但代价不清楚。

**需要什么**：
- Token 消耗归因视图（哪个 agent/任务消耗最多）
- 上下文压力指标（哪个会话接近 context 上限）
- 用量趋势（今天 vs 昨天 vs 7 天均值）
- 可选：简单预算阈值告警

**办公价值**：控制 AI 使用成本是专业用户的核心关切，没有这个他们用着没有底气。

---

#### 🟡 机会点 4：Agent 工作状态精确化（Who's Really Working）

**问题描述**：现在的 agent 列表只能告诉你"正常/异常"，无法区分"正在执行任务"vs"待命等待"vs"卡住了"。专业用户管多个 agent，排班感知是关键。

**需要什么**：
- agent 状态细化：`working` / `idle` / `blocked` / `backlog`
- 显示当前正在处理的任务标题
- 显示最近完成的产出（最近一条 output 摘要）

**办公价值**：管 AI 团队就像管人，知道谁在干活才能做调度。

---

#### 🟡 机会点 5：异常与待办聚合（Surface What Needs Attention）

**问题描述**：现在的告警是被动推送（飞书通知），控制台里没有"待处理事项"的统一视图。专业用户需要一个清单式的"待我处理"面板。

**需要什么**：
- 待办操作队列（`/api/action-queue`）— 基于异常流自动聚合
- 异常流分级展示（按严重程度排序）
- 控制台首页"待处理事项"卡片，一眼看到有什么要做的

**办公价值**：从"被通知"升级为"主动管理"，专业用户看控制台第一眼就知道今天要做什么。

## 五、综合评估矩阵

| 功能 | 办公价值 | 实现复杂度 | 优先级 |
|------|---------|-----------|--------|
| 任务推进写操作 | 极高 | 中（API + 前端联动） | **P0** |
| 全站搜索 | 极高 | 中（搜索 API + 搜索框） | **P0** |
| 成本感知（token 归因 + 压力指标） | 高 | 中低（读取现有 telemetry） | **P0** |
| agent 状态精确化 | 高 | 低（扩展字段 + UI 调整） | **P1** |
| 待办/异常聚合面板 | 高 | 中（异常流 + 操作队列） | **P1** |
| Cron 健康监控 | 中 | 低（新增只读端点） | **P1** |
| 会话详情页 | 中 | 中（路由 + 详情 API） | **P2** |
| 审批操作系统 | 中（多人场景） | 高（安全守卫 + UI） | **P2** |
| 任务回放 | 低中 | 高（时序数据处理） | **P3** |
| 国际化 | 低（当前场景） | 高 | **P3** |

## 六、保持与强化的核心优势

以下是我们相比 CC 已有的差异化能力，不能丢，要强化：

| 能力 | 当前状态 | 强化建议 |
|------|---------|---------|
| **SSE 实时推送** | ✅ 架构优势，CC 无法匹敌 | 持续维护，作为核心卖点 |
| **飞书 Webhook 告警** | ✅ 深度集成，办公场景独有 | 增加更多告警场景（如成本超阈值、cron 失败） |
| **YAML 模板系统** | ✅ 办公场景独有，CC 无 | 增加更多办公模板，支持模板分类浏览 |
| **IP 白名单控制** | ✅ 安全细粒度 | 保持，可在 UI 中增加说明 |

## 七、结论

### 一句话总结

> **当前控制台是"高品质观察工具"，但专业用户需要的是"可以在里面工作的工具"。核心缺口不在功能数量，在于"能不能做事"。**

### 三个关键转变

1. **从只读到可写** — 任务状态可以在控制台里推进
2. **从浏览到搜索** — 不再依赖滚列表找东西
3. **从告警到感知** — 成本、压力、异常在控制台里主动可见

### 最小可用版本定义（Minimal Useful Product）

如果只做一件事，让专业用户觉得"值得打开控制台"：

> **任务状态可写 + 搜索入口 + Token 归因视图**

这三件事做完，控制台就从"监控大屏"变成了"日常工作面板"。
