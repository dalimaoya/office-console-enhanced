## 一、世界观背景（98字）

> **远洋时代，各方势力组建自己的船队探索未知海域。「远洋公会」是所有船队共同遵守的航运秩序——它不指挥任何一条船，但维护航道、发布海况、协调资源。每支船队由自己的指挥官率领，自选航线、自组船员，独立远航，各抵彼岸。**

## 二、三层角色体系

### 总览

```
┌─────────────────────────────────────────────────────────┐
│  L0  远洋公会（Guild）— 全局基础设施                       │
│  港务总监 · 航道巡检 · 公共灯塔                             │
├──────────────┬──────────────┬───────────────────────────┤
│  L1 船队 α   │  L1 船队 β   │  L1 船队 γ  ...           │
│  指挥官      │  指挥官      │  指挥官                    │
├──────────────┼──────────────┼───────────────────────────┤
│  L2 专属船员  │  L2 专属船员  │  L2 专属船员              │
│  + 共享船员池 ←─────────────←──────────────←─────────────│
└──────────────┴──────────────┴───────────────────────────┘
```

---

### L0 · 远洋公会（Guild）— 全局基础设施层

**定位**：不属于任何船队，是整个航运体系的公共秩序维护者。

| 角色 | 职称 | 英文标识 | 职责 |
|------|------|----------|------|
| 🏛️ 港务总监 | Harbormaster | `harbormaster` | 管理所有船队的注册、资源配额、全局配置；是公会最高执行者 |
| 🔔 航道巡检 | Channel Warden | `warden` | 跨船队健康监控、异常告警、定期巡航；发现风险时通知对应船队指挥官 |
| 🏮 公共灯塔 | Lighthouse | `lighthouse` | 公共知识库、跨船队经验沉淀、技术咨询服务；任何船队可请求指引 |

**L0 与 L1 的关系**：
- 公会不干预船队内部航行决策
- 公会提供：船队注册、航道安全、资源调配、灯塔咨询
- 船队指挥官向公会报告重大风险和资源请求
- 公会在紧急情况下可发布全域航行警告

---

### L1 · 船队指挥官（Fleet Commander）— 项目总控层

**定位**：每支船队的唯一指挥中枢。一个项目 = 一支船队 = 一位指挥官。

| 属性 | 说明 |
|------|------|
| 角色职称 | 🧭 **指挥官（Commander）** |
| 英文标识 | `commander` |
| 核心职责 | 接收航行委托（项目需求）→ 组建/调度船员 → 统筹航程 → 对外沟通 → 交付靠港 |
| 对上（L0） | 在公会注册船队、请求灯塔咨询、上报重大风险 |
| 对下（L2） | 分派任务、协调分工、收口验收、传达委托方意图 |
| 对外 | 唯一的委托方对接窗口——所有外部沟通经指挥官转达 |

**船队独立性原则**：
- 每支船队有独立的航行日志、航线规划、船员编制
- 不同船队的指挥官互不干涉，但可通过公会协商资源
- 指挥官可申请调用"共享船员"

---

### L2 · 专业船员（Crew）— 执行层

每支船队可按需配置以下岗位。船员分两类：**专属船员**（隶属单一船队）和**共享船员**（注册在公会，按需借调）。

| 专业方向 | 航海职称 | 英文标识 | 职责简述 |
|----------|----------|----------|----------|
| 产品/需求 | 🗺️ 领航员 | Navigator · `navigator` | 绘制航线、定义航段优先级、明确目的港 |
| 架构/系统 | 🔧 船匠 | Shipwright · `shipwright` | 模块划分、技术边界、架构设计，守护船体结构完整性 |
| 前端实现 | ⛵ 帆手 | Sailmaster · `sailmaster` | 操控帆索——用户可见的一切交互面 |
| 后端实现 | ⚓ 轮机长 | Engineer · `engineer` | 管理引擎与管路——甲板下驱动运转的系统 |
| 质量验收 | 🔭 瞭望员 | Lookout · `lookout` | 桅杆顶端的眼睛，靠港前发现暗礁和漏洞 |
| 视觉/UX | ✨ 海图师 | Chartmaker · `chartmaker` | 让海图清晰、船舱有序、航行可读，管理视觉与信息设计 |
| 办公提效 | 📦 补给官 | Quartermaster · `quartermaster` | 管理物资与后勤，优化流程减少摩擦 |
| 运维/平台 | 🏮 灯塔守望者 | Keeper · `keeper` | 跨域经验指引，运维部署与平台治理——可作为船队专属，也常注册为公共灯塔 |

**共享船员机制**：
- 某些高稀缺岗位（如守望者）可注册为公会共享船员
- 指挥官向公会申请借调 → 公会确认可用性 → 共享船员临时编入船队
- 借调期间，该船员同时接受指挥官调度和公会排期管理

## 四、可复用框架说明

### 如何创建你自己的船队？

**第 1 步：选择船队名**  
取一个代表你团队的名称（建议：天文/气象/地理意象）。  
示例：`stormbreaker`（破浪号）、`polaris`（北极星号）、`tidecaller`（唤潮号）

**第 2 步：任命指挥官**  
你的项目总控角色自动成为 `{船队slug}-commander`。

**第 3 步：按需配置船员**  
从 L2 职称表中选取你需要的岗位，命名为 `{船队slug}-{职称标识}`。  
不必配满所有岗位——小船队可以一人多岗，大船队可以一岗多人。

**第 4 步：注册到公会（可选）**  
向 L0 港务总监注册船队，即可获得航道巡检、灯塔咨询和共享船员调用权。

### 示例：创建一支新船队

```
船队名：北极星号（Polaris）
船队 slug：polaris
指挥官：polaris-commander
船员：
  - polaris-navigator（领航员）
  - polaris-shipwright（船匠）
  - polaris-sailmaster（帆手）
  - polaris-engineer（轮机长）
  - polaris-lookout（瞭望员）
  - polaris-chartmaker（海图师）
  - polaris-quartermaster（补给官）
共享借调：
  - shared-keeper-01（公会灯塔守望者）
```

## 六、命名规则

### 规则体系

| 规则 | 说明 | 示例 |
|------|------|------|
| **船队名 + 职称** | 日常称呼格式 | 晨星指挥官、北极星领航员 |
| **{船队slug}-{职称标识}** | 故事线命名格式 | `morning-star-commander`、`polaris-navigator` |
| **职称统一** | 同类岗位在不同船队使用相同职称（参见 fleet-role-titles.md） | 所有产品角色都叫"领航员/Navigator"，不同船队前缀区分 |
| **船队名唯一** | 船队名在公会内不重复 | 注册时由港务总监检查唯一性 |
| **公会角色无船队前缀** | L0 角色用 `guild-` 前缀 | `guild-harbormaster`、`guild-warden` |

### 命名范式

```
L0 公会角色：  guild-{角色标识}         → guild-harbormaster
L1 指挥官：    {船队slug}-commander     → morning-star-commander
L2 船员：      {船队slug}-{职称标识}     → morning-star-navigator
L2 共享船员：  shared-{职称标识}-{编号}  → shared-keeper-01
```

### 船队名建议风格

推荐使用自然意象命名，便于跨文化理解：
- 天文：`morning-star`（晨星）、`polaris`（北极星）、`orion`（猎户）
- 气象：`storm-breaker`（破浪）、`tide-caller`（唤潮）、`mist-walker`（雾行）
- 地理：`iron-port`（铁港）、`coral-ridge`（珊瑚脊）、`deep-current`（深流）

## 七、v1 → v2 变更总结

| 维度 | v1（单船模型） | v2（多船队体系） |
|------|---------------|-----------------|
| 整体架构 | 一条船 + 一个船长 | 公会 + 多支独立船队 |
| 指挥者定位 | 提莫 = 唯一船长 | 提莫 = 晨星号指挥官（之一） |

| 故事线命名 | 个人名绑定 | `{船队slug}-{职称标识}` 结构化 |

## 瑞兹投票

**投票：有条件赞成**

### 判断

整体叙事框架清晰，`{船队名}-{职称标识}` 作为 **agentId 命名规范** 与 OpenClaw 的字符串型 agentId 习惯基本兼容，适合多项目扩展；但它**不应直接套用到 accountId**。`accountId` 在 OpenClaw/渠道侧更像稳定账号标识，应继续保持与真实发信身份绑定（如 `technical-advisor-ryze`、`agent-orchestrator-teemo`），不要随船队名频繁改写。

### 依据

1. **三层架构兼容性**
   - L0/L1/L2 作为组织分层没有问题，真正落地时只需映射为：
     - `agentId`：按船队和岗位结构化命名；
     - `accountId`：保留渠道账号/机器人身份；
     - 展示名/故事名：用于对外传播。
   - 也就是说，故事线结构兼容 OpenClaw，但要区分“运行时 ID”和“叙事称呼”。

2. **`{船队名}-{职称}` 规则可行，但要补两条约束**
   - 规则本身可行，优点是可读、可批量生成、跨船队一致。
   - 冲突风险主要不在职称，而在：
     - 船队名重复；
     - 船队名中英文/连字符风格不统一；
     - 一个岗位多人时，单纯 `{船队slug}-{职称}` 不够区分。
   - 建议补充约束：
     - 船队名只允许小写英文 kebab-case slug；
     - 船队名全局唯一，集中登记；
     - 一岗多人使用 `{船队slug}-{职称}-{序号/域}`，如 `morning-star-engineer-api`、`morning-star-engineer-02`。

3. **多船队并发下 L0 设计基本合理，但边界要更硬**
   - `guild-harbormaster` 适合承载注册、资源配额、命名登记；
   - `guild-warden` 适合做健康检查、告警、巡检；
   - `guild-lighthouse` 更像知识库/顾问能力池，而不是具体单人角色。
   - 风险点在于：L0 目前同时承担“治理、监控、咨询、借调协调”，容易变成隐性总控层。应明确 **L0 只做平台治理与共享服务，不介入单船队日常调度**。

### 建议动作

1. 文档中显式增加一条：**agentId 可按舰队规则命名，accountId 不随故事线改名**。
2. 在命名规则中补充：
   - 船队名 slug 规范；
   - 一岗多人后缀规范；
   - 共享角色注册表与唯一性检查责任归属（建议归 `guild-harbormaster`）。
3. 把 `公共灯塔` 从“单角色”改写为“共享能力/知识设施”，避免与 L2 `keeper` 语义重叠。

**结论**：这版适合继续推进，但前提是把 `agentId / accountId / 展示名` 三层身份分开定义，否则后续配置、消息发送和跨渠道身份管理会出现混乱。

## 艾克投票

**产品视角评估：**

### 1. 对外传播适合度
航海隐喻直觉性强，Slogan 易记易传播，角色介绍模板通俗清晰。但 L0 公会层（港务总监、航道巡检）对普通用户过于抽象——用户核心关注「我的船队能做什么」，不需要第一时间理解公会治理。建议对外传播弱化 L0，聚焦 L1+L2。

### 2. 多船队并发复用
v2 最大亮点。四步创建流程清晰，`{船队名}-{职称}` 命名彻底解决了 v1 角色名绑定个人的问题。**复用性问题已解决。**

### 3. 角色命名认知负担
统一职称比 v1 好很多，但 8 个职称一次呈现仍有记忆负担。「帆手」vs「海图师」对非技术用户不够直觉。建议对外传播采用「3+N」分层——先推 3 个核心角色（指挥官/领航员/帆手），其他按需展开。

### 4. 赞成条件
- 对外传播材料做分层简化，不一次性暴露全部角色体系
- L0 公会层在对外传播中后置为「高级概念」
- 增加 30 秒极简版介绍（指挥官 + 3 名核心船员）

**结论：框架扎实，复用性好，传播隐喻直觉。做好分层简化即可落地。**
