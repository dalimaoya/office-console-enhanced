# CC 分析报告：办公效率用户视角
**分析角色**：Jayce（aioffice-jayce，办公增强控制台 AI 办公顾问）  
**参考项目**：https://github.com/TianyiDataScience/openclaw-control-center  
**分析日期**：2026-03-18  
**视角定位**：知识工作者 / 每天使用 AI 工具的效率型用户 / 关心"用了之后省了多少步"

---

## 一、高频场景体验：打开 → 看状态 → 解决问题 → 关闭

### CC 的路径
1. 打开 `http://127.0.0.1:4310/?section=overview`
2. **一屏看清**：系统总体是否正常、谁在工作、是否有待处理事项、预算是否有风险
3. 如果有异常：点击对应区域（任务/员工/用量）深入
4. 问题定位后：Settings 确认接线状态或安全风险
5. 关闭

**步骤数：2-4 步，核心问题（"OpenClaw 现在正常吗？"）1 步内可得答案。**

### 我们现在的路径（基于 OpenClaw 默认控制界面）
1. 打开控制台
2. 翻聊天记录或日志找 agent 运行状态
3. 逐一查看 sessions / workspaces 列表
4. 手动估算谁在跑、谁卡住
5. 切换到另一个界面看用量
6. 再去 config 确认有没有连接问题
7. 关闭

**步骤数：5-7 步，核心问题往往需要 3+ 步才能回答。**

### 差距分析
| 维度 | CC | 我们现在 |
|------|-----|---------|
| 首屏答案 | 有：一句话状态 | 无：需要拼凑 |
| 问题定位 | Overview → 深入 | 各分散入口单独查 |
| 行动指引 | 卡片内置下一步提示 | 用户自己判断 |
| 认知负担 | 低（非技术用户也能看懂） | 高（需要懂 session/log 结构） |

**对用户实际工作的意义**：每天开机第一件事是"检查 AI 助手状态"，CC 让这个动作从 5 分钟压缩到 30 秒。按每天 3 次检查、每次节省 4 分钟计算，一周节省约 1.4 小时。

## 二、信息密度：一屏展示多少有用信息

### CC 的 Overview 页面信息层级

CC 的 Overview 是核心情报页，一屏聚合：

```
第一层（一眼确认）：系统是否正常
第二层（快速扫描）：待处理事项 / 异常 / 预算风险 / 谁在工作
第三层（需要时深入）：具体 agent 状态 / 协作链路 / 执行证据
```

具体卡片布局（推断）：
- **系统健康卡**：一句话 OK / WARNING / BLOCKED
- **待处理事项**：审批、卡住的任务（直接列出，不是数字）
- **活跃员工**：谁真的在跑 vs 谁只是排队
- **预算风险**：今日花费趋势、是否接近阈值
- **最近异常**：停滞执行、runtime 错误

**对用户的意义**：信息密度的核心不是"塞多少"，而是"每一格都有决策价值"。CC 的卡片设计对应的是运营者的真实问题，不是技术指标。

### 我们现在摆错了位置的东西

1. **Session 列表在首页**：对用户来说，session 是技术细节，不是决策依据。用户关心的是"这个任务做完了吗"，不是"session 的 ID 是什么"。
2. **日志/流式输出作为状态**：日志是排障工具，不是状态仪表盘。现在把 raw log 当成状态可见性的主要形式，信噪比太低。
3. **用量数据藏在设置里**：用量和花费是每日决策依据（"今天还有多少预算可以跑任务"），应该在主界面高可见，而不是要去设置里翻。
4. **"谁在工作"没有可视化**：我们的界面很难一眼分辨哪个 agent 当前活跃、哪个只是配置了但没在运行。CC 专门有 Staff 页解决这个问题。

## 三、响应速度感知：让用户觉得"快"的设计

### CC 的设计推断（基于 README 描述和项目定位）

**1. 明确的降级提示（而非白屏报错）**

CC 的安装逻辑中强调："如果数据源缺失，继续运行，标注哪些面板会降级。"这反映在 UI 上就是：某个卡片空数据时，显示的是"此信号暂不可用，原因：Gateway 未连接"，而不是整页报错或无限 loading。

对用户的意义：即使环境不完整，用户也能用剩余的面板做事，不会被"整个页面卡住"。

**2. 设置页的"接线状态"卡片**

用户最常见的困惑是"为什么这个数据没有？"CC 直接在 Settings 给出"哪些已接好，哪些还差一步"，用户不用猜，不用去翻文档。这让排障从 10 分钟缩短到 2 分钟。

**3. 只读默认 + 本地运行**

由于是本地运行，没有网络延迟瓶颈。只读模式减少了写操作等待，整体感知速度更快。

**4. URL 参数直达（`?section=overview`）**

可以把特定页面加入书签，直接打开就在想要的 section，无需每次从首页导航。

### 我们缺少的
- **加载占位/骨架屏**：数据没来之前，界面不应该是空白，应该有结构性的占位（用户知道数据在哪个位置，等一下就来）
- **操作结果即时反馈**：触发某个操作后，是否成功/失败应该有明确的 toast 或状态变化
- **错误信息人话化**：我们的错误提示倾向于技术栈错误（"Connection refused to localhost:3000"），CC 的理念是翻译成"Gateway 还没启动，请先运行 openclaw gateway start"

## 四、键盘/鼠标流

### CC 可能支持的（基于定位推断）
- **URL 书签**：直接跳到特定 section，这是最实用的"键盘流"
- **设置页的卡片展开/收起**：减少滚动
- **Task 页的过滤/排序**（推断）

### 我们完全缺失的

| 能力 | 描述 | 对工作的影响 |
|------|------|------------|
| **全局搜索（⌘K / Ctrl+K）** | 直接搜索任务名、agent 名、session | 不用逐层点击，输入关键词直接跳转 |
| **批量操作** | 批量暂停/重启多个 agent、批量清理 session | 现在只能逐个操作，管理 5+ agent 时极其耗时 |
| **快捷键**（J/K 导航、Enter 确认等） | 键盘驱动的列表导航 | 效率型用户（愿意折腾的那类）特别需要这个 |
| **右键菜单** | 右键 agent 卡片 → 复制 ID / 查看 log / 重启 | 减少"找到它 → 点击 → 确认" 的多步操作 |
| **拖拽重排** | 拖拽调整任务优先级 | 任务管理场景需要 |

**最高优先级缺失**：全局搜索（⌘K）。对知识工作者来说，记住在哪点不如直接搜索。这一个功能可以替代 50% 的导航点击。

## 五、办公场景契合度

### CC 中真正减少操作步骤的功能

**1. "员工"（Staff）页 → 区分"真在跑"和"只是排队"**

这是最被低估的功能。没有这个功能时，用户会把"queued"的任务误认为"正在执行"，然后去等一个根本没在跑的任务。CC 直接告诉你 Leona 是 ACTIVE 还是 QUEUED，不用自己去看 log 推断。

**对工作的意义**：当你想"催一下这个任务"时，你需要先知道它是真的在跑还是堵在队列里。这两种情况处理方式完全不同。节省诊断时间 3-5 分钟/次。

**2. 协作（Collaboration）页 → 看清任务从哪个 agent 转给了哪个**

多 agent 协作中，最常见的困惑是"这个任务现在到哪一步了，在谁手里"。CC 的 Collaboration 页直接可视化接力链，不用猜。

**对工作的意义**：我们团队有 8 个 agent（Teemo/Ekko/Jax/Ezreal 等），当一个任务经过 3 个 agent 接力时，如果没有协作视图，用户只能靠聊天记录推断现在卡在谁这里。每次排查平均需要 5 分钟。

**3. 安全风险摘要 → 翻译成人话的风险说明**

技术安全提示对知识工作者来说是噪音。CC 将其翻译为"当前有 1 个高风险操作开启，影响：可能触发写操作，建议：关闭 IMPORT_MUTATION_ENABLED"。这让非技术用户也能做安全决策。

**4. 上下文压力（Context Pressure）卡片**

对我这个每天运行多个 agent 的用户来说，知道"哪个 session 快到上下文上限了"是个关键信号。快到上限意味着要么续会话、要么任务质量下降。CC 在用量页直接显示这个，我们目前完全没有这个可见性。

**对工作的意义**：可以提前安排 session 清理，避免任务执行到一半因为上下文不够而失败。每次失败的善后成本约 15-30 分钟。

## 六、我们现有控制台最大的使用痛点

### 痛点 1：不知道"现在整体是好的还是有问题"

打开我们的控制台，没有一个地方直接告诉我"系统当前状态"。我必须逐个 session 检查、看 log、猜是否有卡住的任务。这就像每天早晨不看天气预报，直接出门然后发现下雨。

**严重程度**：每日开机必遇，极高频痛点。

### 痛点 2：用量数据不在主流程里

我每天都在问自己"今天花了多少 token"、"还剩多少预算可以跑任务"，但这个数据要去设置里找，还不一定准确或及时。结果要么过度保守（不敢跑任务），要么浪费（跑了不必要的重复任务）。

**严重程度**：影响每个任务的决策质量，高频痛点。

### 痛点 3：排障路径太长

出了问题，我要：看 log → 猜原因 → 翻文档 → 尝试 → 再看 log。CC 的"接线状态"和"安全风险摘要"直接把诊断步骤压缩了。我们没有类似设计，排障全靠经验和运气。

**严重程度**：不是每天遇到，但每次遇到都会消耗大量精力。

## 七、优先级排列：最想要 Ezreal 立即实现的 3 件事

### 🥇 第一优先：Overview 状态卡片（1 屏 1 答案）

**具体需求**：主页增加一个"当前状态卡片"，回答：
- OpenClaw 整体：正常 / 有异常 / 有任务等待你的操作
- 活跃 agent 数量（真正在运行 vs 排队）
- 今日用量简报（消耗了多少，相对预算的百分比）
- 是否有卡住的任务（超过 X 分钟没有进展）

**为什么是第一**：这一张卡片解决了 60% 的日常"检查"需求，让开机 5 分钟检查变成 30 秒扫描。投入产出比最高。

**实现建议**：不需要实时推送，每 30 秒轮询更新即可；使用颜色区分健康/警告/错误状态。

---

### 🥈 第二优先：全局搜索入口（⌘K）

**具体需求**：按 Ctrl+K 弹出搜索框，可以搜索：
- Agent 名称（快速跳到该 agent 的状态页）
- 任务关键词（搜到对应任务记录）
- Session ID 后几位（快速定位）

**为什么是第二**：我们有 8+ 个 agent，随着项目推进任务会越来越多。点击导航找东西比搜索慢 3-5 倍。全局搜索是所有高频用户最期待的效率乘数器。

**实现建议**：先做静态搜索（搜已加载的数据）即可，不需要全文检索。模糊匹配优先。

---

### 🥉 第三优先：用量 + 上下文压力面板

**具体需求**：在主界面（或侧边栏常驻）显示：
- 今日已消耗 token 数 / 预算百分比
- 当前哪个 session 的上下文占用最高（警示接近上限的那个）
- 本周趋势（避免月底超支）

**为什么是第三**：用量可见性直接影响任务决策。每次"要不要再跑一次"这个问题，都需要知道还有多少余量。上下文压力是防止任务中途失败的预防性指标，价值被严重低估。

**实现建议**：数据来源接 OpenClaw Gateway API，每 5 分钟更新一次；超过 80% 预算时显示黄色警告，超过 95% 显示红色。

---

## 总结

CC 的核心价值不是功能多，而是**对齐了用户的认知模型**：用户不关心 session 技术细节，用户关心"我的 AI 助手现在是好是坏，在做什么，花了多少"。CC 的每一页都直接回答一个用户级别的问题，而不是呈现系统级别的数据。

我们目前的控制台是"给懂技术的人看的日志仪表盘"，CC 是"给每天使用 AI 工具的知识工作者设计的控制中心"。这个定位差异，决定了信息架构、优先级展示和排障路径的全部不同。
