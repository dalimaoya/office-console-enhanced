## 1. 产品定位

**一句话定位**：  
这是一套基于 OpenClaw 的**项目控制台原型**，让 AI 多角色协作项目的运行状态可见、可追踪、可管理。当前服务于李琪的办公控制台增强项目，未来目标是沉淀为任何 OpenClaw 项目都能复用的标准控制台能力。

**给谁用？**  
一期用户：李琪（项目发起人）+ 项目组内所有 AI 角色。  
二期目标用户：任何运行 OpenClaw 多角色项目的个人开发者。

**解决什么问题？**  
多角色 AI 协作项目中，"谁在干什么、干到哪了、有没有卡住、花了多少钱"这些信息散落在文件和会话里，没有统一的地方看。控制台解决的就是**项目运行的可观测性**问题。

**能不能被别人复用？**  
- **现在**：不能直接复用。当前能力和李琪项目深度绑定（角色配置、目录结构、业务语境）。
- **一期结束时**：核心能力（状态总览、事件日志、文件契约）以"可迁移项目层"形式沉淀，第二个项目接入时只需适配配置，不需重写逻辑。
- **未来**：成为 OpenClaw 官方项目控制台组件，开箱即用。

## 2. 一期能力边界

### 一期必须做（P0）

| # | 能力 | 为什么必须在一期 |
|---|------|----------------|
| 1 | **Gateway/Agent 状态总览** | 不知道角色在线状态，指挥官无法调度，项目无法推进 |
| 2 | **运行健康监控**（连接/延迟/心跳） | 今天已出现过角色失联导致任务卡住的情况，没有监控就是盲飞 |
| 3 | **项目状态总览**（活跃项目/阶段/owner/阻塞项） | 控制台的核心价值就是"一眼看清项目全局"，没这个等于没有控制台 |
| 4 | **Token 用量与费用统计** | 多角色并发消耗不透明，不统计就不知道成本是否可控 |
| 5 | **操作审计日志**（只追加事件日志） | 排查问题的唯一客观依据，今天的 ?? 语法错误排查已证明必要性 |
| 6 | **文件契约 Schema 显式化** | 文件驱动架构的地基，没有 schema 角色交接就是在猜格式 |
| 7 | **对象注册表 V1** | 没有唯一 ID 和归属登记，控制台上的"项目状态"无处聚合 |
| 8 | **角色原型池**（8个角色标准化定义） | 当前角色配置散落在各 SOUL.md 里，不标准化就无法在控制台统一管理和展示 |
| 9 | **内外双名映射** | 对外表达一致性的基础，已完成（naming-map.md），一期维护即可 |

### 一期不做（明确排除）

| # | 能力 | 为什么现在不做 |
|---|------|--------------|
| 1 | **环境诊断（一键检查）** | 一期团队成员固定，环境问题靠人工排查可接受；二期面向新用户时才必须 |
| 2 | **异常摘要与根因推断** | 依赖足够的事件日志积累和模式识别，一期先把日志基础设施建好 |
| 3 | **跨项目调度视图** | 当前只有一个项目，没有"跨"的场景，三期多项目并行时再做 |
| 4 | **项目组模板** | 一期只有一套团队，模板化是二期"第二个用户"场景的需求 |
| 5 | **皮肤包机制** | 双名映射文件已满足一期需要，配置化切换是二期锦上添花 |
| 6 | **项目实例管理（创建/归档/克隆）** | 一期只跑一个项目实例，实例管理是三期多项目支撑能力 |
| 7 | **一键部署向导** | 面向新用户的引导流程，一期用户是李琪本人，不需要引导 |
| 8 | **并发写入保护** | 一期团队规模小，靠串行调度规避冲突，二期再引入锁机制 |
| 9 | **项目状态机** | 需要注册表先就位才能挂载状态转移规则，二期做 |
| 10 | **冷启动索引** | 一期先建好文件契约和注册表，二期再在此基础上做快速上下文重建 |
| 11 | **DTO / 接口契约文档** | 一期前后端还在快速迭代，过早固化接口会拖慢开发节奏，三期稳定后再做 |

## 3. 产品两条主线范围

### 主线一：运行控制台（Control Center）

**一期定义**：让指挥官和项目发起人实时看到"项目在跑没有、跑得怎么样、花了多少"。

**一期做：**
- Agent 在线/离线状态 + 当前任务 + 上下文压力指标
- 连接状态 + 响应延迟 + SSE 心跳监控 + 告警阈值（硬编码）
- 项目维度状态聚合：活跃项目、当前阶段、各对象 owner（归属责任人）、阻塞项清单；事件时间线单独使用 `source_role` 表示来源角色
- Token 用量：今日/本周/按角色分层统计
- 操作审计日志：关键操作时间线，可回溯

**一期不做：**
- 不做自动诊断（环境检查、根因推断）
- 不做跨项目视图
- 不做告警阈值配置化（一期硬编码，二期可配置）
- 不做历史趋势图表（一期只看当前快照和近期日志）

### 主线二：项目组工厂（Team Factory）

**一期最小集合：只做"角色原型池"。**

具体范围：
- 将现有 8 个 specialist 角色（提莫/艾克/贾克斯/伊泽瑞尔/雷欧娜/加里奥/拉克丝/杰斯）+ 1 个顾问角色（瑞兹）的标准化定义收敛为结构化数据
- 每个角色原型包含：职责边界、默认模型、工具集、交互协议
- 在控制台上可查看角色列表和详情

**一期不做：**
- 不做项目组模板（"开发团队"/"内容团队"等预设组合）
- 不做项目实例化（从模板创建新项目组）
- 不做角色动态增删（一期角色固定）
- 不做一键部署向导

**一句话总结**：一期的"工厂"只是一个"零件展示柜"，展示有哪些角色可以用；二期才开始组装成"生产线"。

## 4. 复用层设计

### 三层划分

```
┌─────────────────────────────────────────────────┐
│  第一层：李琪项目专属层                              │
│  （真实场景基础，不抽象，不改动）                       │
├─────────────────────────────────────────────────┤
│  第二层：可迁移项目层                                │
│  （一期沉淀重点，稍加配置即可被新项目使用）               │
├─────────────────────────────────────────────────┤
│  第三层：通用 OpenClaw 平台层                        │
│  （未来目标，一期不追求）                              │
└─────────────────────────────────────────────────┘
```

### 第一层：李琪项目专属

| 内容 | 说明 |
|------|------|
| 当前 9 个角色的 SOUL.md / IDENTITY.md | 包含项目特定的性格设定、协作记忆，是真实场景土壤 |
| LoL 故事线与代号体系 | 团队内部认知工具，换到别的项目不一定适用 |
| 办公控制台的业务功能（日程/邮件/文档管理） | 李琪的具体业务场景，与控制台基础设施无关 |
| 飞书群 `oc_425a0058997fca9570391b562ba15efb` 的消息流 | 特定项目通信通道 |
| 当前项目目录结构 `projects/office-console-enhanced/` | 项目特定的文件组织 |

**原则**：不动它，它是验证一切抽象是否有效的真实基准。

### 第二层：可迁移项目层（一期沉淀重点）

| 内容 | 迁移方式 | 一期沉淀目标 |
|------|---------|------------|
| **文件契约 Schema** | 新项目直接引用同一套 schema 规范 | 输出 `schemas/` 目录，覆盖 handoff/brief/review/artifact |
| **对象注册表规范** | 新项目按规范建自己的注册表 | 输出注册表 schema + 目录结构标准 |
| **事件日志格式** | 新项目接入同一套日志格式 | 输出日志事件类型定义 + 格式规范 |
| **角色原型定义格式** | 新项目可基于格式定义自己的角色 | 输出角色原型的结构化 schema |
| **内外双名映射机制** | 新项目可配置自己的映射 | `naming-map.md` 本身就是可参照的模板 |
| **控制台数据模型**（Agent 状态/项目状态/用量统计） | 新项目的控制台共用同一套数据结构 | 控制台 API 的数据结构定义独立于业务逻辑 |

**命名统一约束（P0 修订补充）**：
- 对象的归属责任人字段统一叫 **`owner`**；
- 事件的来源角色字段统一叫 **`source_role`**；
- 两者共用同一套角色代号风格，但不互相替代。

**一句话**：一期把"怎么定义角色""怎么交接文件""怎么记录事件""怎么展示状态"这四件事的**规范**沉淀下来，新项目拿着规范就能接入控制台。

### 第三层：通用 OpenClaw 平台层（未来目标）

| 内容 | 为什么一期不追求 |
|------|----------------|
| 项目模板市场（选模板→一键建团队） | 需要至少 3 个以上成功项目验证模板有效性 |
| 插件化角色注册 | 需要先有稳定的角色原型格式，一期正在建 |
| 跨项目资源调度 | 单项目场景下无法验证 |
| OpenClaw CLI 集成（`openclaw project init`） | CLI 能力依赖上游 OpenClaw 核心支持 |
| 公开 API / SDK | 接口还在快速变化，过早发布会成维护负担 |

## 5. 成功标准

一期完成后，验收场景如下：

### 场景一：新团队成员（AI角色）加入
> 一个新角色被加入项目组，**30 分钟内**能通过控制台和项目文件了解：
> - 项目当前处于什么阶段
> - 各角色分别在做什么
> - 自己的职责边界是什么
> - 文件交接应该遵循什么格式

### 场景二：项目发起人（李琪）日常查看
> 打开控制台，**10 秒内**能看到：
> - 所有角色是否在线
> - 当前有没有阻塞项
> - 今日 token 消耗了多少
> - 最近一次关键操作是什么

### 场景三：出了问题要排查
> 某个角色执行出错，通过审计日志，**5 分钟内**能定位到：
> - 什么时间发生了什么操作
> - 哪个角色触发的
> - 影响了什么对象

### 场景四：第二个项目想接入（二期预判）
> 一个新用户看到这套东西后，能理解：
> - "这是一个 AI 多角色项目的控制台"
> - "我可以参照这套规范来配置自己的角色和项目"
> - "文件契约和事件日志格式是通用的，我可以直接用"

**量化指标**：
- 控制台页面加载后，核心状态信息 ≤ 3 秒可见
- 角色原型池覆盖全部 9 个角色，每个有完整的结构化定义
- 文件契约 schema 覆盖 4 类核心对象（handoff/brief/review/artifact）
- 事件日志 ≥ 覆盖 5 类关键操作类型
- 对象注册表可通过程序验证格式合法性
