## 一、CC 对专业用户最有价值的功能模式（场景还原）

### 场景 1：每日开机确认（"AI 助手现在好着吗？"）

**CC 的做法**：  
打开 `?section=overview`，一屏内获得三个确认：
1. **状态色块**：绿/黄/红一眼识别，无需阅读
2. **待处理项数量**：直接显示"需要你操作的事"，不是系统状态
3. **活跃 agent 数**：区分"真在跑"和"排队中"

**关键 UI 模式**：用"颜色 + 数字 + 动词短语"的三元素卡片，把系统状态翻译为运营语言。

---

### 场景 2：任务异常排查（"这个任务卡哪里了？"）

**CC 的做法**：  
Overview → 点击异常 → Tasks 执行链 → 看到是哪个 agent 卡住、卡了多久  

**关键 UI 模式**：信息可以"下钻"，每个摘要卡片都是一个可点击的入口，点击后展示更多细节而不是跳转到一个全新页面。行内展开 > 新标签跳转。

---

### 场景 3：多 agent 协作追踪（"Teemo 派给 Ezreal 的任务到哪一步了？"）

**CC 的做法**：  
Collaboration 页直接显示 `会话 A → 会话 B → 等待回复` 的流向图，而不是文字列表。

**关键 UI 模式**：流程型信息用"流向线"表示，不用列表。接力链路一眼可见。

---

### 场景 4：快速找一个 agent（"Leona 的状态是什么？"）

**CC 的做法**：  
Staff 页用卡片网格展示每个 agent，卡片上有"活跃/等待/卡住"角标，用颜色区分。  

**关键 UI 模式**：把人员管理用"卡片角标"替代行内文字标签，信息密度更高，不需要逐行阅读。

---

### 场景 5：判断今天还能跑多少任务（"还剩多少预算？"）

**CC 的做法**：  
Usage 页的环形进度条，直接显示已用 XX%，今日消耗、7 日趋势、上下文压力三张卡片并排。  

**关键 UI 模式**：数字型数据用进度条/环形图替代原始数字，让"趋势"可见而不只是"当前值"。

## 二、CC 的 UI 设计特征提炼（信息设计层面）

以下是从 CC README、中文文档和截图描述中归纳的 UI 设计规律，供 Lux 参考：

### 2.1 语义颜色系统（Semantic Color System）

| 状态 | 颜色语义 | 应用位置 |
|------|---------|---------|
| 健康/正常 | 绿色系 | 卡片边框、角标、进度条 |
| 警告/风险 | 黄色系 | 预算超 80%、卡片顶部色条 |
| 错误/卡住 | 红色系 | 任务停滞、连接断开 |
| 待处理/中性 | 蓝色系/灰色系 | 排队中的任务 |

**设计要点**：颜色是载体，不是装饰。每一处颜色都对应一个可操作的状态，不随机使用。

---

### 2.2 卡片三件套（Card Triad）

CC 的每张信息卡片都包含：
```
标题（1-3个字，说明"这是什么"）
主值（数字/状态词，最大字号，1秒可读）
辅助说明（灰色小字，解释上下文）
```

例：
```
活跃员工         ← 标题
  3 / 8          ← 主值（大字）
  2 个执行中，1 个待命  ← 辅助说明（灰色小字）
```

**设计要点**：卡片不是"展示全部信息"，而是"主值优先，细节后置"。

---

### 2.3 降级友好（Graceful Degradation）

CC 明确要求：数据源缺失时，继续运行，标注哪些面板降级。  
UI 表现为：灰色面板 + "此信号暂不可用，原因：XXX" 的内联说明。

**设计要点**：空状态不是白板，而是有结构的"占位 + 解释"。

---

### 2.4 人话优先（Operator Language）

CC 把所有技术信息翻译成运营语言：
- `Session context at 87%` → `这个会话快到上下文上限了，可能需要续开`
- `IMPORT_MUTATION_ENABLED=true` → `当前有 1 个高风险操作开启，建议关闭`

**设计要点**：用户读到的文字是行动指引，不是系统状态代码。

---

### 2.5 布局层级（Layout Hierarchy）

CC 的每个分区遵循一致的布局模式：
```
[页面标题] [核心问题一句话]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[主状态区：1-2张大卡片，回答核心问题]
[次级区域：4-6张小卡片，补充细节]
[详情区域：可展开的表格/列表]
```

**设计要点**：三层结构让用户知道"我在哪个信息层级"，不会迷失在细节里。

---

## 三、我们控制台的 8 个分区：具体 UI 改进建议

> 格式说明：每条建议包含「现状」「问题」「建议」三部分，力求 Lux/Ezreal 可直接执行。

---

### 🏠 Overview（总览）

**建议 1：添加系统健康状态卡片（最高优先级）**

- **现状**：首屏没有一个地方直接回答"系统现在正常吗"
- **问题**：用户需要逐个检查 session 和 log，5 分钟才能得到 30 秒就能看懂的信息
- **建议**：首屏添加一张横向宽卡，显示 3 个指标：
  - 系统状态（绿色"正常运行" / 黄色"有待处理事项" / 红色"有任务卡住"）
  - 活跃 agent 数（如 `3 个执行中 / 5 个待命`）
  - 今日 token 消耗（如 `已用 42% 日额度`）
- **Lux 参考**：用语义颜色系统，宽卡背景色随状态变化；Ezreal 可每 30 秒轮询更新

**建议 2：把"待处理事项"独立成列表区块**

- **现状**：需要人工干预的事项（审批、卡住的任务）混在各分区里，没有汇总
- **问题**：用户每次都要去每个分区找"今天有什么需要我处理"
- **建议**：Overview 页下方设一个"待你处理"区块，直接列出：
  - 等待审批的任务（含 agent 名 + 任务摘要）
  - 卡住超过 30 分钟的 agent（含卡住原因简描）
  - 接近上下文上限的 session（超过 85%）
- **Lux 参考**：每一行都是可点击行，点击展开行内详情（不跳转），行左侧用颜色条区分优先级

---

### 👥 Agents（员工/Agent 管理）

**建议 1：卡片角标替代状态文字列**

- **现状**：agent 列表用表格行或文字标签显示状态，需要逐行阅读
- **问题**：8 个 agent 的状态扫描需要 10-15 秒；文字颜色区分不够直观
- **建议**：改为卡片网格布局（每行 3-4 张），每张卡片：
  - 左上角：彩色圆点（绿=执行中，黄=待命，红=卡住，灰=离线）
  - 卡片中央：agent 名称（大字）+ 当前任务摘要（小字，单行截断）
  - 右下角：最后活跃时间（如 "3 分钟前"）
- **Lux 参考**：使用 CC Staff 页的角标设计模式，避免纯文字列表

**建议 2：卡片支持右键或悬停展开操作菜单**

- **现状**：对某个 agent 做操作（查看 log、重启、暂停）需要点进详情页
- **问题**：多个 agent 需要批量检查时，反复进出详情页效率低
- **建议**：悬停卡片时显示 3 个快捷操作按钮（不影响布局）：
  - 查看最近 log（行内展开面板）
  - 查看当前任务（行内展开）
  - 复制 agent ID（Toast 确认）
- **Lux 参考**：悬停显示操作栏，悬停离开后消失，不占用常驻空间

---

### 📋 Tasks（任务）

**建议 1：区分"已规划"和"有执行证据"的任务**

- **现状**：任务列表把所有任务放在一起，没有区分规划中/执行中/已完成
- **问题**：用户不清楚哪些是实际在跑的任务，哪些只是写在列表里的计划
- **建议**：任务列表左侧添加一个垂直色条作为状态指示：
  - 蓝色 = 计划中（只是看板条目，没有 session 对应）
  - 橙色 = 执行中（有活跃 session）
  - 绿色 = 已完成（有完成证据）
  - 红色 = 已卡住（执行超时或有错误）
  - 灰色 = 已暂停/取消
- **Lux 参考**：左侧 4px 色条 + 行内状态芯片（文字 + 颜色），双重指示

**建议 2：执行链路可视化（小型流向图）**

- **现状**：多 agent 协作任务的流转路径只有文字记录
- **问题**：看不出"这个任务在谁手里、下一步是什么"
- **建议**：对涉及多个 agent 的任务，在任务详情（行内展开）里显示一个简单的水平流向图：
  ```
  [艾克 ✓] → [贾克斯 ✓] → [伊泽瑞尔 ⏳] → [加里奥 …]
  ```
  - 已完成的节点：绿色 + 勾
  - 当前节点：橙色 + 动态转圈
  - 待执行节点：灰色 + 虚线
- **Lux 参考**：节点大小一致，节点间用箭头线连接，整体宽度适配详情面板宽度

---

### 📊 Usage（用量）

**建议 1：进度环替代原始数字**

- **现状**：用量数据以数字或百分比文字显示
- **问题**：看到"123,456 tokens"不如看到"已用 62% 今日额度"直观；没有趋势感
- **建议**：今日用量改为环形进度条，环内显示百分比，环下显示"已用 XX,XXX / 总额度 XX,XXX"：
  - 0-60%：绿色环
  - 60-80%：黄色环
  - 80-100%：红色环（附 Toast 警告）
- **Lux 参考**：环形进度条直径约 80px，配合下方 3 天趋势小折线图

**建议 2：上下文压力列表**

- **现状**：session 的上下文占用情况没有可见性
- **问题**：任务跑到一半突然质量下降或失败，用户无法预见
- **建议**：Usage 页底部（或右侧栏）加"上下文压力"小列表，显示前 5 个上下文占用最高的 session：
  ```
  agent-backend-leona  ▓▓▓▓▓▓▓░░░  74%  [查看]
  agent-frontend-ezreal ▓▓▓▓░░░░░░  45%  [查看]
  ```
  - 超过 85% 时行背景变黄并出现图标警告
- **Lux 参考**：小横向进度条（bar），文字 + 进度条 + 百分比 + 操作链接四格对齐

---

### 🤝 Collaboration（协作）

**建议 1：时间轴式接力记录**

- **现状**：agent 间的任务交接记录较为平铺
- **问题**：难以一眼看出"任务在哪个时间点从谁传给了谁"
- **建议**：在协作页使用垂直时间轴布局：
  - 每个节点 = 一次交接（时间 + 发起方 + 接收方 + 交接摘要）
  - 当前持有方高亮（橙色边框 + 闪烁圆点）
  - 节点间用线连接，线上显示耗时
- **Lux 参考**：参考 Linear 或 GitHub PR 时间轴的视觉风格，左侧固定竖线，右侧卡片展开

**建议 2：跨 session 消息高亮显示**

- **现状**：跨 session 的消息记录混在普通日志里
- **问题**：看不出哪些是 agent 间的"正式交接消息"，哪些是内部 log
- **建议**：对跨 session 消息（sessions_send / inter-session message）添加特殊样式：
  - 左侧添加紫色标记条（区别于普通 log 的灰色）
  - 消息头部显示发送方 agent 名 + 接收方 agent 名
  - 折叠内容，点击展开完整消息体
- **Lux 参考**：用紫色系区分"正式 agent 通信"，与绿/橙/红的状态色不冲突

---

### 🧠 Memory（记忆）

**建议 1：记忆健康状态卡片**

- **现状**：记忆文件以列表形式展示，没有健康状态信息
- **问题**：用户不知道哪个 agent 的记忆是正常的、哪个可能已过期或有问题
- **建议**：在记忆分区顶部添加一行"记忆健康摘要"：
  - 每个 agent 对应一个小胶囊标签：绿（正常可用）/ 黄（需检查）/ 红（不可搜索）
  - 点击胶囊直接跳到该 agent 的记忆详情
- **Lux 参考**：胶囊标签高度约 24px，圆角，颜色 + agent 名简称 + 状态词

**建议 2：记忆文件编辑态区分**

- **现状**：记忆文件浏览和编辑状态区分不明显
- **问题**：用户误操作修改了记忆文件，难以察觉
- **建议**：只读状态下：右上角显示"只读"徽章 + 浅灰背景；进入编辑时：背景变白 + 顶部出现橙色编辑提示条 + 保存/取消按钮
- **Lux 参考**：编辑提示条高度 32px，橙色背景，白色文字"编辑中 — 未保存"

---

### 📄 Documents（文档）

**建议 1：文件修改状态徽章**

- **现状**：文档列表没有"最近修改"的显著标记
- **问题**：多个 agent 都在写文档时，用户不知道哪些是最新的、哪些刚被修改过
- **建议**：在文档列表每行右侧添加两种徽章：
  - "NEW"（绿色）= 今天新建
  - "UPDATED"（蓝色）= 24 小时内修改
  - 超过 7 天未修改的不显示徽章（减少视觉噪音）
- **Lux 参考**：徽章宽度约 60px，胶囊形状，放在文件名右侧、时间戳左侧

**建议 2：文件归属角标**

- **现状**：文档不明确属于哪个 agent
- **问题**：共享文档和 agent 专属文档混在一起，难以快速判断归属
- **建议**：每个文件行左侧添加小圆形角标，用 agent 缩写（如 "EZ" "LX"），鼠标悬停显示完整 agent 名；共享文档用 "SH" + 灰色角标
- **Lux 参考**：圆形角标直径 20px，颜色和 Agents 分区的颜色系统保持一致

---

### ⚙️ Settings（设置）

**建议 1：接线状态可视化检查清单**

- **现状**：连接配置以文字列表展示，不直观
- **问题**：用户不清楚哪些连接"已完全接好"，哪些"只是部分工作"，哪些"完全没接"
- **建议**：采用检查清单（Checklist）布局：
  ```
  ✅ OpenClaw Gateway          已连接  (127.0.0.1:4310)
  ✅ OpenClaw Home              已找到  (~/.openclaw)
  ⚠️ Usage 数据                部分可见 (缺少订阅快照)
  ❌ Codex Home                未找到   — 用量分区将降级
  ```
  - 每行三栏：图标+名称 / 状态词 / 详情或操作链接
- **Lux 参考**：图标使用语义颜色（✅绿、⚠️橙、❌红），行间距 12px，整体比表格轻

**建议 2：安全风险人话化卡片**

- **现状**：安全配置以配置项形式展示（如 `IMPORT_MUTATION_ENABLED=false`）
- **问题**：非技术用户看到配置项无法判断风险等级和是否需要操作
- **建议**：在 Settings 页顶部添加"当前安全状态"卡片（仿 CC 的 Security risk summary）：
  - 标题：🔒 当前安全状态（绿/黄/红）
  - 正文：`目前处于安全模式。所有写操作已关闭。你不会意外修改系统数据。`
  - 风险项（如有）：以黄色/红色条目列出，每条附"为什么有风险" + "如何关闭"
- **Lux 参考**：卡片整体绿色调（安全时），黄色调（有警告），标题和正文层次分明

## 四、重点分区强化建议（Overview / Agents / Tasks 专项）

### Overview 强化路线图（推荐实现顺序）

| 优先级 | 功能 | 预期效果 |
|--------|------|--------|
| P0 | 系统健康状态宽卡（颜色 + 数字 + 一句话） | 让"开机检查"从 5 分钟 → 30 秒 |

| P2 | 快速操作入口（右上角：全局搜索 ⌘K / 快捷筛选） | 减少为了找一个 agent 要跑 3 个页面的问题 |

### Agents 强化路线图

| 优先级 | 功能 | 预期效果 |
|--------|------|--------|
| P0 | 卡片网格布局 + 状态角标 | 8 个 agent 状态扫描从 15 秒 → 3 秒 |
| P0 | 悬停显示快捷操作 | 减少"进入详情 → 操作 → 返回列表"的往返次数 |
| P1 | 按状态分组（执行中 / 待命 / 离线） | 让用户一眼定位异常 agent |

### Tasks 强化路线图

| 优先级 | 功能 | 预期效果 |
|--------|------|--------|
| P0 | 左侧色条区分任务状态 | 让"计划中"和"执行中"不再混淆 |
| P0 | 执行链路水平流向图 | 多 agent 协作任务的位置和状态一眼可见 |
| P1 | 任务卡片右侧显示"持有方 agent"头像/缩写 | 快速定位"谁在负责这件事" |
| P2 | 任务超时标红（超过设定时长无进展） | 主动提醒，减少人工轮询 |

## 五、给 Lux 的视觉设计指南

### 颜色系统建议（基于 CC 语义颜色模式）

```
系统绿（Active/OK）：   #22C55E（Tailwind green-500）
警告黄（Warning/Busy）：#EAB308（Tailwind yellow-500）
风险橙（Risk/Pending）：#F97316（Tailwind orange-500）
错误红（Error/Stuck）：  #EF4444（Tailwind red-500）
中性灰（Queued/Off）：   #9CA3AF（Tailwind gray-400）
强调蓝（Info/Link）：    #3B82F6（Tailwind blue-500）
协作紫（Cross-session）：#8B5CF6（Tailwind violet-500）
```

### 卡片设计基准

- 圆角：`rounded-xl`（12px）
- 内边距：`p-4`（16px）
- 卡片投影：`shadow-sm`，悬停时 `shadow-md`
- 状态色条（左侧竖条）：宽度 4px，高度 100%，颜色随状态变化
- 字号层级：标题 14px/灰色 → 主值 24px/黑色 → 辅助说明 12px/浅灰

### 布局节奏

- Overview：2 列主卡 + 4-6 格次级卡，移动端 1 列
- Agents：3-4 列卡片网格，每格固定高度
- Tasks：单列列表 + 行内展开详情面板
- 统一左侧 240px 固定导航栏 + 顶部 56px 状态栏

## 六、给 Ezreal 的实现优先级总结

### Phase 1（MVP 必做，影响每日使用）

1. **Overview 健康状态宽卡**：轮询接口 + 颜色状态机 + 三核心指标
2. **Agents 卡片网格 + 状态角标**：替换表格布局，角标逻辑绑定 agent session 状态
3. **Tasks 左侧色条 + 状态分组**：CSS 左 border-left 实现，分组由前端状态过滤

### Phase 2（高频用户会感知的改善）

4. **Agents 悬停操作菜单**：Hover 显示 3 个快捷操作，不跳转，行内展开
5. **Tasks 执行链流向图**：水平 flex + 圆形节点 + 颜色逻辑
6. **全局搜索入口（⌘K）**：弹层 + 模糊匹配已加载的 agent 名 / 任务名

### Phase 3（体验完善）

7. **Usage 环形进度条 + 上下文压力列表**
8. **Settings 接线状态检查清单**
9. **所有分区的骨架屏（Skeleton loading）** — 空状态不显示白板

## 总结

CC 的核心 UI 哲学是：**每一格展示的信息，都对应一个用户层面的问题，不是系统层面的参数。**

我们控制台改进的方向，不是加更多功能，而是：

1. **把状态翻译成运营语言**（颜色 + 数字 + 动词短语）
2. **让信息有层次**（一眼/扫描/深入三层，不要全部平铺）
3. **空状态有结构**（骨架屏 + 降级说明，不是白板）
4. **操作就近可达**（悬停操作、行内展开，减少页面跳转）
5. **异常主动浮出**（不要让用户自己找问题，问题要找用户）
