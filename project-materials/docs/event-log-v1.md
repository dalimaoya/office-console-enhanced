## 1. 设计原则

### 1.1 为什么必须是只追加（append-only）

事件日志在本项目里不是普通调试输出，而是控制台判断“系统是否在按预期运行”的事实基础。因此一期必须坚持 **只追加、不修改、不回写历史**。

原因有四个：

1. **保证可追溯性**  
   一条关键操作一旦发生，就必须留下不可歧义的时间序列记录。后续排查“谁在什么时间做了什么”时，不能因为覆盖写入而丢失上下文。

2. **降低实现复杂度**  
   一期先追求最低可用，不做复杂存储引擎。append-only 文件写入模型最简单、最稳定，也最符合当前单项目、轻量并发的现实。

3. **避免状态被日志反向污染**  
   日志的职责是记录事实，不负责修正事实。若允许修改旧日志，日志本身就会变成“第二状态源”，增加认知冲突。

4. **方便控制台按时间消费**  
   控制台核心需求是最近事件、最近异常、某角色最近动作。追加式日志天然适合按时间倒序读取、尾部读取和增量拉取。

### 1.2 日志和数据库的区别

事件日志不是数据库，二者职责必须分清：

| 维度 | 事件日志 | 数据库 / 注册表 / 状态存储 |
|------|----------|---------------------------|
| 目的 | 记录“发生过什么” | 维护“当前是什么” |
| 写入方式 | 只追加 | 可新增、更新、归档 |
| 数据形态 | 时间序列事实流 | 当前态、索引、关系 |
| 查询重点 | 时间、事件类型、来源、对象 | 主键、状态、聚合结果 |
| 正确性要求 | 完整保留过程 | 保持当前值准确 |
| 控制台用途 | 诊断、审计、回放 | 状态展示、聚合、检索 |

一句话：**日志是过程证据，数据库是当前快照。**

在本项目一期中：
- `registry/objects.md` 负责对象登记，是“对象索引”；
- 事件日志负责记录对象和角色的行为轨迹，是“时间证据”；
- 控制台同时消费二者，但不能拿日志直接替代注册表，也不能要求注册表承担完整审计职责。

### 1.3 哪些事件必须记录，哪些不用记录

#### 必须记录（一期 P0）

凡是会影响控制台判断、系统诊断、责任归属、项目推进的事件，必须记录：

1. **系统生命周期事件**
   - 服务启动 / 停止 / 重启
   - 健康检查通过 / 失败
   - 关键依赖连接异常 / 恢复

2. **角色推进事件**
   - 任务开始
   - 任务完成
   - 任务阻塞
   - 向上升级 / 请求决策

3. **对象关键变更事件**
   - 文件创建
   - 文档/对象状态变更
   - 正式交接发出
   - 对象被纳入注册表或引用链路

4. **安全与执行边界事件**
   - 只读模式触发
   - DryRun 执行
   - 操作被拒绝 / 被拦截
   - 权限不足导致失败

5. **关键失败事件**
   - 写入失败
   - 解析失败
   - 接口调用失败
   - 关键任务超时

#### 不必记录（一期明确不做）

以下内容不进入正式事件日志，避免噪音淹没信号：

1. 高频、无状态意义的调试输出  
   例如普通 console 输出、轮询心跳明细、每一次内部变量变化。

2. 可重复推导的中间过程  
   例如“读取了某文件第 3 次”“发起第 2 次轻量重试”这类不影响结论的细颗粒操作。

3. 纯展示层行为  
   例如控制台前端 tab 切换、列表展开/收起，不属于后端底座日志范围。

4. 无结果的思考过程  
   模型内部推理、草稿阶段未产生外部影响的临时动作，不记为正式事件。

原则是：**只记关键事实，不记一切细节。** 一期宁可少而准，不要多而乱。

## 2. 日志格式设计

### 2.1 单条日志记录字段约定

一期先定义字段契约，不绑定具体代码实现。

#### 必填字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | string | 事件发生时间，ISO 8601 UTC，如 `2026-03-19T15:40:21Z` |
| `event_type` | string | 事件类型，采用分层命名，如 `system.start` |
| `source_role` | string | 来源角色或来源组件，如 `backend-leona` / `system-gateway` |
| `message` | string | 面向人类可读的事件描述，要求一句话说清发生了什么 |
| `object_id` | string | 关联对象 ID，必须直接引用 `file-schema-v1.md` 注册表中的对象 ID 原值（如 `artifact-20260319-01-file-schema`）；如无已注册对象，可使用系统级 ID（如 `system-gateway`、`project-office-console-enhanced`），统一采用连字符分隔，不使用冒号 |

#### 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `context` | object | 触发上下文，如请求来源、任务 ID、项目名、会话 ID |
| `error` | object | 错误信息，如错误码、错误摘要、原始异常字符串 |
| `before_state` | object | 变更前状态快照，仅用于关键状态切换 |
| `after_state` | object | 变更后状态快照，仅用于关键状态切换 |
| `severity` | string | 严重级别，建议 `info` / `warn` / `error` |
| `trace_id` | string | 一次链路追踪 ID，用于串联同一任务内多个事件 |
| `project` | string | 所属项目标识，建议始终携带，如 `office-console-enhanced` |
| `actor_type` | string | 来源类型，如 `agent` / `system` / `user` |

### 2.2 字段使用原则

1. **`message` 必须可读**  
   控制台不能要求用户先理解 JSON 结构才能看懂日志。

2. **`event_type` 必须可枚举、可过滤**  
   事件类型不写自然语言句子，必须是稳定字符串，便于 API 过滤。

3. **`object_id` 必须有锚点**  
   即使不是文档对象，也要能挂到项目对象或系统对象上，不能出现大量“无归属事件”。

4. **`source_role` 与 `owner` 分工必须明确**  
   `source_role` 只表示事件来源角色/组件；对象归属责任人统一由 `file-schema-v1.md` 中的 `owner` 字段承担，二者不得混写。

5. **`before_state` / `after_state` 只用于关键状态切换**  
   例如 `active -> blocked`、`draft -> done`。不要求所有事件都带，避免日志膨胀。

6. **错误信息结构化**  
   `error` 不只是一段字符串，至少应允许包含 `code`、`summary`、`detail` 三层信息。  

### 2.3 推荐编码格式

一期推荐使用 **NDJSON（Newline Delimited JSON）**。

原因：

1. **天然适合 append-only**：一行一条，追加简单；
2. **人和程序都能处理**：文本可读，grep/tail 可用，程序解析成本低；
3. **支持流式读取**：控制台读取最近 N 条和增量刷新都方便；
4. **故障隔离更好**：单条损坏不影响整文件整体结构。

#### 结论

- **存储格式**：NDJSON
- **单条记录编码**：JSON Object
- **不推荐一期使用**：
  - 单个大 JSON 数组（追加不方便）
  - 二进制日志格式（过重）
  - 纯文本非结构化日志（控制台难过滤和聚合）

#### 示例（示意）

```json
{"timestamp":"2026-03-19T15:40:21Z","event_type":"role.task_started","source_role":"backend-leona","message":"开始输出《事件日志底座方案 V1》","object_id":"artifact-20260319-02-event-log","project":"office-console-enhanced","severity":"info","context":{"brief":"一期最小推进","assignee":"backend-leona"}}
```

## 3. 事件类型分类

一期先收敛为四大类，并采用 `domain.action` 命名。

### 3.1 系统事件

用于描述运行环境和服务本身的可用性。

| 事件类型 | 含义 |
|----------|------|
| `system.start` | 系统/服务启动 |
| `system.stop` | 系统/服务停止 |
| `system.restart` | 系统/服务重启 |
| `system.healthcheck_passed` | 健康检查通过 |
| `system.healthcheck_failed` | 健康检查失败 |
| `system.dependency_error` | 外部依赖异常 |
| `system.dependency_recovered` | 外部依赖恢复 |

### 3.2 角色事件

用于描述角色在项目推进中的动作。

| 事件类型 | 含义 |
|----------|------|
| `role.task_started` | 角色开始处理任务 |
| `role.task_completed` | 角色完成任务 |
| `role.task_blocked` | 角色出现阻塞 |
| `role.task_escalated` | 角色向上升级 / 请求决策 |
| `role.handoff_sent` | 角色发出正式交接 |
| `role.handoff_received` | 角色确认接收交接（一期可选） |

### 3.3 对象事件

用于描述文件、文档、注册对象等项目对象的关键变化。

| 事件类型 | 含义 |
|----------|------|
| `object.created` | 对象创建 |
| `object.updated` | 对象内容被更新 |
| `object.status_changed` | 对象状态变化 |
| `object.registered` | 对象进入注册表 |
| `object.handoff_linked` | 对象被纳入交接链路 |
| `object.archived` | 对象归档 |

说明：一期虽然强调 append-only，但对象本身仍可能更新；日志记录的是“发生了更新事件”，不是覆写历史。

### 3.4 安全事件

用于记录执行边界、生效模式和被拒绝操作。

| 事件类型 | 含义 |
|----------|------|
| `security.readonly_enabled` | 只读模式生效 |
| `security.dryrun_executed` | DryRun 执行 |
| `security.operation_denied` | 操作被拒绝 |
| `security.permission_missing` | 权限不足 |
| `security.write_blocked` | 写入被安全策略拦截 |

### 3.5 一期扩展建议：失败事件统一级别

对于失败类事件，不额外单独拆出第五大类，先通过：
- `event_type` 表达业务语义；
- `severity=error`；
- `error` 字段补充错误上下文。

例如：
- `object.updated` + `severity=error` + `error.code=WRITE_FAILED`
- `system.healthcheck_failed` + `severity=error`

这样一期分类更稳，不会过早把枚举炸开。

## 4. 存储方案

### 4.1 一期建议采用什么存储

一期建议采用：**本地文件 + NDJSON + append-only 写入**。

这是当前阶段性价比最高的方案，原因如下：

1. **与文件驱动架构一致**  
   项目本身已明确坚持文件驱动、对象驱动，日志继续走文件最自然。

2. **实现最小、故障面最小**  
   不引入数据库、不引入消息队列、不引入额外服务，便于快速落地。

3. **满足控制台一期诉求**  
   一期控制台只需要最近事件、按类型/角色筛选，不需要复杂 OLAP 能力。

4. **便于后续迁移**  
   未来若接入 SQLite / 专用日志索引，也可以保留 NDJSON 作为事实源，再做异步导入。

#### 一期明确不建议

- 只存在内存：进程重启丢数据，不可接受；
- 直接上数据库：对当前阶段过重；
- 分散写多个无规范文本文件：后续无法统一消费。

### 4.2 文件路径规范

建议在项目目录下新增日志目录：

```text
/root/.openclaw/workspace/projects/office-console-enhanced/logs/events/
```

文件命名建议：

```text
events-YYYY-MM-DD.ndjson
```

示例：

```text
/root/.openclaw/workspace/projects/office-console-enhanced/logs/events/events-2026-03-19.ndjson
```

#### 路径规范说明

- `logs/`：项目级日志根目录
- `events/`：事件日志专属目录，避免与未来诊断日志、访问日志混写
- `events-YYYY-MM-DD.ndjson`：按日自然切分，便于控制台读取最近日志和人工排查

### 4.3 轮转策略

一期建议：**按日轮转 + 单文件大小兜底**。

#### 主策略：按日轮转

优点：
- 命名简单；
- 查询“今天发生了什么”最直观；
- 控制台读取最近 N 条时通常只需读当天文件，必要时回溯前一天。

#### 兜底策略：按大小补充切分（可选）

若单日日志异常膨胀，可加保底阈值，例如 20MB 或 50MB 后切分为：

```text
events-2026-03-19-01.ndjson
events-2026-03-19-02.ndjson
```

但一期先不强制实现大小切分，只在规范里预留。

#### 一期结论

- **默认策略**：按日轮转
- **是否不轮转**：不建议；单文件会越来越难维护
- **是否按小时轮转**：过细，不符合一期规模

## 5. 控制台消费方式

### 5.1 控制台如何读取日志

一期控制台不直接读磁盘文件，建议统一通过后端 API 读取。原因：

1. 前端无需理解文件结构；
2. 后端可统一做筛选、排序、截断；
3. 后续从文件迁移到别的存储时，前端接口不必重写。

因此，一期 API 只需要提供“事件查询接口”，由后端负责：
- 从最近日志文件读取；
- 解析 NDJSON；
- 按条件过滤；
- 返回最近结果。

### 5.2 最小可用接口（一期）

#### 1）查询最近 N 条事件

**接口语义**：
`GET /api/events?limit=50`

**用途**：
控制台首页的“最近事件”时间线。

**返回内容建议**：
- 按时间倒序返回；
- 默认最近 50 条；
- 返回标准日志字段的精简视图。

#### 2）按事件类型过滤

**接口语义**：
`GET /api/events?eventType=role.task_blocked&limit=20`

或支持多值：
`GET /api/events?eventType=system.healthcheck_failed,security.operation_denied`

**用途**：
快速查看错误、阻塞、安全边界事件。

#### 3）按角色过滤

**接口语义**：
`GET /api/events?sourceRole=backend-leona&limit=20`

**用途**：
查看某角色最近动作，辅助判断是否卡住、是否完成阶段工作。

### 5.3 建议的补充查询参数（一期可预留）

虽然题目只要求最小接口，但建议参数设计先留扩展口：

| 参数 | 说明 |
|------|------|
| `limit` | 返回条数，默认 50，上限建议 200 |
| `eventType` | 单个或多个事件类型 |
| `sourceRole` | 来源角色 |
| `objectId` | 关联对象 ID |
| `severity` | 严重级别 |
| `before` | 查询某时间点之前的事件，便于翻页 |

### 5.4 一期需要做什么

一期后端只做以下能力：

1. 统一日志字段契约；
2. 统一落盘到项目内 `logs/events/`；
3. 提供最近 N 条查询；
4. 提供按 `event_type` 过滤；
5. 提供按 `source_role` 过滤；
6. 保证结果按时间倒序；
7. 控制台展示基础时间线。

### 5.5 二期再做什么

二期再补以下能力，不挤进一期：

1. **增量订阅 / SSE 推送**  
   让控制台实时刷新，而不是定时拉取。

2. **组合筛选**  
   支持事件类型 + 角色 + 时间范围 + 严重级别联合过滤。

3. **对象级追踪视图**  
   例如查看某个 brief / handoff / artifact 的完整事件链。

4. **异常摘要与聚合统计**  
   自动聚合最近 24 小时错误、阻塞、拒绝操作次数。

5. **日志索引或轻量缓存**  
   当日志规模上升时，补充读取性能优化。

6. **与对象注册表关联展示**  
   在控制台上把“当前对象状态”和“最近事件轨迹”联动起来。

一期目标不是“日志平台”，而是**先把控制台诊断所需的最低可观测性建立起来**。

## 6. 复用性说明

### 6.1 当前项目专属部分（一期保持）

以下内容先保留项目专属性，不急于抽象：

1. **项目名与目录路径**  
   如 `office-console-enhanced`、当前 `projects/office-console-enhanced/` 路径。

2. **角色命名体系**  
   如 `backend-leona`、`orchestrator-teemo`，属于当前团队配置。

3. **控制台当前关注的事件优先级**  
   例如特别强调阻塞、升级、交接，这与当前项目推进方式强相关。

4. **系统对象命名约定**  
   如 `project-office-console-enhanced`、`system-gateway`，统一使用连字符分隔格式（与 `file-schema-v1.md` 对象 ID 风格一致），可以先按本项目写死后再抽象。

### 6.2 可被通用化的部分

以下内容建议从一开始就按通用底座思路设计：

1. **日志字段契约**  
   `timestamp / event_type / source_role / message / object_id` 这套核心字段，对任何 OpenClaw 项目都成立。

2. **事件分类模型**  
   `system / role / object / security` 四大类，适合大多数多角色协作项目。

3. **NDJSON + append-only 存储模式**  
   轻量、通用、可迁移，适合作为 OpenClaw 项目默认方案。

4. **事件查询 API 语义**  
   最近 N 条、按类型过滤、按角色过滤，是所有控制台的基础查询能力。

5. **对象 ID 关联方式**  
   与 `file-schema-v1.md` 中的对象注册表对齐后，任何项目都能把日志挂到自己的对象链路上。

### 6.3 通用化边界建议

一期不要追求“所有项目统一事件枚举全集”，只需要统一：
- 核心字段；
- 核心分类；
- 查询语义；
- 文件组织方式。

而具体项目可以扩展自己的事件类型，例如：
- `workflow.approval_requested`
- `delivery.package_published`
- `research.source_verified`

也就是说：**底座统一骨架，项目扩展业务事件。**

## 7. 一期落地结论

### 7.1 最终建议

一期事件日志底座采用以下组合：

- **设计原则**：只追加、不修改
- **数据定位**：日志记录过程事实，注册表/状态存储记录当前状态
- **编码格式**：NDJSON
- **存储方式**：项目本地文件
- **目录规范**：`logs/events/events-YYYY-MM-DD.ndjson`
- **事件分类**：`system / role / object / security`
- **控制台接口**：最近 N 条、按事件类型过滤、按角色过滤

### 7.2 一期不做

为避免范围失控，以下能力明确不纳入一期：

- 不引入数据库日志存储
- 不做复杂全文检索
- 不做前端直连文件
- 不做自动根因分析
- 不做多项目统一日志中心
- 不做高频调试级日志采集

### 7.3 与上游文档的关系

- 与 `product-scope-v1.md` 一致：本方案服务于“运行控制台”的最低可观测性需求；
- 与 `file-schema-v1.md` 一致：日志中的 `object_id` 应优先引用已定义对象 ID，形成对象—事件关联；
- 与 `2026-03-19-project-adjustment-v1.md` 一致：日志作为一期 P0 底座能力，先解决“可诊断、可追溯、可感知”，再谈更复杂的项目治理功能。
