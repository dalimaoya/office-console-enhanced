- **对标对象**：https://github.com/TianyiDataScience/openclaw-control-center
- **评审时间**：2026-03-19
- **参与角色**：艾克（产品）、贾克斯（架构）、拉克丝（界面）、杰斯（办公视角）
- **收口**：提莫（orchestrator-teemo）

## 一、对标对象概览

openclaw-control-center 是一个面向**非技术运营用户**的 OpenClaw 本地可观测性控制中心。

**核心定位**："把黑箱变成看得清、信得过、控得住的控制台。"  
**第一设计原则**："Built for non-technical operators who need observability and certainty, not raw backend payloads."

**技术特征**：
- 技术栈：TypeScript + 原生 `node:http`（无框架依赖），CommonJS
- 零运行时依赖（仅 devDependencies）
- SSR（server.ts 直出 18000 行 HTML 字符串）
- 状态存储：本地 JSON 文件（`runtime/*.json`）
- 架构：`adapters/clients/contracts/mappers/runtime/ui` 六层分离，`runtime/` 下 40+ 独立功能模块
- 轮询模式（POLLING_INTERVALS_MS 配置），无 SSE

**页面分区**：
Overview / Usage / Staff / Collaboration / Tasks / Documents / Memory / Settings

**差异化能力**（我们没有的）：
历史回放（replay-index）、导入导出快照、通知中心+ACK/Snooze、审批动作服务（默认关闭）、操作审计日志、预算治理体系（双层）、差量摘要、任务心跳、安全门控五层递进、双语支持

## 二、对方优于我们的点

按影响程度从高到低排列：

### 2.1 历史回放 / 时间线系统（差距最大）
对方有完整的 `timeline.log` + `replay-index.ts` + `digests/` + `export-snapshots/` 体系，可按时间段追溯历史执行记录和摘要，是"行车记录仪"级别的可追溯能力。

我们现状：只有实时状态展示，无任何历史追溯入口。出问题后无法复盘。

### 2.2 安全门控分层
对方实现五层递进式安全门控：ReadOnly → LocalTokenAuth → ApprovalActions → ImportMutation → DryRun。每个可写端点独立受控，默认全部关闭，写操作全部留有审计日志。

我们现状：有基础 helmet/cors 和 readonly 模式，但缺少操作粒度门控和审计日志，写操作没有留痕。

### 2.3 "把技术状态翻译成业务信号"的设计语言
对方每页都有 "Best when you want to..." 的场景说明；Connection Health 用业务语言描述连接问题；Security risk summary 翻译配置参数为用户可读文本；Staff 页区分"真正在执行"与"只是排队"。

我们现状：把技术状态直接对外暴露（如 `READONLY_MODE=true`）。首屏"等待检查"是体验杀手，飞书未配置时空白无引导。

### 2.4 持久化通知中心 + ACK/Snooze 机制
对方有 `notification-center.ts`，通知可 ACK（已读）或 Snooze（延后处理），形成"未处理事项收件箱"。

我们现状：Toast 消失即丢，无持久化通知中心，无"还有什么没处理"的聚合感知。

### 2.5 预算治理体系
对方有 `budget-governance.ts`（运行层）+ `budget-policy.ts`（策略层），预算治理涵盖 agent/project/task 三级 scope，有 warn/over 阈值，自动预警成本风险。

我们现状：有用量 KPI 展示，但无预算策略、无预警阈值、无自动保护。

### 2.6 右侧 Inspector 侧边栏（三栏布局）
对方三栏布局（232px 导航 + 主内容 + 300px Inspector），点击任何条目在右侧面板展示详情，不换页不弹 Modal，是控制台类产品"上下文不中断"的核心交互范式。

我们现状：两栏布局，详情通过路由切换或内联切换，操作流程有断点。

### 2.7 页面视觉质感（玻璃拟态 + 暗色主题）
对方使用 `backdrop-filter: blur(22-56px)` + 多层 `radial-gradient` 光晕，light/dark 双套完整 CSS 变量体系，视觉质感有档次差距。

我们现状：已有暗色配色体系，但无 backdrop-filter 玻璃效果，无明暗主题切换。

### 2.8 增量轮询策略
对方 `OpenClawReadonlyAdapter` 区分 active/inactive session 做差异化轮询，缓存上一轮 active keys，避免全量重查。

我们现状：固定 TTL 缓存，不区分数据活跃度，存在不必要的全量轮询开销。

## 三、我们优于对方的点

### 3.1 飞书深度集成（核心架构优势）
我们以飞书为主入口，AI 团队进展直接推送到办公用户的工作群。对方是纯本地 Web 面板，用户必须主动打开浏览器。这是"信息推到用户面前"vs"用户主动去找信息"的本质区别。

### 3.2 SSE 实时推送
我们有完整的 SSE + 指数退避重连（1→2→4→8→30s）+ 页面感知自动重连，实时性可感知。对方是轮询模式（无服务端推送），实时性弱。

### 3.3 Agent 6 种精确状态 + 前端分区展示
working / idle / blocked / backlog / error / offline 六态精确区分，比对方 Staff 页的活跃/排队/阻塞更细。

### 3.4 写操作能力全面落地
我们有 Tasks POST/PATCH、Memory 编辑、Settings dry-run 写操作，并有 Toast 反馈。对方的写能力被安全开关保护得较严，是"观测工具"定位的体现。

### 3.5 MVC 分层架构 + API 版本化
我们有明确的 Controller/Service/Adapter 三层，`/api/v1/*` 版本化 REST，中间件栈清晰，可维护性更好。对方 `ui/server.ts` 是导入 50+ 模块的单文件路由，分层边界模糊。

### 3.6 多角色 AI 团队 + 飞书多 bot 协作
我们有明确角色分工（Teemo/Ekko/Jax/Ezreal/Leona/Galio/Lux/Jayce），各角色有独立 bot/accountId，办公用户看到"艾克在处理需求"而非"agent-id-3 in task-8872"，认知锚点更强。

### 3.7 全站搜索（Cmd+K）
全局搜索 + 快捷键，按 agent/任务/会话分组，对方无此能力。

### 3.8 PM2 生产部署 + 一键安装
完整 PM2 ecosystem 配置，我们的部署便利性对技术用户更友好。

## 五、不建议照搬的点

| 点 | 原因 |
|----|------|
| **默认只读 / 关闭所有写操作** | 对方是"观测工具"定位，我们已明确为"可工作的工具"，写能力是核心差异化，照搬会倒退 |
| **原生 `node:http` 替代 Express** | 对方零依赖是安全考量，我们需要中间件栈和路由组织，手写 HTTP 处理器维护成本不合理 |
| **SSR 单文件路由（18000 行 HTML）** | 对方强耦合导致代码不可维护，我们应保持前后端分离 |
| **本地 JSON 文件作状态存储** | 单用户本地场景可行，多用户/多实例存在并发冲突风险，不适合我们扩展 |
| **CommonJS 模块系统** | 对方是 `"type":"commonjs"`，我们已采用 ESM，是 Node.js 未来方向，不应倒退 |
| **URL 参数路由（?section=xxx）** | 对方因 SSR 必须如此，我们已用客户端路由，无需改动 |
| **双语 URL 参数切换** | 我们用户群是中文团队，双语是推广需求，当前不是用户痛点，投入产出不高 |
| **纯本地 Web 面板为主入口** | 我们以飞书为主入口是核心架构优势，不应退化为"用户主动打开浏览器"的模式 |
| **avatar-preferences / 用户偏好个性化** | 内部团队工具，个性化偏好价值有限，优先级极低 |

## 六、建议纳入我们项目的功能优先级

### P0（核心差距，建议下一迭代优先处理）

1. **操作审计日志**：所有 mutation 留痕，构建完整操作时间线。是合规基础。
2. **Connection Health 引导卡**：集成未配置时（飞书/Gateway）显示业务语言引导，而非空白。直接解决当前已知的严重 UX 问题。
3. **历史回放 / 时间线入口**：增加事件采集层，为后续回放能力打基础。这是从"实时快照工具"升级为"可追溯审计工具"的关键跃迁。
4. **安全门控扩展**：在现有 readonly 基础上，完善操作级 DryRun 门控，写操作默认需要确认。

### P1（体验提升，建议排入近期迭代）

5. **持久化通知中心 + ACK**：Toast 升级为可持久、可已读的通知收件箱。
6. **每页适用场景提示**：导航副标题加一行"本页最适合..."说明，降低认知成本。
7. **安全设置业务化描述**：技术参数翻译成用户可读语言。
8. **预算告警阈值**：在 Usage 基础上增加预警，接近上限主动标红。
9. **Inspector 侧边栏（三栏）**：不换页查看详情，减少操作流程断点。
10. **页面过渡动效**：panel-in 进场动效，消除切换跳变感。

### P2（品质提升，可选排入）

11. 玻璃拟态视觉效果（backdrop-filter + glass token）
12. 明暗主题切换（data-theme 机制）
13. 导入/导出快照
14. 差量摘要（文档/记忆变更 diff）
15. 增量轮询策略优化
16. 刷新状态保存（sessionStorage）
17. 全局状态条（Status Strip）

## 七、下一步建议动作

**当前状态**：已进入可用专业工具阶段，不适合大范围重构，适合按优先级逐步吸收。

**建议执行顺序**：
1. **先确认要不要进入 P0**：P0 的操作审计 + Connection Health 引导投入小、收益大，建议决策人确认后，安排雷欧娜+伊泽瑞尔一轮实现
2. **P1 中的非技术改进先行**：每页适用场景提示、安全设置业务化描述，只需改前端文案，不需要后端，伊泽瑞尔可快速完成
3. **Inspector 侧边栏和通知中心**是较大的架构改动，建议先出设计草稿，评估改动范围后再启动
4. **历史回放**需要新增事件采集层，是最大的工程投入，建议单独立项而不是附加在当前迭代中

**不建议立即做的**：对方技术栈相关的重构（零依赖/SSR/CommonJS）、双语支持、用户偏好个性化。
