## 四、project-memory skill 评估

### 4.1 这个 skill 是否需要任何现有角色配置文件做对应更新才能生效？

**结论：不需要。当前 skill 已可独立生效。**

#### 证据 1：skill 的生效条件是“使用时触发”，不是“角色配置预注册”
`~/.agents/skills/project-memory/SKILL.md` 明确写的是：
- 解析 project root
- 调脚本写入 `{project_root}/.project-memory/`
- 更新 index

它依赖的是：
1. skill 文件本身存在；
2. 调用时位于一个可解析 project root 的项目目录；
3. 相关脚本可执行。

不是依赖每个角色先改 AGENTS/IDENTITY/配置文件才能使用。

#### 证据 2：skill 明确强调“项目目录隔离”，不是 agent workspace 配置依赖
SKILL.md 中明确：
- `Never write to native memory dirs.`
- `Never read/write agent workspace from project-memory.`
- 存储目录是 `{project_root}/.project-memory/`

这说明它与各角色私有 workspace 配置**天然解耦**。

### 4.2 是否存在“为了团队稳定采用，建议更新配置”的运营性需求？

**有，但这是“建议”，不是“生效前置条件”。**

可选优化包括：
1. 在 orchestrator 或 specialist 的协作规范中补一句：阶段交付后可/应写入 project-memory；
2. 在 Getting Started 或项目协作规范里补一个使用范式；
3. 若希望强制落地，可在主控角色规则里加入“阶段完成后记录到 project-memory”。

但这些属于**流程增强**，不是 skill 技术生效所必需。
