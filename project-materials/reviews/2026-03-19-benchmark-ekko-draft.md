# 艾克产品层对标草稿

## 对方功能概览（我的理解）

openclaw-control-center 是一个面向**非技术运营用户**的 OpenClaw 本地可观测性控制中心，定位是"把黑箱变成看得清、信得过、控得住的控制台"。

### 核心页面分区（信息架构）

| 分区 | 核心价值 |
|------|----------|
| **Overview（总览）** | 系统整体健康态势、待处理事项、预算风险、一句话回答"现在正常吗" |
| **Usage（用量）** | 今日/7天/30天花费趋势、订阅窗口、上下文压力、连接状态 |
| **Staff（员工）** | 真实工作中 vs 只是排队——明确区分 active/queued |
| **Collaboration（协作）** | 父子会话接力 + 跨会话消息，独立页面可视化 |
| **Tasks（任务）** | 任务板 + 审批 + 执行链 + 运行证据一体化 |
| **Documents（文档）** | 源文件工作台，直接读写真实 markdown 文件 |
| **Memory（记忆）** | 记忆工作台，健康状态展示，范围跟随活跃 agent |
| **Settings（设置）** | 接线状态诊断 + 安全风险摘要 + 版本更新状态 |

### 后端能力亮点（从 src/runtime 推断）

- `replay-index.ts`：时间线回放 + 快照导出（Replay/历史追溯）
- `export-bundle.ts`：完整状态导出包（含 sessions、budget、tasks）
- `import-dry-run.ts` / `import-live.ts`：导入机制（带 dry-run 保护）
- `notification-center.ts`：通知中心 + ACK 机制（可 snooze）
- `approval-action-service.ts`：审批动作服务（硬开关保护，默认关闭）
- `operation-audit.ts`：操作审计（写操作留痕）
- `budget-governance.ts` / `budget-policy.ts`：预算治理体系（双层）
- `pixel-state.ts`：像素级状态感知（推断为前端状态快照）
- `avatar-preferences.ts`：用户偏好（头像/个性化）
- `diff-summary.ts`：差量摘要（变更追踪）
- `openclaw-cli-insights.ts`：CLI 洞察（非 UI 层命令感知）
- `ui-preferences.ts`：UI 偏好持久化
- `task-heartbeat.ts`：任务心跳（存活探测）
- `office-session-presence.ts`：办公场景会话在场感知

### 产品定位特点

- 明确的**安全优先**设计哲学：默认只读、默认 token 鉴权、高风险写操作全部有硬开关
- 面向**非技术运营用户**，翻译后端 payload 为人话
- **双语支持**（中文/英文），URL 参数切换 `?lang=zh`
- 自带**安装提示词**（INSTALL_PROMPT.md），可让 OpenClaw 自己完成安装

## 对方比我们强的功能点

### 1. 🔁 Replay / 历史回放系统（P0 差距）
对方有 `replay-index.ts` + `timeline.log` + `digests/` + `export-snapshots/` 构成完整的时间线回放能力。可以按时间段查看历史执行记录和摘要，相当于"行车记录仪"。
**我们目前**：只有当前状态实时展示，没有历史追溯入口。

### 2. 📦 导入/导出快照系统（P1 差距）
对方有完整的 import（dry-run + live）/ export bundle 机制，支持完整状态打包导出，且有安全保护（dry-run 默认开启）。
**我们目前**：CURRENT.md 中把导入/导出列为"P2 开放项，不紧急"，尚未实现。

### 3. 🔔 通知中心 + ACK/Snooze 机制（P1 差距）
对方有 `notification-center.ts` 提供通知确认（ACK）和静音/延后（snooze）能力，通知可以被"已读"或"推迟处理"。
**我们目前**：有 Toast 通知系统，但无持久化的通知中心和 ACK 机制，通知刷新后消失。

### 4. 💰 双层预算治理（P1 差距）
对方有 `budget-governance.ts`（运行层） + `budget-policy.ts`（策略层）两层结构，预算风险作为独立治理对象管理。
**我们目前**：有 Usage 页面的 token 归因和上下文压力图，但没有预算策略层（告警阈值、截止策略、自动保护）。

### 5. 🔍 操作审计 / diff 摘要（P2 差距）
对方有 `operation-audit.ts`（操作留痕）+ `diff-summary.ts`（差量追踪），可追溯"谁改了什么"。
**我们目前**：写操作有 Toast 反馈，但无审计日志和变更 diff。

### 6. 🌐 双语 UI（P2 差距）
对方通过 URL 参数 `?lang=zh` / `?lang=en` 原生支持双语切换，连 README 和安装提示词都做了双语版本。
**我们目前**：全中文界面，无多语言机制。

### 7. 👤 用户偏好持久化（P2 差距）
对方有 `ui-preferences.ts` + `avatar-preferences.ts` 提供 UI 偏好和头像个性化设置持久化。
**我们目前**：无用户偏好存储。

## 我们比对方强的功能点

### 1. ✅ 写操作能力全面落地
我们有 Tasks POST/PATCH 接口、Memory 编辑、Settings dry-run 写操作，且有 Toast 实时反馈。
对方的写操作被安全开关保护得较严，强调 read-only 优先。

### 2. ✅ SSE 实时推送 + 指数退避重连
我们实现了完整的 SSE 推送（1→2→4→8→30s 退避）和页面感知自动重连，实时性更强。
对方从 README 看未特别强调实时推送机制。

### 3. ✅ Agent 6 种精确状态展示
working / idle / blocked / backlog / error / offline 六态精确区分，比对方 Staff 页的"真实工作 vs 排队"更细。

### 4. ✅ 设计 Token 系统完整落地
我们有间距/字号/圆角/颜色/阴影五维设计 Token 体系，视觉规范更系统化。

### 5. ✅ 飞书告警集成
Context 压力 ≥80% 自动飞书告警，Agent 空闲超 2h 告警，异步防重。这是对方没有的飞书深度集成。

### 6. ✅ 全站搜索（Cmd+K）
全局搜索框 + 快捷键，按 agent/任务/会话分组展示，对方无此功能。

### 7. ✅ PM2 生产部署配置
完整的 PM2 ecosystem 配置，一键启动/停止/日志，对方在部署层没有特别提及 PM2。

## 建议借鉴的产品能力（注明优先级 P0/P1/P2）

### P0 — 必须做，核心差距

| 借鉴点 | 具体行动 | 理由 |
|--------|----------|------|
| **历史回放 / 时间线** | 设计 Timeline 视图，记录执行事件到 `timeline.log`，提供按时间筛选的历史回放入口 | 这是从"当前观测工具"升级到"可追溯的审计工具"的核心能力跃迁。没有回放，出问题后排查成本极高。 |

### P1 — 重要，近期规划

| 借鉴点 | 具体行动 | 理由 |
|--------|----------|------|
| **通知中心 + ACK/Snooze** | Toast 系统升级为持久化通知中心，支持"已读/延后处理"，保留最近 N 条通知历史 | 现有 Toast 消失即丢失，飞书告警也只是通道，我们缺一个"未处理事项收件箱"。 |
| **预算策略层** | 在 Usage 基础上增加预算阈值配置（如：7天花费 > X 触发告警，上下文压力 > Y 自动降频提示） | 现在只能"看到"压力，不能"设置保护"。 |
| **导入/导出快照** | 实现 export bundle（含 sessions + tasks + budget 摘要），支持一键下载当前快照 | 对灾备和团队交接有实际价值。 |

### P2 — 有价值，可选排入

| 借鉴点 | 具体行动 | 理由 |
|--------|----------|------|
| **操作审计日志** | 所有写操作留痕（谁/何时/改了什么），提供 audit 查看页 | 多人使用场景或高风险操作后的溯源需求。 |
| **安装提示词工程** | 仿照 INSTALL_PROMPT.md，为我们的控制台提供一段可直接给 OpenClaw 执行的安装提示词 | 降低新用户上手门槛，适合开源推广场景。 |
| **差量摘要 / diff 视图** | 在 Docs/Memory 编辑保存时，显示本次变更 diff 摘要 | 提升文档编辑的安全感和可追溯性。 |

## 不建议照搬的点及原因

| 点 | 原因 |
|----|------|
| **默认只读 / 关闭所有写操作** | 对方的产品定位是"面向非技术运营者的观测工具"，写能力是附加项。我们已经明确定位为"可以在里面工作的工具"，写操作是核心差异化。照搬只读优先会倒退。 |
| **双语切换 URL 参数** | 我们的用户群是中文团队，双语是对外推广需求，不是当前用户痛点。投入产出不高，且 URL 参数切换方案对 SPA 有一定架构影响，暂不推荐。 |
| **`avatar-preferences.ts` 用户偏好个性化** | 我们是团队内部工具，个性化偏好价值有限，优先级极低。 |
| **INSTALL_PROMPT 大段提示词** | 对方的安装提示词很长（>1000字），是为了兜住各种环境差异。对我们的场景，环境相对固定，简短版安装说明更合适。 |
| **`office-session-presence.ts` 办公会话在场感知** | 这是对方特有模块，功能尚不明确（可能是在线状态感知），在我们尚未厘清具体需求前不建议引入。 |

## 综合结论

对方是一个**产品思路清晰、安全设计扎实**的开源控制台，最值得我们借鉴的是：

1. **历史回放 / 时间线系统**（P0）：这是目前我们最明显的功能盲区，补上之后控制台从"实时快照工具"升级为"可追溯的运营审计工具"，用户价值跃迁明显。

2. **持久化通知中心 + ACK 机制**（P1）：现有 Toast + 飞书告警的组合缺少一个"未处理事项收件箱"，通知中心补上这个缺口，让运营用户有一个清晰的待办感知入口。

这两点也是最符合我们当前阶段（已完成基础能力，准备进入深化迭代）的优先级排列。
