## 一、办公场景适用性（信息密度）

### 总览页
- KPI 卡（5个）：活跃 Agent、任务数、用量、Gateway、队列，信息密度适中，对管理者友好 ✅
- 但 `/api/v1/status` 路由 404（艾克已发现），首屏"等待检查"状态是办公用户的第一印象杀手 🔴
- Inspector 侧边栏有"系统状态/今日用量/活跃 Agent/待处理事项"四项摘要，办公用户真正需要的正是这种快照，结构合理 ✅
- 告警摘要（alerts-list）存在，但未见主动推送机制，办公用户依赖被动刷新 🟠

### 任务页
- **57 个任务，全部 status=active，0 个完成** — 这是当前任务页面最大的办公体验问题
  - 办公用户打开任务页，第一眼看到的是一个"全部进行中"的任务海，完全不知道优先级
  - 任务名称如 `cc borrow P1 settings cards leona` 是内部编码格式，非开发者无法理解
  - 没有任务创建入口（页面无"新建任务"按钮），任务页变成纯只读展示面板
  - 过滤器仅有"全部/进行中/已阻塞/已完成"，缺少按负责人、优先级、日期的过滤
  - 没有任务归属人（assignee）字段展示，不知道谁在负责哪个任务
  - 任务详情面板（task-detail）存在但只读，无法操作 🟠
- 看板视图结构合理（进行中/已阻塞/已完成三栏），但在只读+任务全 active 的情况下对办公用户无实际价值 🟡

### 记忆页
- 记忆文件仅 5 条，且只覆盖 **提莫** 和 **Codex** 两个 Agent
- 团队 12 个 Agent 中有 10 个 Agent 的记忆文件不可见，"记忆"页对团队整体状态感知价值有限
- 页面标题"记忆文件 — Agent 工作区文件"对非技术办公用户偏抽象，不知道这个页面有什么用 🟡

### 文档页
- 文档页面有"文档列表 — 项目交付文档"，结构为列表+内容预览
- 缺少文档分类/标签/搜索能力，文档多了之后浏览成本高
- 没有说明"文档从哪里来"或"如何提交文档"，新用户不知道这个页面如何使用 🟡

### 协作页
- 协作页面副标题是"协作会话 — **Subagent 任务树**"——完全是开发者术语
- 办公用户不理解"Subagent"，也不清楚"任务树"的业务含义
- 该页面对普通办公用户来说等于黑盒 🟠

## 二、中文化体验（术语友好性）

### 已做好的中文化 ✅
- 导航标签全中文（总览/Agent 状态/协作/任务/用量/记忆/文档/设置）
- 操作按钮中文（刷新/全部/进行中/已阻塞/已完成/列表/看板）
- 搜索触发按钮有中文描述"搜索 agent/任务/会话..."
- 空状态提示"请选择左侧文件查看内容"有中文

### 未中文化的技术术语 🟠
| 术语 | 出现次数 | 建议替换 |
|------|---------|---------|
| SSE（连接状态） | 1 | 实时连接 |
| token / Token | 10 | 用量 / Token 用量（保留 Token 可以，但前后一致） |
| cron / Cron | 4 | 定时任务 |
| cache | 3 | 缓存（或不展示给普通用户） |
| READONLY_MODE=true | 1 | 只读模式（技术参数不应直接暴露到界面） |
| Subagent 任务树 | 1 | 协作任务 / 子任务结构 |

### 中文化一致性问题
- "Agent 状态" 中 Agent 未译，页面内其他地方也用"Agent"，可以保持，但应在首次出现时加括号说明（如"Agent（智能体）"）
- 安全设置页 "READONLY_MODE=true" 直接暴露底层环境变量参数，应改为友好描述
- 顶栏 SSE 状态 pill 只有 title tooltip 有中文描述，pill 本身显示"🟡 连接中"没有说明这是什么的连接

## 三、飞书集成状态影响

- **当前状态：飞书 not_configured**（`/api/v1/settings/wiring-status` 返回 `feishu.status: not_configured`）
- 总览页有"飞书通知 — Bot 连接状态"专属面板，但飞书未配置时用户会看到一个状态空白/加载中的面板
- 这对于"以飞书作为主要工作入口"的办公用户是核心缺失：
  - 控制台的事件和告警无法推送到飞书
  - 用户需要主动打开浏览器才能感知系统状态，不符合办公习惯
  - SSE 实时性做得好（艾克已确认），但只在控制台页面内有效，飞书集成断路后推送链路不完整
- **建议**：飞书未配置时，总览页的飞书面板应显示明确引导——"尚未配置飞书推送，点击前往设置"而非空白 🔴（对办公场景是高优先）

## 四、任务管理实用性

### 当前可用性评分：2 / 10（办公场景）

**根本问题：任务页是只读展示面板，不是任务管理工具**

| 功能 | 是否存在 | 对办公用户价值 |
|------|---------|--------------|
| 查看任务列表 | ✅ | 中（但全是 active，信噪比低） |
| 创建任务 | ❌ | 缺失，无法新建 |
| 修改任务状态 | ❌（只读模式禁用） | 缺失 |
| 任务归属人 | ❌ | 缺失，不知道谁负责 |
| 按人/优先级过滤 | ❌ | 缺失 |
| 任务优先级展示 | 仅在文件名中 | 无界面显示 |
| 任务完成归档 | ❌ | 0 个完成任务 |

**具体问题：**
1. 57 个任务全部 active，任务系统未实现状态流转，这是数据层问题（任务从未被标记为完成）
2. 任务名称是内部文件名编码（`cc borrow P1 settings cards leona`），办公用户无法理解
3. 只读模式下状态变更按钮仍显示（艾克已发现），但对办公用户还会产生额外困惑："我能改吗？为什么按了没反应？"
4. 缺少按负责 Agent 过滤，不能快速找到某个 Agent 的任务

## 五、首次使用引导友好性

### 当前引导评分：0 / 10

**完全没有任何首次使用引导：**
- 无 onboarding 流程
- 无功能介绍浮层
- 无帮助入口（页面内无"帮助"或"使用说明"链接）
- 无 tooltip（仅 3 个技术性 title 属性，对操作引导无帮助）
- 无空状态引导（如"暂无任务，你可以这样开始..."）
- 飞书未配置时没有配置引导
- 文档页没有说明文档来源

**首次用户面临的具体困惑：**
1. 打开控制台，"等待检查"状态挂在那里 → 不知道系统是否正常
2. 看到"记忆"标签 → 不知道是什么的记忆，能做什么
3. 点击"协作" → "Subagent 任务树"，完全不理解
4. 点击任务 → 全是进行中，不知道从哪里改，为什么改不了
5. 飞书面板空白 → 不知道是配置问题还是故障
6. 只读 pill 在那里 → 不知道只读是临时状态还是永久限制，如何解除

## 对 Product / UI / Coding-QA 的建议输入

### → Product（艾克）
1. **任务状态流转必须打通**：任务一直是 active 状态，说明后端没有实现任务状态变更流程，这是任务管理 P0 问题
2. **飞书集成必须进入当前版本验收**：控制台的核心价值是"让飞书用户感知 AI 团队状态"，飞书断连就是核心功能缺失
3. **任务展示友好名称**：建议后端支持 display_name 字段，与内部文件名分离

### → UI（璐珂）
1. **添加三个空状态设计**：飞书未配置空状态（含引导按钮）、任务完成空状态、记忆无文件空状态
2. **协作页副标题调整**："Subagent 任务树" → "协作任务 — 多 Agent 协同进展"
3. **设计一个轻量级首次访问提示条**（非 modal，一行横幅即可）

### → Coding-QA（伽利奥）
**用户侧验收诉求（当前版本必须项）：**
1. 飞书未配置时，设置页和总览页应显示明确的引导提示（不允许空白）
2. 任务列表至少能按状态正确分类（done 任务不能出现在 active 过滤下）
3. 只读模式的操作按钮必须灰化或显示"只读模式"标记，不允许点击后无反应
4. 顶栏 SSE pill 文字必须有中文标注或替换为"实时连接"

**建议项（后续版本）：**
- 任务页增加按 Agent 过滤
- 记忆页覆盖全部 Agent
- 添加轻量 onboarding 引导

## 办公顾问总结

控制台当前是一个**对开发者友好、对办公用户陌生**的状态仪表盘。  
优势在于：数据实时性好（SSE）、导航结构清晰、整体中文化基本完成。  
核心差距在于：**任务管理不可用（全 active）、飞书推送断路、零引导体验**。

对真实办公用户来说，当前版本的使用路径是：打开 → 看到等待检查 → 点任务 → 全是进行中 → 想操作 → 被只读阻断 → 不知道下一步。  
这条路径是"体验死胡同"，需要在当前版本修复 🔴 问题后，才能正式向非技术办公用户推广。
