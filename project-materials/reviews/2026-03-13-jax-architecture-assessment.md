## 1. 总体结论：有条件支持 ✅

**判定：有条件支持，建议继续推进。**

### 支持理由

1. **问题域适配**：Gateway 确实不提供原生 REST `/api/v1/*`，适配层是在现有约束下的合理工程选择，不是过度设计。
2. **关注点分离正确**：适配层把"Gateway 接口形态差异"与"前端业务需求"解耦，前端只面向稳定 DTO，这是正确的架构隔离。
3. **缓存作为必需项**：实测 CLI/RPC 调用延迟 2.5s 左右（本次实测 `status` 2.51s、`agents.list` 2.48s），`/health` 仅 33ms。缓存不是优化，而是使系统可用的前提条件。方案已认识到这一点。
4. **最小范围冻结**：D4-D5 只冻结 6 个只读 GET 接口，不进入写操作和 WebSocket，范围控制合理。
5. **降级策略完整**：stale-while-revalidate + 最近快照 + freshness 标记的三层降级方案是工业界标准做法。

### 条件（必须满足）

1. **必须建立版本兼容隔离层**（见建议#1）
2. **必须限制适配层膨胀**（见建议#2）
3. **缓存预热必须作为 P0 实现**，不能推迟到后期优化（见建议#3）

## 2. 关键架构建议（3 条）

### 建议 #1：建立 Gateway 版本契约与兼容隔离

**问题**：当前适配层直接依赖 Gateway CLI/RPC 输出结构。Gateway 版本升级时，输出字段、嵌套层级、甚至命令名都可能变化，这会直接穿透到适配层核心逻辑。

**建议**：

```
Gateway CLI/RPC
  ↓
[ Raw Response Parser ]  ← 一个 gateway 版本一套 parser
  ↓
[ Canonical Internal Model ]  ← 适配层内部稳定模型，不对外暴露
  ↓
[ DTO Mapper ]
  ↓
Office REST DTO
```

具体措施：
- 在 `RpcInvoker` / `CliInvoker` 之上增加一层 **Raw Parser**，负责把 Gateway 原始输出映射到内部 canonical model。
- Parser 按 Gateway `runtimeVersion`（当前为 `2026.3.11`）做版本标记。
- 版本升级时只需新增/修改对应 parser，不动 DTO Mapper 和 Service 层。
- `skills list` 的表格解析器必须有回归测试样例，这是最脆弱的环节。

**代价**：增加一层映射，约增加 10-15% 代码量。  
**收益**：Gateway 升级时改动被限制在 parser 层，不会雪崩到 DTO 和前端。

### 建议 #2：适配层责任边界必须刚性约束

**问题**：适配层天然有"万能中间层"膨胀倾向。一旦团队习惯了"后端包一层"，所有需求都会往适配层堆——告警规则、趋势计算、权限检查、任务编排，最终适配层变成第二个 Gateway。

**建议**：

明确适配层 **只做** 四件事：
1. **调用封装**：CLI/RPC 调用 + 超时 + 重试
2. **数据规整**：原始结构 → canonical model → DTO
3. **缓存管理**：TTL、预热、stale-while-revalidate、熔断
4. **安全裁剪**：过滤敏感字段，不透出原始配置

适配层 **不做**：
- 告警规则引擎（应由独立模块或未来 Gateway 能力承接）
- 趋势计算和历史对比（Dashboard 契约中的 `trend` 字段，MVP 阶段建议先用 `stable` 固定值或移除）
- 复杂权限模型（MVP 阶段 JWT 鉴权在网关/入口层做，适配层只做 pass-through）
- 任何写操作代理

**建议在项目 DECISIONS.md 中写入适配层边界条款，作为后续需求评审的守门依据。**

### 建议 #3：缓存预热必须前置为 P0

**问题**：当前方案描述了预热策略（后台 15-60s 刷新），但没有把它定义为 MVP 联调的 P0 必备。如果联调时只实现了 on-demand cache（请求触发缓存），那么：
- 首次访问必定 2-3s+（多个 CLI 调用并发仍需 2.5s）
- 缓存过期后下一个用户同样触发慢路径
- 多用户并发时出现 cache stampede（缓存击穿）

**建议**：

1. 服务启动时立即执行一次全量预热（status + agents.list + config.get + skills list）
2. 后台定时刷新间隔：dashboard 核心数据 30s，skills/workspace 5min
3. 联调验证 D4-D5 的通过条件必须包含："缓存预热后，dashboard 首次访问 P95 < 500ms"
4. 缓存实现建议采用 **singleflight** 模式（同一 key 并发刷新只执行一次），防止 stampede

## 3. 风险识别与规避

| # | 风险 | 等级 | 影响 | 规避措施 |
|---|------|------|------|----------|
| R1 | CLI/RPC 进程开销高 | 高 | 每次调用 spawn 子进程，CPU/内存峰值显著；实测单次调用 CPU user 2.9s | 缓存预热 + singleflight 减少调用频率；监控进程 spawn 频率 |
| R2 | Gateway 版本升级破坏输出格式 | 中 | 字段变更穿透到前端 | 建议#1 的 Raw Parser + 版本标记方案 |
| R3 | `skills list` 表格格式变化 | 中 | 解析器失败，skills 数据不可用 | 独立解析器 + 回归样例 + 失败降级为空列表 |
| R4 | 适配层膨胀为第二 Gateway | 中 | 维护成本失控，架构退化 | 建议#2 的刚性边界 + DECISIONS.md 守门 |
| R5 | Dashboard 契约中趋势/告警字段缺乏数据源 | 低 | 前端拿到的是固定值或空数据 | MVP 阶段明确标记为 mock/placeholder，不伪造趋势 |
| R6 | 单点故障：适配层自身挂掉 | 低 | MVP 阶段可接受，但生产需考虑 | MVP 不处理；后续考虑进程守护或容器化 |

### 特别说明：R1 进程开销

实测数据显示，`openclaw gateway call` 每次调用消耗 user CPU ~3s、sys CPU ~0.5s。这不是网络延迟，而是 Node.js 进程启动 + Gateway 内部处理的综合开销。这意味着：
- 如果不做缓存，每个用户请求消耗 3 次 CLI 调用 ≈ 9s CPU 时间
- 高并发场景下 CPU 会成为瓶颈
- **缓存预热是性能可行性的基础条件，不是可选优化**

## 4. 接口契约评审补充

Dashboard 接口契约总体合理，以下是需要注意的几点：

1. **`quickStats` 中的 `trend` 字段**：当前没有历史数据存储，无法真正计算趋势。建议 MVP 阶段要么返回 `"stable"` 固定值，要么从 DTO 中移除此字段。不要返回伪造的趋势数据。

2. **契约中的 `system.uptime`**：Gateway `status` 输出中未直接包含 uptime。需要确认数据源——可能需要从 Gateway 进程启动时间推算，或标记为"不可用"。

3. **契约与修订方案的 DTO 不一致**：修订方案定义了 `OfficeDashboardDto`（含 `gateway.online` / `summary` / `agents[]` / `risks[]`），接口契约定义了另一套结构（`system` / `agents` / `workspaces` / `alerts`）。**建议 Leona 和 Ezreal 在联调前统一为一套结构**，否则前后端各实现一套会产生不必要的对齐成本。

4. **缓存标识字段统一**：修订方案建议 `meta.cache.hit/stale/fallbackUsed/fetchedAt`，接口契约用 `cached/cacheExpiresAt/stale`。风格不同但语义可对齐。建议采用修订方案的 `meta` 包装结构，更规范。

**当前 D4-D5 联调验证方向基本正确，不需要大幅调整。**

4. **回归样例先行**：在实现 `skills list` 解析器之前，先录一份当前输出的 snapshot 作为回归基线。
