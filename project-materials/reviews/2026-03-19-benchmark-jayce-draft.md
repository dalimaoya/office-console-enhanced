## 对方办公友好度概览

对方项目（OpenClaw Control Center）的整体定位是：**为非技术运营者打造的本地控制中心**。

README 第一段就说："Built for non-technical operators who need observability and certainty, not raw backend payloads."

这句话本身就是办公友好度的核心宣言——他们从立项起就把"让不懂技术的人也能看懂"作为设计前提。

对比我们当前方案：我们的控制台仍处于"给技术人员用的调试面板"阶段，对办公用户更多是展示原始状态，而非翻译成可操作的业务信号。

**办公友好度对比评分（初步）：**

| 维度 | 对方 | 我们 |
|------|------|------|
| 面向非技术用户的语言 | ✅ 显式设计目标 | 🟠 部分中文化，但技术术语仍多 |
| 状态可信性（打开即知道系统好坏） | ✅ Overview "Is OpenClaw okay right now?" | 🔴 首屏"等待检查"是体验杀手 |
| 谁在做什么（Staff 感知） | ✅ Staff 页专门区分活跃/队列/阻塞 | ❌ 无人员维度状态区分 |
| 预算/成本风险感知 | ✅ 总览有 Budget risk + Context pressure | 🟠 用量 KPI 存在，但不会主动预警 |
| 安全设置友好性 | ✅ 翻译成业务语言（不暴露底层参数） | 🔴 READONLY_MODE=true 直接暴露 |
| 引导体验 | ✅ 有 INSTALL_PROMPT，各页有"最适用场景"说明 | ❌ 零引导，无 onboarding |
| 飞书/办公工具集成 | ❌ 无（纯本地 web 面板） | 🟠 有飞书集成但当前断路 |

## 对方更适合办公用户的地方

### 1. 每个页面都有"最适用场景"说明

对方 README 里，每个功能板块都有一行 `Best when you want to...` 的说明：

- Overview → "Best when you want one fast answer to: Is OpenClaw okay right now?"  
- Staff → "Best when you want to know who is busy, idle, blocked, or waiting."  
- Tasks → "Best when you want to understand what is being carried, what is only planned, and what needs intervention."  

这不是技术说明，而是**从用户目的出发的入口提示**。办公用户不知道要去哪个页面，看到这句话就知道该点哪里。

**我们缺失：** 我们的导航只有标签名，没有任何场景提示。用户打开控制台，不知道"协作"和"任务"有什么区别。

---

### 2. Staff 页：区分"真正在工作"与"只是排队"

对方有独立的 Staff 页面，专门区分：
- 当前真正活跃执行的 Agent
- 只有排队任务、还没开始执行的 Agent
- 已阻塞/等待的 Agent

对于一个办公团队的管理者来说，这解决了最核心的焦虑：**"我部署了这么多 AI，到底谁在干活？"**

内部编码格式的 57 个全 active 任务无法回答这个问题；一个清晰的"当前值班 AI：3 个活跃 / 2 个待机 / 1 个阻塞"才是办公用户真正需要的。

**我们缺失：** 我们没有 Staff 概念，Agent 状态页展示的是技术状态（连接状态/模型类型），不是业务状态（今天在做什么/当前是否有工作）。

---

### 3. 安全设置用业务语言而非技术参数

对方有 `Security risk summary` 卡片，其描述是：
> "translates current risk, impact, and next-step guidance into plain operator-facing language"

例如，不会告诉用户 `APPROVAL_ACTIONS_ENABLED=false`，而是告诉用户"当前处于安全模式：所有审批动作已禁用，如需开启请联系管理员"。

**我们现状：** 我们安全设置页直接暴露 `READONLY_MODE=true`，是直接把配置文件变量丢给用户看，对办公用户来说是技术噪音。

---

### 4. Connection Health 卡片：明确告诉用户哪里没接好

对方有 `Connection health` 卡片，区分：
- 已正确连接的模块
- 部分连接/降级状态的模块
- 完全未连接/需要补充配置的模块

对比我们的飞书面板问题：飞书 `not_configured` 时，我们的总览页显示空白，用户不知道是配置问题还是功能故障。

这一点直接对应了我上次 UX 测试的 🔴 严重问题：**飞书未配置时缺少引导提示**。

---

### 5. Context pressure 卡片：主动预警成本风险

对方在 Usage 页有 `Context pressure` 卡片，告诉用户哪些 Agent 会话已接近上下文限制——这意味着即将：
- 速度变慢
- 花费增加
- 可能需要手动干预

这是把技术限制（context window）翻译成了业务风险（成本/效率），是办公管理者真正关心的内容。

**我们现状：** 我们有用量 KPI，但没有预警机制，用量超出前用户不会提前感知到。

---

### 6. 安全第一的首次安装体验

对方提供了 `INSTALL_PROMPT`，设计成可以直接粘贴给 AI 执行的安装脚本，考虑了：
- 没有技术背景的用户
- 各种非标准路径
- 缺少依赖时的降级提示（不是报错停止）

这对于要把控制台推广给团队内非技术同事的场景非常友好。

## 我们更适合办公用户的地方

### 1. 飞书深度集成（如果打通）

对方是纯本地 Web 面板，用户必须主动打开浏览器才能看到 AI 团队状态。

我们的方案以飞书为主入口，AI 团队的任务进展、通知、报告都直接发送到飞书群——办公用户不需要改变工作习惯，在飞书里就能感知 AI 协作状态。

这是我们架构上最根本的办公友好性优势：**把信息推到用户日常工作的地方，而不是要求用户来找信息**。

### 2. 多角色 AI 团队设计，对应真实组织结构

我们有明确的角色分工（提莫、杰斯、艾克、贾克斯、伊泽瑞尔等），每个角色对应真实团队岗位（项目管理/办公顾问/产品/架构/前端/后端）。

办公用户看到"艾克在处理需求"、"杰斯在做 UX 分析"，比看到"Agent-3 正在执行 task-id-8872"更有认知锚点。

### 3. 工作流是"办公人员托管给 AI 执行"，而非"开发人员观测 AI 状态"

对方的定位是观测工具（observability dashboard）；我们的定位是办公执行工具（AI 团队实际干活，办公用户在飞书里看结果）。

两个定位不同，对于"不懂技术、只关心用 AI 提效"的办公用户，我们的目标更接近真正的使用价值。

## 建议借鉴的办公友好特性（P0/P1/P2）

### P0：立即借鉴（影响当前版本可用性）

| 特性 | 来源 | 具体建议 |
|------|------|---------|
| **Connection Health 引导卡** | 对方 Settings | 飞书未配置时，总览/设置页显示专门引导卡，明确说明"飞书推送未连接，点击配置"，而非空白 |
| **"谁在真正工作"状态** | 对方 Staff 页 | 在总览页或 Agent 状态页增加"当前活跃 / 待机 / 阻塞"三态区分，用业务语言而非技术连接状态描述 |

### P1：尽快借鉴（影响用户留存和推广）

| 特性 | 来源 | 具体建议 |
|------|------|---------|
| **每页"适用场景"提示** | 对方各页说明 | 在导航标签或页面标题副标题加一行用途说明，如"协作 — 查看多 Agent 任务传递状态" |
| **安全设置业务化描述** | 对方 Security risk summary | 把 `READONLY_MODE=true` 等技术参数翻译成"当前模式：只读安全模式，AI 不会修改任何系统文件" |
| **成本预警信号** | 对方 Context pressure | 在用量页增加"接近上限预警"，而不只是展示已用数字；当某 Agent 接近 context 上限时主动标红 |

### P2：中长期借鉴（提升专业感）

| 特性 | 来源 | 具体建议 |
|------|------|---------|
| **Update status 卡片** | 对方 Settings | 告知当前版本、最新版本，管理者关心 AI 工具是否是最新状态 |
| **Memory status 卡片** | 对方 Memory 页 | 告知每个 Agent 的记忆是否健康可用，目前我们 10/12 个 Agent 记忆不可见 |
| **安全安装引导** | 对方 INSTALL_PROMPT | 为团队内新用户/新管理员提供可直接粘贴的"安全首次启动"指南 |

## 不建议照搬的点及原因

### 1. 纯本地 Web 面板的交互模式

对方是需要用户主动打开浏览器的"控制台"，这对我们是降级体验——我们已经有飞书主入口的优势，不应该要求用户再去打开另一个网页。

建议：飞书通知 + 可选的 Web 控制台作为补充，而非把 Web 控制台作为主入口。

### 2. 纯观测定位（Observability only）

对方明确说"not a replacement for OpenClaw itself"，只做观测。我们不应该把自己限定为观测工具——我们的价值在于 AI 团队实际执行任务、产出结果，控制台是辅助感知层，不是核心价值层。

照搬这个定位会让我们的控制台退化为"看热闹的面板"。

### 3. 英文为主的受众设定

对方以英文为主，中文是次级选项。我们的用户是中文办公场景，中文必须是第一优先级，包括所有空状态文案、错误提示、引导文字。

## 杰斯总结

对方最值得借鉴的核心理念只有一句话：**"把技术状态翻译成业务可读的办公信号"**。

这体现在：Connection Health、Security risk summary、Context pressure、Staff 页的活跃/队列区分，以及每页的"最适用场景"说明。

我们当前方案的根本差距不是功能多少，而是**把技术信息直接对外暴露，没有做"翻译层"**。READONLY_MODE=true 就是典型——这是一个环境变量，不是给人看的。

**最重要的两个借鉴点（向 Teemo 的结论）：**

1. **P0：Connection Health 概念** — 当飞书/Gateway 等核心集成未配置时，应有明确的业务语言引导卡，而非空白或技术错误提示。这直接解决了我们当前 🔴 严重问题。

2. **P1："谁在真正工作" + 每页适用场景提示** — Staff 状态区分和场景说明，是办公管理者打开控制台后能立刻看懂、立刻找到自己需要的信息的关键设计。这是从"给开发者的调试面板"变成"给管理者的状态看板"的最短路径。
