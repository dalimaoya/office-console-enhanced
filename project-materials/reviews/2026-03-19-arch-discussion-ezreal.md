# 架构论证评审（伊泽瑞尔 / 前端工程师）

**日期**：2026-03-19  
**视角**：工程实现与可视化呈现  
**结论先行**：从前端与工程落地角度看，`当皇上` 更像“以消息为中心的可视化外挂”，而我们的办公控制台应坚定走“以对象/状态/文件流为中心的项目操作系统”路线。真正的差异不在有没有 Dashboard，而在 Dashboard 背后是否存在稳定的对象模型、状态机、审计链和可持续的项目级上下文承载方式。

## 一、前端/工程实现角度：`当皇上`三层 GUI 与我们的技术路线差异

### 1.1 `当皇上` 的前端路线本质

从简报看，`当皇上` 已实现的三层 GUI 是：
- **Dashboard**：集中展示任务、状态、角色分工
- **Discord**：消息触发与人机交互入口
- **Notion**：文档沉淀与结果承载

这条路线的优点是：
1. **上手快**：直接复用 Discord、Notion 现成生态，低成本搭出可演示系统。
2. **传播性强**：角色设定鲜明，Dashboard 容易做出“系统在运转”的观感。
3. **消息闭环简单**：@mention → 调度 → 回复，适合轻量流程。
4. **外围集成友好**：前端只要把多个外部系统拼起来，就能快速形成“有界面”的产品形态。

但它的工程问题也很明显：
1. **UI 是多系统拼接，不是统一产品面**。Dashboard、Discord、Notion 各自拥有不同的对象定义和交互范式，用户是在“跨系统跳转”，不是在一个统一控制台里工作。
2. **消息是主索引，状态只是附属信息**。这导致前端很难稳定回答：一个任务现在到底处于什么状态、由谁持有、依据哪份正式文档推进。
3. **Notion 更适合结果陈列，不适合作为高频状态机底座**。前端若想做实时监督、调度、批量操作、异常巡检，会受到文档系统模型限制。
4. **上下文与流程耦合在消息链里**。这让前端只能展示“发生过什么”，很难展示“为什么发生、接下来应该是什么、有没有偏离预期”。

### 1.2 我们办公控制台的路线本质

我们现在最值得坚持的路线，是：
- 飞书只是**通知/入口层**；
- 控制台是**统一操作面**；
- 共享文件区是**正式对象层**；
- Agent 协作基于**文件交接与状态推进**，而不是相互聊天。

这意味着我们的前端不应该做成“消息面板的大屏版”，而应该做成：

> **项目对象操作系统（Project Object OS）**

它的核心实体不是“群消息”，而是：
- 项目（Project）
- 任务/工作包（Task / Work Package）
- 阶段（Stage）
- 角色（Role / Owner）
- 文件对象（Brief / Review / Handoff / Artifact / Decision / Status）
- 运行实例（Run / Session）
- 风险/阻塞（Risk / Blocker）
- 通知事件（Notification Event）

### 1.3 我们的优势

#### 优势一：可做“单一真相源”
`当皇上` 的真相分散在消息、Dashboard、Notion 三处；我们有机会把“共享文件区 + 控制台状态层”做成单一真相源。前端只要围绕统一对象模型构建，就能天然支持：
- 状态一致性
- 可回溯性
- 审批/调度链可视化
- 任务生命周期追踪

#### 优势二：更适合深度项目执行
深度办公场景的核心不是“谁回复了我”，而是：
- 目标是否明确
- 责任是否清晰
- 交接是否正式
- 阶段是否闭环
- 风险是否暴露

这些都更适合通过**对象、状态机、文件流**来承载，而非消息流。

#### 优势三：前端可做成真正的工作台，而不是展示板
如果我们把共享文件区抽象成对象层，前端能支持：
- 任务推进
- 阶段交接
- 文件比对
- 决策追踪
- 风险巡检
- 批量调度
- 审计回放

这比一个“看起来很热闹”的 Dashboard 更有长期价值。

#### 优势四：后续易扩展为多项目、多指挥官结构
消息中心型架构一旦扩展到多项目，就容易让总控角色变成超级拥堵节点；而对象中心型架构可以自然演进为：
- 全局监督层看所有项目对象摘要
- 项目指挥官看本项目对象全集
- 专业角色只看与自己相关的工作包和交接物

前端信息架构会更干净。

## 二、如果实现“全局监督层”：前端可视化应该如何设计

我建议把“全局监督层”设计为一个**监督与调度驾驶舱（Supervision Cockpit）**，定位不是日常编辑界面，而是“发现异常、理解态势、触发动作”的上层中枢。

### 2.1 页面结构建议

#### A. 顶部：全局态势栏
展示全局关键数字：
- 活跃项目数
- 运行中任务数
- 阻塞任务数
- 今日新增/完成工作包数
- 待处理通知数
- 超时会话数
- 文件交接积压数

这是一眼判断系统健康度的地方。

#### B. 中部左侧：项目/战区列表
按项目展示：
- 项目健康分
- 当前阶段
- 指挥官
- 活跃角色数
- 阻塞数
- 最近更新
- SLA 风险

支持排序、筛选、异常高亮。

#### C. 中部主区域：监督地图 / 调度图
这是全局监督层的核心视觉区域，建议提供两种视图切换：
1. **流程泳道视图**：按项目阶段展示对象推进情况。
2. **节点网络视图**：展示项目、角色、任务、文件之间的关系与流转方向。

#### D. 右侧：异常与告警面板
集中展示：
- 超时未响应
- 文件缺失
- 交接断链
- 重复调度
- 状态不一致
- 通知失败
- 风险升级未处理

这里要支持一键跳转到对象详情和发起补救动作。

#### E. 底部：事件时间线
按时间回放：
- 谁在何时创建/更新了什么对象
- 哪个阶段被推进/回退
- 哪个 blocker 被提出/解决
- 哪条通知已发出/失败

它是“系统审计层”的可视化入口。

### 2.2 关键指标建议

全局监督层不应只看“消息数量”，要看真正的执行质量指标：

#### 1）流程效率指标
- 工作包平均流转时长
- 阶段停留时长（按产品/架构/前端/后端/QA 分布）
- 首次响应时长
- 从 brief 到 review 的完成时长
- blocker 平均解除时长

#### 2）状态健康指标
- 正常 / 风险 / 阻塞 / 超时 对象占比
- 缺少成果物的任务数
- 缺少责任人的任务数
- 有通知但无正式交接物的任务数
- 有文件更新但未推进状态的任务数

#### 3）协作质量指标
- 角色切换次数
- 交接成功率
- 回退率（进入下一阶段后又被打回）
- 重复劳动率（同类文件多次重写、无效往返）
- 非结构化沟通占比（若未来能统计）

#### 4）系统运行指标
- agent 会话成功率
- 平均运行时长
- tool 调用失败率
- 消息通知成功率
- 文件同步延迟

### 2.3 交互原则

前端上一定要避免“只可看不可控”。监督层至少应支持：
- 一键催办
- 一键升级 blocker
- 一键分派下一角色
- 一键查看缺失文件模板
- 一键筛出即将超时对象

否则它只是大屏，不是控制台。

## 三、故事线改成军队体系或项目管理体系，对 UI/UX 设计意味着什么

我的判断是：**故事线不是纯文案问题，而是会直接影响信息架构、命名体系、视觉层级和用户心理模型。**

### 3.1 如果采用军队体系

例如：战区 → 集团军 → 师/旅 → 连 → 单兵执行。

#### UI/UX机会
1. **命令链很清晰**：适合做层级化指挥界面、任务编成图、战术地图式态势板。
2. **天然适合监督/巡检/调度表达**：例如“待命”“执行中”“集结完成”“阻塞”“回撤”等状态，非常容易视觉化。
3. **可做强态势感**：全局总览、战线推进、资源占用、异常告警，都容易形成统一风格。

#### UI/UX挑战
1. **语义容易过硬**：办公用户可能对军事术语产生压迫感或距离感。
2. **细颗粒执行任务不一定适配军事隐喻**：像写 PRD、改前端、验收接口，用“连排班组”隐喻不一定自然。
3. **对外传播要谨慎**：如果产品面向更广泛办公场景，军事叙事可能引发误解。

### 3.2 如果采用项目管理体系

例如：PMO → 项目经理 → 工作包负责人 → 执行团队。

#### UI/UX机会
1. **认知成本最低**：更贴近真实办公场景。
2. **状态命名更标准**：待澄清、待分派、执行中、待验收、已归档等，天然利于表格、看板、详情页设计。
3. **对象建模更顺手**：项目、阶段、工作包、风险、依赖、里程碑，本来就是成熟概念。

#### UI/UX挑战
1. **戏剧性弱、辨识度弱**：容易做成普通项目管理工具。
2. **角色人格感下降**：多 Agent 产品如果没有鲜明角色，用户记忆点会变少。
3. **演示感染力可能不如故事化体系**。

### 3.3 如果继续使用英雄联盟体系

#### 优点
- 角色辨识度高
- 群内协作和展示更有趣
- 便于形成团队人格与产品记忆点

#### 缺点
- 与“深度办公增强”存在语义跳跃
- 对新用户不够自解释
- UI 命名若过度依赖角色设定，会降低专业感与学习效率

### 3.4 我的建议：采用“项目管理骨架 + 轻故事皮肤”的混合方案

这是我最推荐的方向：

- **底层对象模型、导航命名、状态体系** 全部采用项目管理语言；
- **角色人格、插画、部分文案、辅助视图** 承载故事线；
- 支持主题切换，例如：
  - 专业模式：项目管理体系
  - 演示模式：英雄联盟/军队体系

这样做的好处是：
1. 不牺牲可理解性；
2. 保留产品记忆点；
3. 前端可以用同一套数据模型映射不同的视觉主题；
4. 后续对外可灵活切换品牌叙事。

换句话说，**故事线应该是 View Layer，不应该反向绑架 Domain Model。**

## 四、共享文件区作为核心交互方式：前端如何可视化“文件流转”和“对象状态”

这是我认为我们最有机会做出差异化的部分。

### 4.1 不要把共享文件区做成“文件夹浏览器”

如果只是把目录树搬到前端，那价值有限。前端应该把文件区升级为：

> **对象流转视图 + 文件状态视图 + 交接审计视图**

用户真正关心的不是“某个 md 文件放在哪”，而是：
- 这个对象是谁创建的？
- 属于哪个任务？
- 当前是否生效？
- 替代了哪个旧版本？
- 下一个接收方是谁？
- 有没有被正式引用？

### 4.2 建议的核心可视化方式

#### A. 对象卡片化
每个文件对象都不是裸文件名，而是一张对象卡片，展示：
- 标题
- 类型（brief/review/handoff/artifact/decision/status）
- 所属项目 / 工作包
- 当前状态（草稿/生效/已交接/已归档/已废弃）
- 创建者 / 当前责任人
- 更新时间
- 关联对象数量
- 是否存在风险或待处理动作

#### B. 文件流转链路图
对于单个任务，展示：
`brief → architecture review → frontend handoff → backend handoff → QA review → final artifact`

每一步都应可见：
- 生成时间
- 责任角色
- 是否缺失
- 是否回退
- 是否被后续对象引用

这会极大提升“项目推进是如何发生的”可理解性。

#### C. 对象生命周期时间线
展示某对象从创建到归档的全过程：
- created
- assigned
- updated
- approved / rejected
- handed off
- superseded
- archived

这能让前端从“看文件”升级到“看状态机”。

#### D. 双栏详情页：内容 + 元数据
左侧看文档内容，右侧看对象元数据：
- 当前阶段
- 来源对象
- 下游对象
- 状态历史
- 关联通知
- 关联风险
- 最近操作记录

这样用户无需来回跳转多个系统。

#### E. 差异对比视图
对于 review、decision、status 这类对象，应该支持：
- 版本 diff
- 摘要变化高亮
- 责任变更标注
- 决策点变更标注

因为办公控制台的核心不是最终文档本身，而是“文档如何推动协作”。

### 4.3 关键前端机制建议

#### 机制一：显式状态标签系统
文件不能只有“存在/不存在”，必须有显式状态，例如：
- 草稿 Draft
- 待接收 Pending Accept
- 生效 Active
- 已替代 Superseded
- 已归档 Archived
- 异常 Exception

#### 机制二：对象关系索引
一个文件必须能关联：
- 父任务 / 父项目
- 上游对象
- 下游对象
- 责任角色
- 关联通知
- 关联运行实例

这要求前端背后有统一元数据层，而不只是读文件内容。

#### 机制三：缺口可视化
对于流程中的缺失环节，前端应该用“空槽位”显式展示：
- 缺少 architecture review
- 缺少 QA 验收
- 通知已发但 handoff 未生成

把缺口看见，才能管理。

## 五、从工程角度，当前控制台最欠缺的能力是什么

如果只说一个，我认为是：

> **缺少统一的对象元数据层（Object Registry / Workflow State Layer）**

### 5.1 为什么这是最关键缺口

现在我们已经有共享文件区思路，但如果没有一层稳定的元数据与状态层，前端会遇到几个问题：
1. 只能“读文件”，不能真正理解对象关系；
2. 很难做实时状态总览；
3. 很难可靠判断任务是否卡住；
4. 文件更新和流程推进之间缺少机器可读的桥；
5. UI 只能停留在目录树、列表、日志这类低级展示。

### 5.2 我建议补齐的能力栈

#### 第一优先：统一对象注册表
每个正式成果物生成时，都自动登记：
- object_id
- object_type
- project_id
- work_package_id
- owner_role
- current_status
- upstream/downstream
- created_at / updated_at
- source_path
- visibility_level

这是所有高级前端能力的基础。

#### 第二优先：流程状态机
定义清楚：
- 任务有哪些阶段
- 每个阶段允许哪些对象进入/退出
- 什么情况下推进、回退、阻塞、归档

没有状态机，监督层和调度层都会失真。

#### 第三优先：事件总线 / 审计日志
需要一个统一事件流，记录：
- 文件创建/变更
- 角色接手/交接
- blocker 提出/解除
- 通知发出/失败
- 会话启动/完成

前端才能做时间线、回放、告警和监控。

#### 第四优先：操作式前端，而非只读前端
最终控制台应该可以直接：
- 发起交接
- 修改状态
- 补发通知
- 标记 blocker
- 升级风险
- 生成下一阶段模板

否则还是要回到聊天里操作，控制台价值会被削弱。

## 六、对简报中几个核心问题的补充回答

### 6.1 如何评价 `当皇上` 架构
**优势**：角色清晰、传播性强、消息驱动流畅、可快速做出演示级产品。  
**缺陷**：中心调度上下文膨胀、正式交接弱、对象模型弱、难承载持续型复杂项目。  
**结论**：适合轻办公/娱乐化协作，不适合作为深度项目执行的长期底座。

### 6.2 英雄联盟故事线是否适合深度办公增强
**适合做产品人格，不适合直接做系统骨架。**  
角色本身有记忆点，但若把核心导航、状态体系、职责层级都建立在游戏设定上，会增加认知成本。更适合做“皮肤层”和“角色层”，不适合做“领域层”。

### 6.3 如果重设计故事线，我推荐什么
我推荐：
**项目管理体系作为主骨架，辅以可切换的故事化主题层。**

原因是：
- 工程上最稳
- 前端信息架构最清晰
- 后续多项目扩展最自然
- 对外可保留演示感，对内可保持专业性

### 6.4 对第7条架构分层方案的完善
我建议把李琪给出的分层方案补成：

```text
全局监督/巡检/调度/通知层
    ↓
项目组合 / 项目集视图（可选，管理多个相关项目）
    ↓
垂直项目指挥官（每项目1个）
    ↓
专业领域 Agent（产品/架构/前端/后端/QA/设计/办公）
    ↓
执行运行层（Codex / tool runs / scripts / browser / integrations）
    ↓
对象与审计层（共享文件区 + 元数据注册表 + 事件流）
```

关键补充是最底部必须有：
- **对象层**（承接正式成果物）
- **事件层**（承接运行事实）

否则上层再漂亮，也会悬空。

### 6.5 对共享文件区的补充建议
共享文件区要从“目录约定”升级为“对象协议 + 元数据协议 + 状态协议”。  
也就是说，不仅要规定文件放在哪里，还要规定：
- 什么文件代表什么业务语义
- 哪些字段必须有
- 文件之间怎么建立引用关系
- 什么状态变化对应什么事件

前端只有基于协议，才能实现稳定可视化。

## 七、我的最终建议（前端视角）

### 建议 1：控制台不要以聊天为中心，而要以对象与状态为中心
聊天可以保留，但只能是入口和通知，不应成为主工作面。

### 建议 2：优先补“对象注册表 + 状态机 + 事件流”三件套
这比先做更炫的界面更重要，因为它决定前端能否做出真正可用的监督与调度能力。

### 建议 3：全局监督层应是可操作驾驶舱，不是展示大屏
要能发现问题，也要能直接采取动作。

### 建议 4：故事线采用“专业骨架 + 角色皮肤”的分层设计
保证工程建模稳固，同时保留产品个性与传播力。

### 建议 5：把共享文件区产品化为“对象流转系统”
这是我们相对 `当皇上` 最有机会建立壁垒的地方，也是前端最能做出差异的地方。

---

## 八、一句话结论

**`当皇上` 是把消息系统做得更有组织；我们应该把项目执行过程本身做成可视、可控、可追溯的对象操作系统。前者适合热闹地协作，后者更适合真正把复杂办公项目跑完。**
