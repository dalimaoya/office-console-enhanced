## 场景测试结论

### 场景1：早晨上班，快速了解团队进展

**接口数据验证：**
- `/api/v1/overview` ✅ 有效：返回 Agent 总数/活跃数/状态分布、今日 token 用量（95M tokens, $86.1）
- `/api/v1/agents` ✅ 有效：返回 8 个 Agent，状态区分 working/idle，带中文名称（如"杰斯"）
- `/api/v1/action-queue` ✅ 有效：返回 3 条 warnings（空闲超时告警），blocked 为空

**前端展示分析：**
- KPI 网格（活跃 Agent / 进行中任务 / 今日用量 / Gateway状态 / 待处理事项）结构清晰 ✅
- 冷启动快照卡片（`#cold-start-panel`）已实现，调用 `/api/v1/cold-start`，数据有内容 ✅
- 项目阶段卡片（`#project-status-panel`）已实现，当前阶段为 `active` ✅
- **痛点**：Overview 页面卡片数量偏多（冷启动快照 + 项目阶段 + 待办聚合 + KPI网格 + 系统概览 + 健康状态 + 工作区活动 + 告警摘要 + 活跃对象 + 事件日志），信息密度极高，30秒内很难聚焦。
- 冷启动快照标题写法"快速上下文 — 冷启动快照"对不懂"冷启动"含义的用户不直观，建议改为"今日工作摘要"或"当前进展快照"

**场景评分：⭐⭐⭐（3/5）** — 数据完整，但信息层次不够，用户视觉焦点容易被分散

### 场景2：了解项目待处理事项

**接口数据验证：**
- `/api/v1/registry?status=active&limit=20` ✅ 有效：返回 5 个活跃对象（artifact、project 类型）
- `/api/v1/events/log?limit=20` ✅ 有效：主要是 `system.healthcheck_passed` 和 `system.start` 事件
- `/api/v1/action-queue` ✅ 有效：3 个 warnings（agent 空闲告警），blocked 为空

**发现问题：**
- 活跃对象注册表展示的是文档/artifact 类型，缺少"任务/交接/验收"等典型办公事项对象 —— 这是数据录入问题，控制台结构没问题
- **重要痛点**：事件日志实际内容以 `system.healthcheck_passed`（系统心跳请求）为主，这对办公用户毫无价值，看到的全是技术噪音。用户期望看到的是"张三完成了任务X"、"提莫交接了任务Y"之类的业务事件。技术事件应过滤或降级显示。
- 时间线（`/api/v1/timeline`）目前内容也主要是 `system_start`，同样是技术噪音

**场景评分：⭐⭐（2/5）** — 结构在，但内容对办公用户无参考价值，事件日志类型过于技术化

### 场景3：飞书没有配置，需要设置

**接口数据验证：**
- `/api/v1/settings` ✅ 有效：返回 readonlyMode/tokenEnabled/dryRunEnabled/alertThresholds
- `/api/v1/settings/alerts` ❌ 404 —— 告警阈值 GET 接口未实现（只有 POST 端点？）

**前端展示分析：**
- Settings 页飞书通知区块 ✅：有专门的"飞书通知"区块（`#feishu-notify-status`），位置显眼
- "飞书通知未配置"有空状态提示卡（带说明文字和"去配置"按钮）✅
- **关键可用性问题**：配置引导仅告诉用户"在服务器 .env 文件中配置 FEISHU_WEBHOOK_URL，重启服务后生效"。对于不懂技术的用户，这是死路 —— 无法在界面上直接操作。
- 告警阈值配置（`#alert-thresholds-section`）✅ 有输入框（上下文压力%/空闲分钟/每日费用），可以修改数值，界面直观 ✅
- 告警阈值保存按钮存在，但 `GET /api/v1/settings/alerts` 返回 404，表明读取已有阈值的接口缺失

**场景评分：⭐⭐⭐（3/5）** — 告警阈值配置界面直观，但飞书配置仍依赖技术操作，非技术用户被拦

### 场景4：新同事首次使用

**界面结构分析：**
- 侧边栏标题"OpenClaw / 办公增强控制台"，副标题"多角色协作控制台，覆盖 8 大功能分区" ✅ 大方向表达清楚
- Onboarding Banner 顶部有引导步骤："① 查看实时 Agent 状态 → ② 在飞书接收任务通知 → ③ 在任务页管理工作队列" ✅ 流程直观
- 每页有 `page-subtitle`（如"最适合：快速了解当前 AI 团队整体状态"）✅ 说明用途

**问题：**
- 导航顺序：总览→Agent状态→协作→任务→时间线→用量→记忆→文档→**项目组**→设置。对新用户来说"记忆"和"协作"这两个词的含义不明确（Agent的记忆文件？多人协作？）。
- "项目组"页面 ✅ 界面简洁，有项目列表和"＋ 新建项目组"按钮，有 Modal 弹窗，操作直观。
- 但"项目组"和其他页面的关系对新人不清晰 —— 它是用来管理控制台项目数据，还是业务项目？

**场景评分：⭐⭐⭐（3/5）** — 引导流程到位，但"记忆/协作"命名对新人有理解门槛

## 最需要改进的2点（办公用户优先级）

### 🔴 P0：事件日志内容几乎全是技术噪音

现状：事件日志（Overview 底部）和时间线页面显示的都是 `system.healthcheck_passed`、`system_start` 等技术事件，对有项目要推进的用户没有任何价值。

**建议**：
1. 默认过滤掉 `healthcheck_passed` 类事件，只展示 `task_*`、`role_*`、`object_*` 等业务事件
2. 或在事件日志上方加"业务事件/技术事件"切换，默认展示业务事件
3. 接入任务/对象变更时的业务事件写入机制（目前缺少 task_created/task_updated 类事件写入）

### 🟡 P1：Overview 页信息密度过高，30秒内难以聚焦

现状：Overview 有 10+ 个卡片，从 KPI 到冷启动快照到活跃对象到事件日志全挤在一起，初次开屏需要大量滚动。

**建议**：
1. 增加"今日快照"视觉分区，将冷启动快照、项目阶段、待办聚合合并为一个"今日状态"区（折叠其余内容）
2. 将工作区活动、告警摘要等低频信息折叠或移入二级抽屉
3. "冷启动快照"标签改为更易懂的"今日工作摘要"
