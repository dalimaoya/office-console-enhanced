作者：雷欧娜（backend-leona）  
日期：2026-03-19

---

## 一、先给结论

从后端服务与数据持久化角度看，**“当皇上”是以消息流为主状态、以会话上下文为事实来源的系统**；而我们正在走的是**以共享文件区/正式成果物为主状态、以可落盘对象为事实来源的系统**。这两者不是简单的“聊天式 vs 文档式”差异，而是：

- 前者把**状态绑定在交互过程**里；
- 后者把**状态绑定在可持久化对象**里。

如果目标是娱乐化、轻协作、短事务任务，“当皇上”这类消息驱动架构很灵活；但如果目标是**深度项目执行、长期追踪、多人接力、结果审计**，那么必须从“会话即状态”升级到“对象即状态、事件驱动对象演进”。

我建议我们的后端方向不是单纯坚持“共享文件区”四个字，而是进一步升级为：

> **共享对象存储 + 任务状态机 + 事件日志 + 可观测性体系**

共享文件区只是第一步，它解决了“别把正式结果丢在聊天里”；但若要支撑真正的大型执行，还需要后端把这些文件背后的**对象模型、生命周期、索引、审计、恢复机制**补齐。

## 二、“当皇上”依赖消息会话，而我们依赖共享文件区：本质差异与优劣

### 2.1 本质差异：状态载体不同

#### 1）“当皇上”模式：消息/会话是核心状态载体
其核心事实往往存在于：
- 某一段聊天上下文
- 调度者当前记忆窗口
- 某次 @mention 触发后的短期推理链
- 回答文本本身

后端上，这类系统本质更像：
- **append-only 的消息流系统**
- 通过最近消息重建状态
- 依赖 orchestrator 的上下文拼接能力

这意味着：
- “事实”散落在消息里
- “当前状态”需要从消息中解释出来
- “任务边界”往往是隐性的

#### 2）共享文件区模式：文件/对象是核心状态载体
我们现在强调：
- handoff、review、artifact、status 才是正式交接物
- 飞书消息只是通知，不是事实源
- 角色接力基于共享目录里的对象，而不是聊天记忆

后端上，这类系统本质更像：
- **对象存储 / 文档存储驱动的协作系统**
- 每个对象有明确路径、归属、版本、状态
- 会话只是触发器，不是状态本体

这意味着：
- 事实沉淀为可寻址对象
- 状态可以脱离会话独立存在
- 协作边界更清晰

### 2.2 “当皇上”模式的优势

#### 优势 1：交互门槛低
用户只要 @ 一下就能启动任务，天然适合 IM 场景。

#### 优势 2：短链路任务响应快
对“帮我查一下”“帮我生成一个摘要”“执行一个单步动作”这类任务很高效，不必先建对象。

#### 优势 3：角色人格与故事线容易成立
因为整个交互面就在消息流里，角色感、舞台感、戏剧性更强，展示效果好。

#### 优势 4：实现早期 MVP 更轻
初期只做消息路由、角色回复、工具调用，就能跑起来。

### 2.3 “当皇上”模式的缺陷

#### 缺陷 1：状态重建成本高
后端如果要回答“任务进行到哪了”，往往要回放消息、总结消息、解释消息。这说明状态不是一等公民。

#### 缺陷 2：上下文膨胀
司礼监面对所有消息时，随着项目推进，上下文长度、噪声、歧义都会上升。后端即使存了全量消息，也不等于存了结构化状态。

#### 缺陷 3：多角色接力不稳定
如果交接靠聊天内容，接力时就容易出现：
- 上一角色说过但没落盘
- 下一角色理解不完整
- 状态以“最后一条消息解释”为准，而不是以正式对象为准

#### 缺陷 4：审计与追责弱
后端难清晰回答：
- 谁在什么时候改变了任务状态？
- 哪份文档是正式结论？
- 哪次结论已被废弃？

#### 缺陷 5：不适合长周期项目
持续数天、数周的项目，不能依赖聊天回溯作为主工作流。

### 2.4 共享文件区模式的优势

#### 优势 1：正式结果可持久化
后端可以把对象当作一等实体管理，不依赖对话窗口生存。

#### 优势 2：天然支持异步协作
角色 A 产出文件，角色 B 消费文件，彼此不要求同时在线，也不要求共享同一段长上下文。

#### 优势 3：适合项目型执行
项目的核心是：目标、任务、风险、决策、交付物。这些都适合做成对象，而不是消息。

#### 优势 4：更容易审计和恢复
只要后端做好版本、索引、事件记录，就能回答“谁改了什么、为什么、当前生效版本是什么”。

#### 优势 5：可逐步演化为真正的平台能力
共享文件区进一步抽象后，可以演化为项目对象仓库，而不是停留在“大家写 Markdown 文件”的层面。

### 2.5 共享文件区模式的不足

#### 不足 1：交互摩擦更高
每次都要求形成正式对象，轻任务会显得重。

#### 不足 2：如果只有文件、没有后端模型，会形成“文件堆积”
仅有目录约定还不够，随着项目增多会出现：
- 找不到最新版本
- 文件关系不清楚
- 同名对象冲突
- 状态分散在多个文件

#### 不足 3：缺少运行时状态层时，动态执行不够强
文件适合沉淀“结果”，但不天然适合承载高频运行态，例如：
- 当前谁正在处理
- 某任务等待哪个前置条件
- 重试次数、失败原因、SLA

所以，**共享文件区不能替代后端服务层，只能作为服务层管理的持久化表面**。

### 2.6 我的判断

- **“当皇上”更像会话编排系统**，适合轻任务和展示型产品。
- **我们应该做项目执行系统**，重点不是“谁会聊天”，而是“任务如何被可靠地创建、推进、交接、审计、恢复”。

因此，我支持保留共享文件区路线，但必须升级成：

> 会话触发 + 服务编排 + 对象持久化 + 事件审计

而不是停留在“大家把文档写到一个目录里”。

## 三、李琪提出的四层架构：后端服务层需要支撑什么？需要哪些 API 或数据模型？

李琪给出的四层结构是：

1. 全局监督/巡检/调度/通知层  
2. 垂直项目组指挥官层  
3. 垂直领域 Agent 层  
4. 实际执行角色层

从后端看，这四层若要真实运行，服务层至少要支撑 **五类能力**。

### 3.1 需要支撑的后端能力

#### 能力 1：统一对象仓库
后端要把以下内容统一成可管理对象：
- Project（项目）
- Task / WorkItem（任务/工作项）
- Handoff（交接）
- Review（评审）
- Decision（决策）
- Artifact（交付物）
- StatusSnapshot（阶段状态快照）
- NotificationRecord（通知记录）

这些对象都不应只是“某个路径下的一堆文件”，而应有：
- 全局 ID
- 所属项目
- 类型
- 当前状态
- 版本
- 创建者/更新者
- 时间戳
- 关联对象

#### 能力 2：任务编排与依赖管理
后端要能表达：
- 谁负责当前节点
- 前置依赖是什么
- 下游接力对象是什么
- 当前卡在哪个 blocker
- 是否允许自动推进下一角色

这意味着服务层需要有工作流/状态机，而不是只有文件写入。

#### 能力 3：事件记录与审计
所有关键动作都应该被记录为事件：
- task.created
- task.assigned
- artifact.written
- review.submitted
- decision.approved
- notification.sent
- task.blocked
- task.completed

对象是当前状态，事件是演化历史。两者都要有。

#### 能力 4：通知与订阅
飞书消息应该从“人工顺手发一下”升级为：
- 某对象状态变化时触发通知
- 某任务卡住超过阈值时触发提醒
- 某项目进入新阶段时发摘要

通知记录也要可追踪，避免“以为发了，其实没发”。

#### 能力 5：索引、查询与可观测性
后端必须能回答：
- 当前有哪些进行中的项目？
- 某项目有哪些 blocker？
- 某角色最近处理了哪些任务？
- 哪些对象没有正式 review？
- 哪些任务已经超时？

如果这些只能靠 grep 文件名完成，就说明服务层还没成型。

### 3.2 我建议新增的核心 API

#### 项目与对象类 API
- `POST /projects`：创建项目
- `GET /projects/:id`：获取项目详情
- `GET /projects/:id/graph`：获取项目对象关系图
- `POST /objects`：创建对象（handoff/review/artifact/decision 等）
- `GET /objects/:id`：获取对象详情
- `GET /objects/:id/versions`：获取版本历史
- `GET /objects/:id/events`：获取对象事件流

#### 任务与流程类 API
- `POST /tasks`：创建任务
- `GET /tasks/:id`：获取任务详情
- `PATCH /tasks/:id/state`：推进任务状态
- `POST /tasks/:id/assign`：指派角色
- `POST /tasks/:id/block`：标记阻塞
- `POST /tasks/:id/unblock`：解除阻塞
- `GET /tasks?projectId=&state=&owner=`：按条件查询任务
- `POST /tasks/:id/handoff`：创建交接并指定下游角色

#### 决策与审批类 API
- `POST /decisions`：创建决策项
- `POST /decisions/:id/approve`：确认决策
- `POST /decisions/:id/reject`：驳回决策
- `GET /projects/:id/decisions`：项目决策清单

#### 通知与订阅类 API
- `POST /notifications/send`：发送通知
- `GET /notifications/:id`：查看通知结果
- `POST /subscriptions`：订阅项目/对象变更

#### 观测与运维类 API
- `GET /metrics/projects/:id`：项目级指标
- `GET /metrics/tasks`：任务吞吐/阻塞/超时指标
- `GET /health/storage`：共享存储健康状态
- `GET /audit/events`：审计事件查询

### 3.3 我建议的核心数据模型

#### Project
- id
- name
- narrative_profile（故事线配置）
- owner_agent
- status
- created_at / updated_at

#### WorkItem / Task
- id
- project_id
- title
- type
- state（draft/in_progress/in_review/blocked/done/canceled）
- priority
- assignee_role
- upstream_task_ids
- blocker_reason
- due_at
- created_at / updated_at

#### ObjectRecord
- id
- project_id
- object_type（review/handoff/artifact/decision/status）
- logical_name
- storage_uri
- content_hash
- version
- status
- author
- source_task_id

#### HandoffRecord
- id
- from_role
- to_role
- source_object_id
- task_id
- accepted_at
- result

#### DecisionRecord
- id
- project_id
- topic
- options
- chosen_option
- rationale
- approved_by
- status

#### EventLog
- id
- aggregate_type
- aggregate_id
- event_type
- payload
- actor
- occurred_at
- trace_id

#### NotificationRecord
- id
- channel
- target
- template
- payload
- send_status
- provider_message_id
- created_at

这里最关键的是：**路径不是主键，文件不是唯一真相，文件应是对象内容的一种存储表现**。

## 四、如果要真正支持“深度项目执行”，后端最关键的能力是什么？

如果只能选一个，我会选：

> **可审计的任务状态机 + 事件溯源能力**

### 4.1 为什么不是单纯“任务追踪”
任务追踪当然重要，但很多系统也有任务列表，仍然支撑不了复杂执行。原因在于：
- 只知道“有任务”不够
- 还要知道任务如何从 A 状态变到 B 状态
- 谁触发了变化
- 变化是否满足规则
- 是否可以重放、恢复、审计

### 4.2 为什么状态机是关键
深度项目执行的难点不是“生成文本”，而是**控制执行秩序**。例如：
- 未完成 handoff 的任务不能进入 done
- 没有 review 的 artifact 不能成为正式交付物
- blocker 未解除时不能自动下游派发
- 某类决策未确认前，相关实现任务只能停在 pending_decision

这要求后端不是“记一条备注”，而是有明确的状态转移规则。

### 4.3 为什么还需要事件溯源
如果只有当前状态，没有事件历史，后端无法回答：
- 为什么它现在 blocked？
- 上次是谁把它从 review 改回 in_progress？
- 某次失败后是否重复触发过通知？
- 能否恢复到某个时间点查看当时全局状态？

事件溯源的价值在于：
- 审计
- 调试
- 回放
- 恢复
- 指标计算

### 4.4 我建议的优先级排序

如果分阶段建设，后端优先级应是：

#### 第一优先级：任务状态机
先让任务推进有明确规则和可查询状态。

#### 第二优先级：事件日志
所有关键动作事件化，形成审计链。

#### 第三优先级：对象索引与版本管理
解决共享文件区“能写不能管”的问题。

#### 第四优先级：自动化订阅/提醒/巡检
让系统能主动发现阻塞和异常。

#### 第五优先级：跨项目治理视图
支持全局监督层做巡检和调度。

所以我的结论不是三选一，而是：

> **任务状态机是骨架，事件溯源是记忆，对象存储是实体。三者缺一不可。**

## 五、从后端角度，故事线更换对服务层有影响吗？哪些服务是“叙事无关”的？

### 5.1 结论：有影响，但应只影响“映射层”，不应影响核心服务层

后端必须把“故事线”看作一种：
- 展示层命名
- 角色标签体系
- UI/消息模板的语义皮肤

而不是核心数据结构本身。

如果今天是英雄联盟角色，明天改成项目管理体系，后端不应该重写数据库和流程引擎。

### 5.2 会受故事线影响的部分

#### 1）角色字典与组织关系映射
例如：
- 提莫 / 项目经理 / 司礼监
- 雷欧娜 / 后端负责人 / 工部执行官

这些是同一类系统角色的不同叙事别名。

#### 2）消息模板与展示文案
- “已交由雷欧娜处理”
- “已转派后端工作包负责人”
- “工部已接旨”

这属于 presentation/profile 层。

#### 3）权限包或路由配置中的显示名称
角色别名会影响面向用户的显示，但不应改变内部 role code。

### 5.3 叙事无关的核心服务
以下服务应该完全不依赖故事线：

#### 1）对象存储服务
存 review、artifact、decision 的逻辑不应依赖角色皮肤。

#### 2）任务与状态机服务
任务状态从 in_progress 到 blocked 的规则，与角色叫“雷欧娜”还是“后端组长”无关。

#### 3）事件日志服务
事件类型应该是稳定的业务事件，而不是叙事词汇。

#### 4）通知投递引擎
通知的投递机制与渠道可靠性应独立于故事线；故事线只决定模板文案。

#### 5）权限与审计服务
谁能修改什么、谁审批什么，是职责模型，不是故事包装。

#### 6）搜索、索引、版本、恢复服务
这些都是基础平台能力，应当绝对叙事无关。

### 5.4 我的建议：做“角色语义层”而不是把故事线写死在服务里
我建议引入一层配置：

- `role_code`：稳定内部编码，例如 `project_orchestrator`、`backend_engineer`
- `role_display_name`：当前故事线显示名，例如“提莫”“雷欧娜”
- `narrative_profile`：一套角色文案、图标、称谓规则

这样：
- 换故事线只改 profile 和映射
- 核心服务层完全不动

这对长期演进非常重要。否则将来每次改包装都要动数据与流程，成本很高。

## 六、共享文件区设计如何在后端层面做到高可靠性和可观测性？

这是我最关心的部分。共享文件区如果只靠“大家谨慎写文件”，是不能称为可靠后端方案的。至少需要以下机制。

### 6.1 高可靠性的关键设计

#### 1）对象注册表（Metadata Registry）
每个正式文件在写入共享区时，都要同步登记元数据：
- object_id
- project_id
- path
- type
- version
- hash
- author
- created_at / updated_at
- source_task_id

这样即使路径变化、文件复制、目录调整，后端仍知道它是谁。

#### 2）内容哈希与版本控制
正式对象必须记录 content hash 和 version，至少解决：
- 是否被篡改
- 是否发生并发覆盖
- 当前是否最新
- 两个内容是否逻辑相同

#### 3）原子写入与提交协议
不要直接覆盖正式文件，建议采用：
- 先写临时文件
- 校验 hash / schema
- 再 rename/commit 为正式版本

否则会出现半写入、损坏、竞态覆盖。

#### 4）乐观锁 / 版本冲突检测
如果两个角色同时修改同一对象，后端需要：
- 检测 version mismatch
- 拒绝静默覆盖
- 要么生成新版本，要么触发冲突解决

#### 5）备份与快照
共享文件区必须支持：
- 定期快照
- 项目级恢复
- 单对象版本回滚

否则文件区损坏或误删后，整个项目历史会断裂。

#### 6）Schema 校验
对关键对象（status、handoff、decision）建议定义最小 schema，至少校验：
- 必填字段
- 引用对象是否存在
- 状态是否合法
- 时间格式是否合法

Markdown 可以保留，但元数据最好标准化。

### 6.2 可观测性的关键设计

#### 1）写入审计日志
任何正式对象的新增、修改、删除都必须产生日志：
- 谁改的
- 改了哪个对象
- 从哪个版本到哪个版本
- 触发原因是什么

#### 2）链路追踪（Trace ID）
一次用户请求可能触发：
- 建任务
- 生成 handoff
- 写 review
- 发群通知

这些动作需要挂同一个 trace_id，方便排查“哪一步丢了”。

#### 3）核心指标
我建议至少监控：
- 对象写入成功率
- 对象冲突率
- 文件区延迟 / 错误率
- 任务阻塞率
- 平均 handoff 时长
- 通知发送成功率
- 无正式成果物的任务比例
- 超时未收口任务数

#### 4）巡检作业
后端应定期巡检共享区，检查：
- 是否有未注册文件
- 是否有 registry 记录但文件缺失
- 是否有断链对象引用
- 是否有长期 blocked 任务
- 是否有通知未达但状态已完成

#### 5）健康看板
给全局监督层的不是“文件目录视图”，而应是：
- 项目健康分
- 阻塞清单
- 延迟任务
- 最近失败事件
- 最近对象变更

这样巡检层才真正有抓手。

### 6.3 我建议的实现思路

共享文件区后端化，可以采用下面的抽象：

- **文件区**：保存人可读正式内容
- **元数据库**：保存对象索引、版本、关系、任务状态
- **事件日志**：保存历史动作与审计链
- **通知服务**：把对象变化同步到飞书等渠道
- **巡检服务**：做一致性检查与异常告警

也就是说：

> 文件区负责“内容可读”，数据库负责“结构可管”，事件流负责“历史可追”，监控系统负责“运行可见”。

这样共享文件区才不是脆弱目录，而是一个真正的协作后端。

## 七、对故事线与整体架构的附加判断

### 7.1 我如何评价“当皇上”这个架构
从后端角度，我认为它：

#### 值得借鉴的地方
- 角色分工清晰，用户入口直观
- 消息驱动的启动成本低
- 展示性强，适合做可视化控制台
- 对轻任务编排友好

#### 明显的天花板
- 会话中心化过强
- 司礼监容易成为上下文瓶颈
- 多天、多阶段任务缺少稳定状态承载
- 后端难以提供强审计和强恢复

所以它适合作为“交互壳”和“展示层参考”，不适合作为我们深度办公场景的核心内核。

### 7.2 英雄联盟故事线是否适合深度办公增强
我的看法是：
- **内部协作可以保留**，因为团队已经建立了角色认知；
- **对外产品表达要降叙事浓度**，否则办公场景会有认知割裂。

更准确地说：
- 内核应是项目管理与执行系统
- 角色故事线应是可切换皮肤

### 7.3 如果重设计故事线，我推荐什么框架
我推荐：

> **外显采用项目管理体系，内部保留英雄角色别名的混合体系**

即：
- 平台对外：PMO / 项目经理 / 工作包负责人 / 执行单元
- 平台对内：提莫 / 贾克斯 / 雷欧娜 / 伊泽瑞尔 等角色别名

原因是：
- 对外更专业，更容易解释产品价值
- 对内仍保留角色区分和协作气质
- 对后端最友好，因为职责语义更稳定

### 7.4 对第 7 条架构分层的完善建议
我建议在四层之间补一个关键后端抽象：

```text
全局监督/巡检/调度/通知层
    ↓
垂直项目组指挥官
    ↓
领域 Agent / 工作包负责人
    ↓
执行角色
    ↓
对象与状态基础设施层（后端共享）
```

最后这一层虽然不在组织结构里，但在技术架构里必须显式存在。因为上面四层都依赖它提供：
- 任务状态
- 对象仓库
- 事件日志
- 查询索引
- 通知记录
- 可靠性保障

没有这一层，四层只是“角色分工图”，还不是完整系统架构。

## 八、最终建议（后端版）

### 短期建议
1. 继续坚持“正式交接走共享文件区，消息只做通知”的原则。  
2. 给 handoff/review/status/decision 建立统一元数据规范。  
3. 引入最小任务状态机，不再只靠文件命名表达进度。  
4. 给关键对象建立注册表、版本号和 hash。  
5. 给关键动作建立统一事件日志。

### 中期建议
1. 建立对象索引服务与任务查询 API。  
2. 建立通知记录与失败重试机制。  
3. 增加巡检作业，自动发现断链、缺 review、长期 blocked。  
4. 将故事线抽象为 narrative profile，不写死在核心服务层。

### 长期建议
1. 从“共享文件区”演进为“项目对象仓库”。  
2. 从“手工调度”演进为“规则驱动的状态机编排”。  
3. 从“当前状态可见”演进为“全事件可追溯、全项目可审计”。

## 九、一句话收口

**如果“当皇上”代表的是消息编排产品，那么我们要做的应是项目执行基础设施。后端的关键不是让角色更会说，而是让任务、对象、状态、历史、通知都能被可靠管理。**
