## 一、"当皇上"三省六部制架构评价：上下文爆炸的本质

### 问题本质

司礼监（中央编排者）是一个**单点汇聚的有状态路由器**。每一条用户消息、每一次六部回执、每一个审批流转都必须经过司礼监的上下文窗口。这不是一个"可优化的性能问题"，而是一个**架构范式缺陷**：

1. **线性增长的不可逆性**：消息驱动架构中，编排者的上下文与系统活跃度成正比增长。六部越多、任务越并行，司礼监的上下文增长越快。
2. **状态与路由耦合**：司礼监同时承担"理解当前状态"和"决定下一步路由"两个职责。要路由就必须理解全局，要理解全局就必须持有全量上下文——这是个死锁。
3. **会话即记忆**：整个系统的"记忆"存在于聊天历史中，没有独立的持久化状态层。一旦上下文窗口截断，系统就"失忆"。

### 能否解决？

**在消息驱动+中心化编排的范式内，无法根本解决，只能缓解。** 缓解手段包括：
- 上下文摘要/压缩（有损，且摘要本身消耗 token）
- 会话分片（破坏全局一致性）
- 分层编排（本质上就是我们正在讨论的四层架构）

真正的解法是**范式转换**：从"消息驱动+中心化有状态编排"转向"文件驱动+去中心化无状态协作"——这正是我们当前架构的方向。

## 二、共享文件区交互设计的架构优劣

### 架构优势

1. **无状态编排**：提莫不需要持有所有角色的交互历史，只需要读取项目目录中的当前状态文件。每次唤醒都是"从零读取状态→做决策→写回状态"，天然避免上下文爆炸。
2. **持久化即协作**：文件系统就是状态存储。角色之间的交接不依赖消息传递的时序，而是依赖文件的存在性和内容。这让异步协作变得天然可靠。
3. **可追溯性**：所有决策、交接、产出都有文件形式的 audit trail，不会因为聊天窗口滚动而丢失。
4. **角色可替换**：因为协作协议是"读文件→做事→写文件"，角色的内部实现可以自由替换，只要遵守文件契约。

### 潜在缺陷

1. **实时性不足**：文件交互是"拉取式"的，角色 A 写了文件，角色 B 不会自动知道。当前靠飞书通知补偿，但通知和文件之间没有强一致性保证。如果通知丢了或延迟了，角色 B 可能读到过期状态。
2. **文件契约隐式化**：哪些文件是"正式交接物"、文件格式应该是什么、写入时机是什么——这些契约目前散落在各角色的 SOUL.md 和项目协议中，没有一个统一的 schema 层。随着角色和项目增多，契约漂移的风险很高。
3. **并发写入冲突**：两个角色同时写同一个状态文件怎么办？当前文件系统没有锁机制。虽然目前角色串行执行概率高，但架构设计不能依赖"恰好不并发"。
4. **冷启动成本**：每次角色被唤醒都要重新读取项目状态，如果项目目录文件量大，冷启动的 token 消耗会成为新的成本问题。

### 改进建议

- 引入 **MANIFEST.md** 或 **INDEX.md** 作为项目级状态索引，角色冷启动时只读索引，按需加载细节。
- 为关键交接物定义 **schema**（至少是 frontmatter 格式约定），让文件契约显式化。
- 考虑引入轻量级的 **文件变更事件机制**（如 watch 或 webhook），补偿纯拉取模式的实时性缺陷。

## 三、四层架构的设计方案

李琪提出的四层架构方向是正确的。从系统架构角度，关键设计约束如下：

### 架构蓝图

```
┌─────────────────────────────────────────┐
│  L0: 全局监督层（Azir/Main）            │
│  职责：跨项目资源分配、全局策略、升级仲裁 │
│  状态：读取各项目的 STATUS.md            │
│  触发：定时巡检 + 被动升级               │
└────────────────┬────────────────────────┘
                 │ 项目创建/销毁/升级
┌────────────────▼────────────────────────┐
│  L1: 垂直指挥官层（Teemo 等）            │
│  职责：单项目内任务编排、链路调度、收口    │
│  状态：读写项目目录 status/ tasks/        │
│  边界：只管自己的项目，不跨项目            │
└────────────────┬────────────────────────┘
                 │ 任务分派 + 成果物路径
┌────────────────▼────────────────────────┐
│  L2: 领域 Agent 层（Ekko/Jax/Lux 等）   │
│  职责：领域级分析、决策、产出正式文档      │
│  状态：读写项目 docs/ reviews/ briefs/    │
│  边界：只在自己领域内做判断，跨域走 L1     │
└────────────────┬────────────────────────┘
                 │ 执行指令 + 验收标准
┌────────────────▼────────────────────────┐
│  L3: 执行角色层（Codex/开发型 Agent）     │
│  职责：代码实现、测试执行、部署操作        │
│  状态：读写项目 src/ tests/ artifacts/    │
│  边界：只执行，不做领域级决策              │
└─────────────────────────────────────────┘
```

### 关键设计原则

1. **单向依赖**：L(n) 只能调度 L(n+1)，不能反向调度。L3 遇到问题只能上报给 L2，不能直接找 L0。
2. **状态下沉**：每层只读写自己职责范围内的文件。L0 不直接改 L3 的代码，L3 不直接改 L0 的策略。
3. **接口契约化**：层间交互通过明确的文件格式（brief → handoff → review → artifact）传递，不通过消息内容传递。
4. **失败上浮**：L3 失败报 L2，L2 无法解决报 L1，L1 无法解决报 L0。每层有明确的"自恢复→上报"边界。

### 扩展性保证

- 新增项目 = 新增一个 L1 实例 + 复用 L2/L3 角色池
- 新增领域 = 在 L2 增加一个角色，L1 的调度表增加一行
- 新增执行能力 = 在 L3 增加工具或 Skill，L2 的指令模板增加一项

## 四、技术底座 vs 故事线绑定层

### 与故事线无关的技术底座（不需要变）

| 组件 | 说明 |
|------|------|
| 四层架构分层 | L0-L3 的职责划分与调度协议 |
| 共享文件区协作协议 | 文件驱动交接、status/handoff/review/artifact 体系 |
| OpenClaw Gateway 路由 | 消息入口→agent 路由→工具调用链路 |
| 飞书通知机制 | 群通知作为可见性补充的定位 |
| 模型路由策略 | 决策层用推理模型、执行层用代码模型 |
| 项目目录结构 | status/ tasks/ docs/ reviews/ protocols/ |
| 角色间契约格式 | brief/handoff/review 的 frontmatter 和内容规范 |

### 与故事线绑定的（需要调整）

| 组件 | 说明 |
|------|------|
| 角色命名和人设 | 提莫/艾克/贾克斯 等英雄联盟名称与人设描述 |
| SOUL.md 中的语气和性格 | 角色的说话方式、态度、隐喻 |
| 飞书 Bot 的显示名和头像 | 用户可见的品牌表现层 |
| 对外传播素材 | 介绍文案、示例对话、角色关系图 |
| accountId 命名 | `orchestrator-teemo` 等标识符（技术上可映射，但会影响可读性） |

### 关键判断

**技术底座占系统复杂度的 80% 以上，故事线绑定层只占表现层的 20%。** 换故事线的成本远低于换架构的成本。因此：
- 先把技术底座做稳，故事线可以后换
- 故事线选择应该服务于"降低用户认知成本"和"提升传播效率"，而不是影响架构设计
- 架构文档中应该使用**角色职能名**（Orchestrator/Architect/Frontend-Dev）而不是故事线名称，故事线名称只出现在表现层

## 五、最大技术债与演进方向

### 当前最大技术债

**角色契约隐式化 + 调度协议硬编码。**

具体表现：
1. 角色之间"应该产出什么文件、文件格式是什么、写到哪里"散落在各自的 SOUL.md 和 AGENTS.md 中，没有统一的契约注册表。
2. 调度链路（需求→产品→架构→实现→验收）硬编码在 Teemo 的 AGENTS.md 中，不能动态配置。新增一条链路就要改 Teemo 的规则文件。
3. L0（Main/Azir）和 L1（Teemo）的边界目前靠文字约定，没有技术层面的强制隔离。

### 演进方向（三阶段）

**Phase 1：契约显式化（近期）**
- 定义统一的角色契约 schema：每个角色的输入/输出/文件路径/触发条件
- 在项目根目录增加 `CONTRACTS.md` 或 `contracts/` 目录
- 调度链路从硬编码改为配置文件（如 `workflows/` 目录下的 YAML 或 Markdown）

**Phase 2：事件驱动补偿（中期）**
- 在文件驱动的基础上增加轻量事件层：文件写入触发通知
- 减少"提莫手动检查状态→决定下一步"的开销
- 实现"文件变更→自动触发下游角色"的半自动链路

**Phase 3：多项目实例化（远期）**
- L1 层变成可实例化的模板：新建项目 = 实例化一个 Orchestrator + 配置一套 workflow
- L2/L3 角色池化：角色不绑定单一项目，而是按需分配
- 跨项目资源调度由 L0 统一管理

---

## 总结

1. "当皇上"的架构本质问题是**有状态中心化编排**，在深度项目场景下不可扩展。
2. 我们的文件驱动设计方向正确，但需要补上**契约层**和**事件层**。
3. 四层架构是正确的分层方式，关键在于**单向依赖**和**接口契约化**。
4. 故事线是表现层问题，不影响技术底座；先稳底座，再选故事线。
5. 最大技术债是契约隐式化，演进方向是"契约显式化→事件驱动→多项目实例化"。
