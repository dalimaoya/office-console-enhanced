## 2. 测试项详细评估

### 2.1 飞书集成引导

#### Overview 页飞书未配置提示

**状态：✅ 逻辑实现正确，⚠️ 导航按钮存在 Bug**

Overview 页存在 `connection-health-card` 组件，当飞书未配置时会自动显示：
- 标题："飞书通知未配置"
- 描述："连接后可接收 Agent 工作通知"
- 触发逻辑：调用 `/api/v1/settings/wiring-status`，检测 `feishu.webhookConfigured` 为 false 时展示

当前环境下该卡片**应当显示**（feishu.status = "not_configured"）。

**🔴 Bug #1：「去配置 →」按钮导航失效**

HTML 定义：
```html
<a href="#settings" class="health-card-action" id="connection-health-action-btn">去配置 →</a>
```

JS 中 `bindConnectionHealthAction()` 函数注释声称 `onclick="navigateTo('settings')"` 已在 HTML 中设置，但 HTML 实际上**没有 onclick 属性**。点击「去配置 →」将尝试跳转到 `#settings` 锚点，而页面中不存在 `id="settings"` 的元素（实际元素 id 为 `settings-page`），导致点击后页面无明显响应，用户无法进入设置页。

**复现步骤**：在 Overview 页（飞书未配置状态下）点击告警卡片中「去配置 →」→ 页面无反应或滚动到顶部。

---

#### Settings 页飞书空状态 CTA

**状态：🟡 逻辑循环，UX 混乱**

Settings 页飞书通知面板包含 `feishu-empty-state` 组件，内有「去配置」按钮：
```html
<a href="#settings" class="feishu-empty-state__cta btn btn--accent" id="feishu-empty-state-cta">去配置</a>
```

JS 将此按钮绑定到 `goToSettings()` → `navigateTo('settings')`，但**当前页面本身就是 Settings 页**，点击后等同于刷新当前页面。

用户已在 Settings 页看到「飞书通知未配置」→ 点击「去配置」→ 仍停在 Settings 页，没有任何新信息出现，形成**死循环式导航**。

**建议**：该 CTA 应改为「查看配置说明 ↓」并平滑滚动到下方的飞书 Webhook 配置卡片，或直接展开配置步骤。

---

#### Settings 页飞书告警通知卡片配置说明

**状态：🟡 说明存在但不够完整**

当前配置说明文字：
> "如需启用：在服务器 .env 文件中配置 FEISHU_WEBHOOK_URL，重启服务后生效。"

**问题**：
1. 缺少获取飞书 Webhook URL 的步骤（需前往飞书开放平台创建 Bot，新手不知道从哪里获取 URL）
2. 缺少 URL 格式示例（如 `https://open.feishu.cn/open-apis/bot/v2/hook/xxx`）
3. 缺少「测试连接」入口，用户配置后无法快速验证是否生效
4. 触发条件"上下文压力 ≥ 80% · Agent 空闲 > 2小时"只做展示，不可调整，且没有说明这是默认值还是固定值

### 2.2 Settings 页整体评估

**状态：✅ 结构清晰，🟡 部分交互需优化**

Settings 页共有以下板块：
- 安全设置（4 项：访问模式、Token 鉴权、Dry-run 模式、版本/运行时长）
- 飞书通知（Bot 连接状态 + 告警通知配置）
- 定时任务 Cron 健康
- 接线状态诊断（Wiring Status）
- 模板管理

**优点**：
- 安全设置 4 卡片布局清晰，readonlyMode 状态突出展示
- 接线状态诊断将 Gateway/飞书/文件系统三项合并展示，对运维人员友好

**问题**：
1. **飞书 Webhook 无表单输入**：当前页面仅展示状态，没有直接输入 Webhook URL 的表单。用户需要通过修改 .env 文件配置，这对办公场景非技术用户不友好
2. **告警阈值不可配置**：告警阈值（上下文压力 ≥ 80%）是硬编码展示，无法调整
3. **Cron 健康板块用途不明**：对普通办公用户，"定时任务 Cron 健康"的用途和意义不够清晰，缺少场景说明

### 2.3 Onboarding 体验

**状态：✅ 基础设计合理，🟡 细节有改进空间**

**Onboarding Banner 设计**：
- 顶部横幅显示三步引导：`① 查看实时 Agent 状态 → ② 在飞书接收任务通知 → ③ 在任务页管理工作队列`
- 支持一键关闭，状态存入 localStorage（`oc_onboarding_dismissed`）
- 关闭后不再显示（不支持重新查看）

**优点**：
- 三步式引导覆盖了核心使用路径
- 每个页面均有 `page-subtitle`（"最适合：..."），帮助用户快速判断当前页用途
  - 总览：最适合快速了解当前 AI 团队整体状态
  - Agent 状态：最适合查看哪些 Agent 正在工作、空闲或阻塞
  - 用量：最适合了解 API 用量、成本和上下文压力
  - 设置：最适合配置飞书通知、安全模式和系统集成

**问题**：
1. **飞书集成是核心 Onboarding 步骤，但 Step 2「在飞书接收任务通知」缺乏可操作性**：引导文案未说明「如何配置」，对首次使用的用户会产生疑惑
2. **关闭后无法重新查看**：如果用户误关闭引导，需要手动清除 localStorage 才能重新看到。建议在 Settings 或 Help 中加入「重新查看引导」入口
3. **只读模式未在 Onboarding 中解释**：系统处于只读模式，但引导中未提及，用户若尝试执行操作会遇到拦截，但引导中没有铺垫

**上下文压力警报**：
- 提莫：237% ⚠️（超过 200k 窗口上限，估算值）
- 伊泽瑞尔：100% ⚠️（满载）
- 贾克斯：56.4%（警告级）

**办公管理角度评估**：

✅ 有用的数据：
- 总费用和 Token 汇总对管理层核算 AI 使用成本有直接价值
- 按 Agent 分组展示，可快速定位高消耗角色
- 上下文压力提示，可预警哪些 Agent 需要清空上下文

🟡 展示方式问题：
1. **费用单位为 USD 但未显示人民币换算**：办公场景下国内用户更关注人民币成本，建议加注汇率折算
2. **总费用 $46.83 但无历史对比**：没有「昨日/本周平均」对比基准，无法判断今日用量是否异常
3. **提莫 Token 占比 93%（42M/45M）异常突出**，但页面没有告警标注，也没有说明这是否正常
4. **上下文压力 > 100% 的数据展示令人困惑**：提莫显示 237% 压力，这在 UI 上会显示为超出进度条的状态，需要有说明文字解释"超限"的含义（估算值、历史累计 vs 单次窗口）
5. **时间范围切换（今天/本周/本月）**：「今天」和「本周」数据目前是同一数据集（均返回 period="today"），是否支持历史按周/月聚合尚不明确，需核实
