### 决策3：Gateway技术方案重大调整
- **决策时间**：2026-03-14  
- **决策内容**：
  1. 放弃"前端直连OpenClaw Gateway REST接口"的原假设
  2. 改为"Gateway CLI/RPC + 后端适配层 + 缓存 + 办公友好DTO"方案
- **决策者**：基于Leona技术验证结论，由李琪确认
- **理由**：技术验证发现Gateway不提供假设的REST `/api/v1/*`接口，而是通过CLI/RPC方式调用
- **影响**：
  - 需要新增适配层服务
  - 必须实现缓存机制应对CLI/RPC延迟
  - 改变前后端集成方式

### 决策4：适配层架构边界（Jax架构评估建议）
- **决策时间**：2026-03-13（架构评估完成时间）
- **决策内容**：适配层核心职责边界刚性约束
- **决策者**：Jax（架构师评估），由项目采纳
- **理由**：防止适配层膨胀为"万能中间层"，确保架构清晰
- **核心条款**：

#### 适配层 **只做**：
1. **调用封装**：Gateway CLI/RPC调用 + 超时重试机制
2. **数据规整**：原始结构 → 内部canonical model → 办公友好DTO
3. **缓存管理**：TTL、预热、stale-while-revalidate、熔断保护
4. **安全裁剪**：过滤敏感字段，不透出原始配置全文

#### 适配层 **不做**：
1. 告警规则引擎（应由独立模块或未来Gateway能力承接）
2. 趋势计算和历史对比（MVP阶段简化处理）
3. 复杂权限模型（JWT鉴权在网关/入口层处理）
4. 任何写操作代理
5. 前端业务渲染逻辑

### 决策5：缓存策略与性能保障
- **决策时间**：2026-03-14
- **决策内容**：缓存必须是适配层P0核心功能，而非可选优化
- **决策者**：基于技术验证和架构评估，由项目采纳
- **理由**：
  - Gateway CLI/RPC实测延迟2.5-5秒，无缓存系统不可用
  - 缓存是系统可用的前提条件，而非性能优化项
- **实现要求**：
  1. 服务启动时立即执行全量预热
  2. 后台定时刷新间隔：dashboard核心数据30秒，辅助数据5分钟
  3. 首次访问P95 < 500ms（预热后）
  4. 必须实现singleflight模式防止缓存击穿

### 决策6：版本兼容隔离设计
- **决策时间**：2026-03-13
- **决策内容**：建立Gateway版本契约与兼容隔离层
- **决策者**：Jax（架构师评估），由项目采纳
- **理由**：隔离Gateway版本变化对适配层和前端的影响
- **实现模式**：
  ```
  Gateway CLI/RPC
    ↓
  [Raw Response Parser] ← 基于Gateway runtimeVersion标记
    ↓
  [Canonical Internal Model] ← 适配层内部稳定模型
    ↓
  [DTO Mapper]
    ↓
  Office REST DTO
  ```
