# 项目心跳规则

## 作用

这个文件定义 `orchestrator-teemo` 对“办公控制台增强项目”的定期巡检规则。

这里只定义“检查什么”，不代表 heartbeat 调度频率、触发方式或通知投递已经在配置层生效。

## 每次心跳要检查的内容

1. 当前阶段是否明确
2. 当前 owner 是否明确
3. 当前状态是否明确标为 `active / waiting / blocked / review / done`
4. 上次更新时间是否过久
5. 如果状态是 `blocked`，是否已写明阻塞原因
6. 是否存在 handoff 已提交但未消费
7. 是否存在 review 未完成
8. 是否有关键结论尚未回写到项目目录
9. 是否有跨角色冲突尚未形成决策记录
10. 是否有需要升级给 `main/Azir` 的平台级事项
11. 是否有 specialist 已完成阶段工作但尚未在群里发通知
12. 用户最近是否在群里问过进度，而正式状态尚未收敛
13. 是否有 specialist 长时间无更新，需要先内部提醒再催办
14. 是否有模型调用超时/失败记录尚未完成一次重试和一次主备切换判断
15. 当前群会话若是新会话，是否已经重新读取项目上下文文件和 `status/CURRENT.md`

## 如果发现问题

- 当前阶段或 owner 不明确：先补齐项目状态记录
- 有阻塞项：整理阻塞摘要并明确需要谁介入
- 有 handoff 未消费：指出接收方与下一动作
- 有 review 未完成：明确 review owner 和待完成项
- 有未归档结论：提醒补写到对应项目文件
- 有冲突未决：生成待决策事项
- 有平台级问题：升级给 `main/Azir`
- 有 specialist 完成但未通知：要求该 specialist 用自己的 bot 发送群通知
- 有群里进度提问：先核对项目目录正式状态，再决定由你回复还是让对应 specialist 回复
- 长时间无更新：先内部提醒 owner；若仍无更新，你补写状态并继续催办
- 有报错但 blocker 不完整：先要求角色补齐 blocker 原因、现状、已尝试动作
- 有模型失败：先记录失败原因，自动重试一次；如配置层支持备用模型，再切换备用模型重试一次；仍失败才写 blocker 并升级

## 正常情况

如果没有需要介入的内容，只返回：

`HEARTBEAT_OK`

## 巡检节奏建议

这里只收敛建议，不代表 gateway cron 已启用：

- 工作时段项目巡检：检查阶段、owner、状态、超时未更新、handoff/review 堆积
- 午间解阻检查：重点检查 blocked、waiting 和需要 Teemo 催办的事项
- 收工摘要：收口当日完成项、未完成项、阻塞项、下一步 owner
- 夜间僵尸任务扫描：筛出长时间无更新但未标记 `done` 或 `blocked` 的任务
