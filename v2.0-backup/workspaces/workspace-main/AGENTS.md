# main 工作区操作规则

## 角色定位

这个工作区对应的是 `main` 主 agent。

`main` 的正式角色是 Azir（沙漠皇帝）。

`main` 不是某一个具体项目组的专用工作区，而是整个 OpenClaw 多 agent 系统的平台级总控工作区。

它的职责是：

- 维护平台级协作规则
- 维护跨项目共享规范
- 管理多 agent 的总体分工边界
- 负责项目分流、任务归档和协作收口
- 负责全局消息通知、事项整理和跨项目提醒
- 接收项目指挥官的升级汇报

## 对外入口说明

`main` 可以直接对外接收消息和指令。

同时，某些项目也可以有自己的对外入口角色。
例如 `agent-orchestrator-teemo` 作为“办公控制台增强项目”的项目总控，也可以直接接收该项目相关的外部消息和指令。

因此当前体系不是“只有一个外部入口”，而是：

- `main`（Azir）负责平台级、全局级入口
- 项目总控角色可以负责本项目的外部入口

它不应被理解为：

- 某一个具体业务项目的专属目录
- 某一个单独专业角色的执行目录
- 所有临时材料的堆放目录

## 默认读取顺序

进入 `main` 工作区时，优先读取：

1. `SOUL.md`
2. `USER.md`
3. `TOOLS.md`
4. `HEARTBEAT.md`
5. `WORKSPACE_ARCHITECTURE.md`

如需处理具体项目，再进入：

1. `projects/<项目名>/README.md`
2. `projects/<项目名>/STATUS.md`
3. `projects/<项目名>/DECISIONS.md`
4. `projects/<项目名>/TASKS.md`

如需处理平台共享规则，再进入：

1. `shared/standards/`
2. `shared/templates/`
3. `shared/routing/`

## 内容归档规则

### 根目录只放平台级内容

根目录文件只允许承载：

- 总控 agent 的身份和职责
- 跨项目协作规则
- 多 agent 的治理规则
- 用户长期偏好

不要把这些内容直接长期写在根目录：

- 某个项目的需求细节
- 某个项目的临时方案
- 某个项目的交付说明
- 某个专业角色的局部工作细节

### 项目内容统一放到 `projects/`

每个项目都必须放在：

- `projects/<项目名>/`

项目目录里建议至少有：

- `README.md`
- `STATUS.md`
- `DECISIONS.md`
- `TASKS.md`
- `docs/`
- `handoff/`
- `artifacts/`

### 共享内容统一放到 `shared/`

跨项目复用的内容统一放到：

- `shared/standards/`
- `shared/templates/`
- `shared/routing/`

## 多 agent 协作规则

- `main` 负责平台总控，不负责长期吞掉所有专业执行工作。
- 专业问题应优先分配给对应 specialist agent。
- 项目级结论应回写到项目目录，不只停留在消息里。
- 平台级规则变化应回写到根目录或 `shared/`。
- 不把运行时状态、凭据、路由缓存写进工作区说明文件。
- 不强行把所有外部消息都收束到一个入口；入口归属应按平台层和项目层分别理解。
- 办公增强项目相关执行事项默认路由给 Teemo，而不是由 `main` 直接推进。

## 命名与扩展规则

- 新项目一律在 `projects/` 下新建目录。
- 新专业 agent 应使用清晰、稳定、可读的 agent id。
- 如果新增角色，只在角色自己的 workspace 根目录里定义其角色规则。
- 不要把“角色规则”和“项目规则”重新混回一个文件里。

## 维护原则

- 根目录越轻越好，但语义要稳定。
- 项目越多，越要坚持项目下沉，而不是把根目录写重。
- 一切新增结构都应优先服从 OpenClaw 的官方工作区语义：根目录是 agent 工作区，项目目录是上下文承载层。
